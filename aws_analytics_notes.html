Module 1 — Query
Amazon Athena
Serverless SQL on S3 — query data in place without loading it into a database.
ATH
Athena Overview core
Serverless interactive SQL query service for data in Amazon S3
▶

Amazon Athena lets you run standard SQL queries directly against data stored in S3 — with no servers to manage, no databases to load data into, and no infrastructure to provision. You query the data in place and pay only for the data scanned.

→ Serverless: no infrastructure to manage. Query runs on demand and you pay $5 per TB of data scanned. Queries that scan less data cost less.
→ Supported formats: CSV, JSON, Parquet, ORC, Avro, and columnar formats. Parquet and ORC are strongly recommended — they are columnar (only scan required columns) and splittable, reducing cost by up to 90% vs CSV.
→ Compression: compress data (GZIP, Snappy, LZ4) to reduce the amount scanned — directly reduces cost. Snappy is preferred for Parquet.
→ Partitioning: organize data in S3 by partition keys (e.g. s3://bucket/year=2024/month=01/). Athena skips partitions that don't match the WHERE clause — dramatically reduces data scanned on large datasets.
→ Uses Apache Presto under the hood. Supports standard ANSI SQL including JOINs, subqueries, window functions, and DDL.
→ Glue Data Catalog: Athena uses the Glue Data Catalog for table metadata (schema, location, partition information) — the central metadata store shared with Glue, Redshift Spectrum, and EMR.
→ Federated Query: query data sources beyond S3 — RDS, DynamoDB, Redshift, CloudWatch, DynamoDB, HBase, DocumentDB — using data source connectors (Lambda functions). Results can be stored back to S3.
EXAM TIP"Analyze S3 data with SQL, no infrastructure" → Athena. "Reduce Athena costs" → use columnar formats (Parquet/ORC), compress data, partition by date/region. "$5/TB scanned" means unused columns in CSV still get scanned — Parquet only scans the columns your query references. "Query data across multiple sources including S3 and RDS in one query" → Athena Federated Query.
USE
Athena Use Cases & Integrations core
When to choose Athena and how it fits into data pipelines
▶
→ Ad-hoc analysis: one-time or infrequent queries on large datasets. No need to spin up a Redshift cluster for a query you'll run once a month.
→ Log analysis: query CloudTrail logs, ELB access logs, CloudFront logs, VPC Flow Logs, S3 access logs — all stored in S3 — directly with SQL.
→ Athena + QuickSight: connect QuickSight directly to Athena for BI dashboards on S3 data. No ETL into a warehouse needed.
→ Data lake query layer: Athena is often the query interface on top of a data lake built in S3 — data lands in S3 via Firehose/Glue, Glue catalogues it, Athena queries it.
EXAM TIP"Analyze VPC Flow Logs stored in S3 using SQL" → Athena. "Query CloudTrail audit logs" → Athena. Athena is the default answer for "SQL queries on S3 data" — it's serverless and requires zero setup compared to loading data into Redshift.
Module 2 — Data Warehouse
Amazon Redshift
Petabyte-scale columnar data warehouse — OLAP analytics at massive scale with sub-second query performance.
RS
Redshift Overview core
Columnar OLAP warehouse — based on PostgreSQL, optimized for analytics not transactions
▶

Amazon Redshift is a fully managed, petabyte-scale data warehouse service. It is based on PostgreSQL but is optimized for OLAP (Online Analytical Processing) — complex analytical queries over large datasets — not OLTP (row-level transactions like RDS).

→ Columnar storage: data is stored by column rather than by row. Analytical queries that aggregate only a few columns don't read the entire row — dramatically faster for analytics. Enables high compression ratios per column.
→ Massively Parallel Processing (MPP): queries are automatically distributed across all nodes in the cluster. Each node processes a slice of the data in parallel — linear scale-out as you add nodes.
→ Cluster architecture: one Leader Node (receives queries, coordinates execution, aggregates results — no charge) + multiple Compute Nodes (execute query slices, store data — you pay for these). 1–128 compute nodes.
→ Node types: RA3 — separate compute and storage (pay for compute + managed storage in S3 behind the scenes, scale each independently). DC2 — compute-optimized with local NVMe SSD (fixed compute+storage, fast for hot data). Use RA3 for most new workloads.
→ Not Multi-AZ by default: standard Redshift runs in one AZ. For RA3, Multi-AZ is available. Snapshots (automated and manual) are stored in S3 and can be restored in any region.
→ Redshift Spectrum: extend Redshift queries to exabytes of data in S3 without loading it into Redshift. Uses the same SQL, same cluster, reads Parquet/ORC/CSV from S3. Like Athena but from within Redshift. Query local Redshift tables and S3 data in the same query.
→ Redshift Serverless: automatically provisions and scales warehouse capacity — pay per RPU (Redshift Processing Unit) per second. Ideal for variable, unpredictable workloads or development environments.
EXAM TIPRedshift = OLAP, analytics, data warehouse. RDS = OLTP, transactions, operational database. "Query petabytes of data with complex joins and aggregations" → Redshift. "Query S3 data directly from Redshift without ETL" → Redshift Spectrum. Redshift is NOT Multi-AZ by default — this is a frequent exam trap. Use snapshots for DR.
LOAD
Loading Data into Redshift advanced
COPY command, Firehose, and the right ingestion path
▶
→ COPY command: the most efficient way to load data into Redshift. Reads from S3, DynamoDB, EMR, or SSH in parallel across all compute nodes simultaneously. Much faster than INSERT statements for bulk loads.
→ Kinesis Data Firehose → Redshift: Firehose writes to S3 first, then issues a COPY command to Redshift automatically. Best for streaming data ingestion into Redshift.
→ Enhanced VPC Routing: forces all COPY and UNLOAD traffic between Redshift and S3 to go through your VPC instead of the public internet — required for compliance/security in some environments.
EXAM TIP"Fastest way to bulk load data into Redshift" → COPY command from S3 (parallel across all nodes). "Stream data into Redshift in near-real-time" → Firehose → S3 → COPY. Never INSERT row-by-row at scale — always COPY for bulk loads.
Module 3 — Search & Analytics
Amazon OpenSearch Service
Managed Elasticsearch/OpenSearch — full-text search, log analytics, and real-time application monitoring.
OS
OpenSearch Overview core
Successor to Amazon Elasticsearch Service — managed distributed search and analytics engine
▶

Amazon OpenSearch Service is a managed service for deploying, operating, and scaling OpenSearch (and legacy Elasticsearch) clusters. OpenSearch enables full-text search, real-time log analytics, and any use case requiring fast arbitrary field queries and fuzzy/partial matching that SQL databases struggle with.

→ Formerly called Amazon Elasticsearch Service. OpenSearch is the open-source fork of Elasticsearch. All existing Elasticsearch applications are compatible.
→ Unlike SQL databases (query on indexed columns only), OpenSearch can search on any field — including partial text, fuzzy matches, relevance scoring, and nested objects.
→ Data is stored in indices (similar to tables) which are split into shards across nodes. Each shard can have replicas for high availability.
→ Common ingestion patterns: DynamoDB Streams → Lambda → OpenSearch (index DynamoDB data for search), CloudWatch Logs → Subscription Filter → Lambda → OpenSearch (log analytics), Kinesis Data Streams / Firehose → OpenSearch.
→ OpenSearch Dashboards: built-in visualization layer (formerly Kibana) — build dashboards, visualize log data, create metric charts. Accessed via browser.
→ Deployment modes: managed clusters (provision master and data nodes) or OpenSearch Serverless (fully managed, no cluster management, auto-scales, pay per OCU).
→ Security: VPC support, IAM-based access policies, Cognito integration for Dashboards authentication, encryption at rest (KMS) and in transit (TLS).
EXAM TIPOpenSearch vs Athena: OpenSearch = real-time search on indexed documents, arbitrary field queries, log analysis, fuzzy matching. Athena = SQL on S3 files, batch/ad-hoc analytics. "Search product catalog with partial text match" → OpenSearch. "Analyze S3 log files with SQL" → Athena. OpenSearch is the DynamoDB complement — DynamoDB for primary data, OpenSearch for search.
Module 4 — Big Data Processing
Amazon EMR
Managed Hadoop ecosystem — process petabytes of data using Spark, Hive, Presto, and other open-source tools.
EMR
Amazon EMR Overview core
Elastic MapReduce — managed Hadoop/Spark clusters for big data processing
▶

Amazon EMR (Elastic MapReduce) provisions and manages clusters of EC2 instances running open-source big data frameworks. EMR handles the undifferentiated work of installing, configuring, and tuning Hadoop, Spark, HBase, Presto, Hive, and others — you write the data processing code.

→ Supported frameworks: Apache Spark (most popular, in-memory processing), Hive (SQL on Hadoop), Presto (fast ad-hoc SQL), HBase (NoSQL on HDFS), Flink, Hudi, Iceberg.
→ Cluster nodes: Master node (coordinates the cluster, manages resources — one per cluster), Core nodes (run tasks + store data in HDFS — cannot be removed without data loss), Task nodes (run tasks only, no HDFS — can be Spot instances, safe to remove).
→ Long-running vs transient clusters: long-running clusters stay up for ongoing workloads; transient clusters spin up, process a job (data from S3), save results to S3, then terminate. Transient clusters using Spot instances are very cost-effective for batch jobs.
→ EMR on EKS: run EMR workloads on an existing EKS cluster — submit Spark jobs without managing EC2 instances.
→ EMR Serverless: run Spark/Hive jobs without provisioning or managing clusters. AWS auto-provisions workers per job. Pay only for resources used.
→ Use cases: large-scale ETL, log processing, clickstream analysis, genomics, machine learning preprocessing, data transformation before loading into Redshift.
EXAM TIP"Process petabytes of data with Spark/Hadoop" → EMR. "Cost-effective batch processing" → EMR transient cluster with Spot instances for Task nodes. Core nodes should NOT be Spot (risk of HDFS data loss). Task nodes = safe for Spot (no data stored). EMR is the answer for "managed Hadoop/Spark" — if you see those keywords, think EMR.
Module 5 — Business Intelligence
Amazon QuickSight
Serverless BI and data visualization — interactive dashboards for everyone in your organization.
QS
Amazon QuickSight Overview core
Serverless BI — create and share dashboards from any AWS data source
▶

Amazon QuickSight is a fully managed, serverless business intelligence service for creating interactive visualizations and dashboards. It connects to AWS data sources and external databases, requiring no servers and no BI software licensing.

→ Data sources: Athena, Redshift, S3, RDS/Aurora, OpenSearch, Timestream, Salesforce, on-premises SQL databases (via VPC or public endpoint), Jira, and more.
→ SPICE (Super-fast, Parallel, In-memory Calculation Engine): QuickSight's in-memory cache. Import data into SPICE for sub-second query response times, independent of the source database. SPICE is allocated per user and region.
→ Enterprise edition: supports column-level security, row-level security (users see only rows they're allowed to), Active Directory integration, and cross-account data source access.
→ Embedded analytics: embed QuickSight dashboards into your own applications using the Embedding SDK. Users see dashboards within your app without going to the QuickSight console.
→ ML Insights: built-in ML features — anomaly detection (automatically surfaces unusual spikes or drops), forecasting (project future values), and narrative explanations (natural language summaries of chart data).
→ Pricing: per-session (pay per 30-minute session) for readers, per-user/month for authors. No per-query charges unlike Athena.
EXAM TIP"Create dashboards on AWS data for business users" → QuickSight. "Visualize Athena/Redshift query results" → QuickSight. SPICE = in-memory cache that makes dashboards fast without hitting the source. QuickSight is the BI layer — it visualizes data, it doesn't process it. "Ad-hoc SQL analysis" → Athena. "Visualize the Athena results" → QuickSight.
Module 6 — ETL
AWS Glue
Serverless ETL and data catalog — discover, prepare, and integrate data for analytics.
GLU
AWS Glue Overview core
Serverless ETL + central metadata catalog — the glue between your data stores
▶

AWS Glue is a fully managed, serverless ETL (Extract, Transform, Load) service. It has two main components that work together: the Glue Data Catalog (centralized metadata repository) and Glue ETL Jobs (serverless Spark/Python data transformation).

→ Glue Data Catalog: central metadata store for all your data assets — tables, schemas, locations, partition information. Shared and used by Athena, Redshift Spectrum, and EMR. One catalog per region per account. Acts as a persistent Hive metastore.
→ Glue Crawlers: automatically discover data in S3, RDS, Redshift, DynamoDB — infer schema, detect partitions, and populate the Glue Data Catalog. Schedule crawlers to run on a cadence to keep the catalog up to date.
→ Glue ETL Jobs: serverless Apache Spark or Python Shell jobs. Write transformations in Python or Scala. Glue provisions Spark workers, runs the job, and terminates — pay only for DPU-seconds (Data Processing Units) consumed.
→ Glue Studio: visual drag-and-drop interface for building Glue ETL jobs without writing code — connect source → transform → target nodes on a canvas.
→ Glue DataBrew: visual data preparation tool for analysts — clean and normalize data using 250+ pre-built transformations without writing code. Outputs go to S3 or back into the Glue catalog.
→ Glue Elastic Views: replicate and combine data across multiple data stores into a materialized view without custom ETL code.
→ Common Glue workflow: S3 raw data → Glue Crawler (catalog) → Glue ETL Job (transform CSV to Parquet) → S3 processed data → Athena/Redshift queries the catalogued data.
EXAM TIP"Convert CSV files to Parquet in S3" → Glue ETL Job. "Automatically discover schema of S3 data" → Glue Crawler. "Central metadata for Athena, Redshift, and EMR" → Glue Data Catalog. Glue is serverless ETL — no cluster to manage. EMR is for heavy custom big data processing where you need full control of the Spark cluster.
Module 7 — Data Governance
AWS Lake Formation
Build, secure, and manage a data lake in days — fine-grained access control on top of S3 and the Glue Catalog.
LF
Lake Formation Overview core
Data lake governance — centralized permissions for S3 data accessed via Athena, Redshift, and EMR
▶

AWS Lake Formation is a service that makes it easy to set up, secure, and manage a secure data lake. It builds on S3 and the Glue Data Catalog by adding centralized, fine-grained access control — column-level, row-level, and cell-level security — enforced across Athena, Redshift Spectrum, and EMR regardless of the underlying S3 bucket permissions.

→ Data lake setup: Lake Formation orchestrates data ingestion, cataloguing, cleaning, and transformation into S3 — replacing manual Glue Crawler + ETL workflows with a more streamlined experience.
→ Centralized access control: grant and revoke permissions on databases, tables, and columns in the Glue catalog from one place. These permissions are enforced across all Lake Formation-integrated services — no need to manage separate S3 bucket policies per user/service.
→ Fine-grained security: column-level security (user A can see name column but not SSN column in the same table), row-level security (user sees only rows for their region), and cell-level masking (return masked/null value for specific cells).
→ Permissions are applied at query time — even if S3 allows access, Lake Formation can deny it. The Lake Formation permission model overrides IAM and S3 bucket policies for registered data sources.
→ Cross-account data sharing: share data lake tables with other AWS accounts without copying data — the recipient queries via Athena/Redshift and Lake Formation enforces access policies.
→ Built on the Glue Data Catalog — Lake Formation extends Glue's catalog with a permission layer on top.
EXAM TIP"Column-level or row-level security on S3/Athena data" → Lake Formation. "Centralize data lake access control across multiple teams/accounts" → Lake Formation. "Share a data catalog table with another AWS account without copying data" → Lake Formation cross-account sharing. Lake Formation = governance layer on top of Glue + S3.
Module 8 — Real-Time Streaming
Apache Flink & MSK
Managed Apache Flink for stream processing, and Managed Streaming for Apache Kafka for event streaming.
FLK
Amazon Managed Service for Apache Flink advanced
Serverless Apache Flink — real-time stream processing without cluster management
▶

Amazon Managed Service for Apache Flink (formerly Kinesis Data Analytics) lets you run Apache Flink applications on AWS without managing servers or clusters. Flink is a stream processing framework — it processes data in real-time as it flows through, performing stateful computations, aggregations, and transformations.

→ Write Flink applications in Java, Scala, or Python. Deploy without managing clusters — AWS handles provisioning, scaling, and availability.
→ Sources: Kinesis Data Streams, MSK (Kafka), S3, custom sources via Flink connectors.
→ Sinks: S3, Kinesis Data Streams, Kinesis Firehose, OpenSearch, custom sinks.
→ Use cases: real-time fraud detection, anomaly detection on streaming metrics, real-time ETL (filter/enrich/transform events as they arrive), streaming analytics dashboards.
→ Flink vs Kinesis Data Streams: KDS is the transport layer (stores and delivers events). Flink is the processing layer (transforms, aggregates, analyzes events as they flow). They are complementary: KDS → Flink → results to S3/OpenSearch.
EXAM TIP"Process streaming data in real-time with complex stateful aggregations" → Managed Apache Flink. "Detect fraud in real-time as transactions stream in" → Flink. Flink processes streams; KDS stores/delivers streams. They are often used together: MSK/KDS (transport) → Flink (processing) → S3/OpenSearch (destination).
MSK
Amazon MSK — Managed Streaming for Apache Kafka core
Fully managed Kafka — migrate existing Kafka workloads to AWS
▶

Amazon MSK (Managed Streaming for Apache Kafka) is a fully managed service that runs Apache Kafka on AWS. AWS handles Kafka broker provisioning, configuration, patching, and ZooKeeper/KRaft management — you focus on producing and consuming messages.

→ Fully compatible with Apache Kafka — existing Kafka producers, consumers, and Kafka Streams applications work with MSK without code changes.
→ Data is stored on EBS volumes attached to broker nodes. Configurable retention period (unlike KDS which maxes at 365 days, Kafka retention can be much longer or unlimited).
→ Runs inside your VPC — brokers are private by default. Clients connect via private endpoints.
→ MSK Serverless: fully managed Kafka without managing brokers or capacity — auto-scales, pay per partition-hour.
→ MSK Connect: run Kafka Connect connectors (source/sink connectors for databases, S3, OpenSearch, etc.) as a managed service.
→ Consumers using Lambda via MSK Event Source Mapping — Lambda can poll MSK topics and process batches of Kafka messages.

Reference Architecture
Big Data Ingestion Pipeline
How all the data analytics services connect — the canonical end-to-end pipeline architecture.
ARC
End-to-End Data Lake Pipeline core
From real-time ingestion to BI dashboards — the full pipeline every architect should know
▶

The canonical AWS big data pipeline combines streaming ingestion, batch ETL, a queryable data lake, and BI visualization. Understanding how each service plugs into the others is the key to answering architecture questions.

1. Ingestion: IoT devices / apps / servers → Kinesis Data Streams (real-time stream) or MSK (Kafka protocol). For batch: files dropped directly to S3.
2. Real-time processing: KDS/MSK → Managed Apache Flink (stateful stream processing, anomaly detection, enrichment) → results to S3 or OpenSearch.
3. Near-real-time delivery: KDS → Kinesis Data Firehose (buffer, compress, transform via Lambda) → S3 raw data bucket (the data lake landing zone).
4. Cataloguing: Glue Crawlers scan S3 → populate Glue Data Catalog with schemas and partitions.
5. ETL transformation: Glue ETL Jobs (or EMR for heavy workloads) → convert CSV/JSON to Parquet, clean data, join datasets → write to S3 processed zone.
6. Governance: Lake Formation enforces column/row-level access policies on the Glue Catalog tables.
7. Ad-hoc query: Athena queries the Glue-catalogued S3 Parquet data with SQL. Results written to S3.
8. Warehouse: structured/aggregated data → Redshift (COPY from S3) for complex multi-table analytics needing sub-second performance at petabyte scale.
9. Visualization: QuickSight connects to Athena or Redshift → dashboards for business users.
10. Search: OpenSearch receives a copy of events (via Lambda/Firehose) → powers full-text search and real-time log analytics dashboards.
EXAM TIPThe exam often presents a scenario describing parts of this pipeline and asks which services to use for which role. Map each function to its service: real-time transport = KDS/MSK, real-time processing = Flink, delivery to S3 = Firehose, catalog = Glue, ETL = Glue/EMR, governance = Lake Formation, SQL on S3 = Athena, warehouse = Redshift, BI = QuickSight, search = OpenSearch.

TRAP
Common Exam Traps core
The confusing pairs and gotchas most likely to appear
▶
→ Athena vs Redshift: Athena = serverless SQL on raw S3 files, ad-hoc, pay per scan. Redshift = managed data warehouse, structured data, complex multi-table BI at scale. Athena for one-off analysis; Redshift for ongoing BI workloads with high query frequency.
→ Glue vs EMR: Glue = serverless ETL, no cluster management, lower control. EMR = managed clusters with full control over Spark/Hadoop configuration, better for complex workloads. Glue for simple ETL; EMR for heavy custom processing.
→ Athena vs OpenSearch: Athena = SQL batch queries on files. OpenSearch = real-time arbitrary field search on indexed documents. They complement each other — use both in the same architecture.
→ MSK vs Kinesis Data Streams: MSK = Kafka protocol (migration from on-prem). KDS = AWS-native, simpler, tighter integration. Both are streaming transports — choose based on whether Kafka compatibility is needed.
→ Redshift is NOT Multi-AZ by default: standard Redshift runs in one AZ. This is a trap — RDS Multi-AZ is common, but Redshift RA3 Multi-AZ is a separate feature. DR = automated snapshots to S3 + restore in another AZ/region.
→ Lake Formation vs IAM + S3 bucket policies: for data lake security, Lake Formation permissions override and simplify IAM + S3 policies. One place to manage access for all services (Athena, Redshift Spectrum, EMR) instead of per-service IAM policies.
→ Glue Data Catalog is shared: Athena, Redshift Spectrum, and EMR all read from the SAME Glue Data Catalog in the region. Define the table once, query it from any of the three.
