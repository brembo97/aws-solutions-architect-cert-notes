S3 — Module 1.1
S3 Fundamentals
Core concepts — buckets, objects, naming rules, and the S3 data model every exam question assumes you know.
S3
S3 Overview core
Infinitely scalable object storage — the backbone of AWS architectures
▶

S3 (Simple Storage Service) is AWS's core object storage service. It stores files as objects inside buckets, scales infinitely, and is used for everything from static website hosting to data lake storage and backup archives.

→ Buckets are globally unique (across all AWS accounts and all regions) but are created in a specific region. Data never leaves that region unless you explicitly replicate it.
→ Bucket naming rules: 3–63 characters, lowercase letters/numbers/hyphens only, no uppercase, no IP addresses, must start with a letter or number.
→ Objects are the files stored in S3. Maximum object size: 5 TB. A single PUT upload can transfer up to 5 GB — for larger objects you must use Multipart Upload.
→ Objects have a Key (full path from bucket root, e.g. images/2024/photo.jpg), a Value (the data bytes), metadata, tags, and optionally a version ID.
→ S3 has no true folder hierarchy — the key is the full path string. The console shows "folders" as a UI convenience, but they are prefixes in the key name.
→ Default limit: 100 buckets per account (soft limit, can be increased to 1,000 via support). No limit on number of objects per bucket.
→ S3 is strongly consistent since December 2020 — read-after-write consistency for all operations (PUTs, DELETEs, overwrites) in all regions.
EXAM TIPMax single PUT = 5 GB; max object = 5 TB. Use Multipart Upload for files > 5 GB (recommended for > 100 MB). Buckets are regional but names are globally unique. S3 is object storage — not block storage (EBS) or file storage (EFS).
VRS
S3 Versioning core
Keep every version of an object — protects against accidental deletes and overwrites
▶

S3 Versioning preserves every version of an object in a bucket. Overwrites and deletes become recoverable — accidental changes can be undone by restoring a previous version.

→ Versioning is enabled at the bucket level. Once enabled, it can only be suspended (not fully disabled) — all previous versions are retained.
→ Objects uploaded before versioning was enabled get a version ID of null.
→ Deleting a versioned object places a delete marker on it — the object is hidden but not actually removed. To permanently delete: explicitly delete the specific version ID.
→ Suspending versioning does not delete existing versions — it stops creating new ones for future uploads.
→ Versioning is a prerequisite for S3 Replication and MFA Delete.
→ Versioning increases storage costs — every version is billed. Use Lifecycle rules to expire old versions.
EXAM TIP"Protect against accidental deletion" → enable Versioning. Deleting a versioned object creates a delete marker — the object is NOT gone until you delete the specific version. Versioning cannot be turned off once enabled — only suspended.
REP
S3 Replication core
CRR and SRR — async object replication across buckets
▶

S3 Replication automatically and asynchronously copies objects from a source bucket to a destination bucket. Versioning must be enabled on both source and destination.

CRR — Cross-Region ReplicationSRR — Same-Region Replication
ScopeSource and destination in different regionsSource and destination in same region
Use casesCompliance (data residency in multiple regions), lower latency for global users, cross-region DRLog aggregation from multiple buckets, live replication between prod and test accounts
CostData transfer fees between regions applyNo data transfer fee (same region)
→ Replication is asynchronous — there may be a short delay before the object appears in the destination.
→ Only new objects are replicated after replication is enabled. Existing objects are NOT automatically replicated — use S3 Batch Replication to copy existing objects.
→ Delete markers are not replicated by default. This can optionally be enabled with Delete Marker Replication.
→ Replication is not chained — if bucket A replicates to bucket B, and B replicates to C, objects in A do NOT automatically end up in C.
→ Can replicate between buckets in different AWS accounts — requires bucket policy on destination granting the source account permission to write.
EXAM TIPReplication does not copy existing objects — only new uploads after setup. Use Batch Replication for existing objects. Delete markers are NOT replicated by default. Versioning is required on both buckets.

S3 Security
Bucket policies, encryption options, MFA Delete, access logs, and CORS.
POL
S3 Bucket Policies & Access Control core
Resource-based policies that control who can access a bucket and how
▶

S3 uses three access control mechanisms. Understanding when to use each is critical for the exam.

MechanismScopeUse when
Bucket PolicyBucket + objects inside it. JSON resource-based policy attached to the bucket.Grant cross-account access; make a bucket public; require encryption on upload; restrict access to specific IPs or VPC endpoints.
IAM PolicyAttached to IAM users/roles/groups. Controls which S3 actions an identity can perform.Grant access to specific users within your AWS account.
ACLs (Access Control Lists)Object-level or bucket-level XML-based permissions.Legacy mechanism — AWS recommends disabling ACLs and using bucket policies instead. Still tested on exams.
→ Block Public Access settings (4 settings, all on by default at account and bucket level) override any bucket policy that would grant public access. Must be turned off to make a bucket public.
→ Evaluation order: if IAM allows AND bucket policy allows (or is absent) → access granted. If either explicitly denies → denied.
→ For cross-account access: the bucket policy must grant the other account access, AND the other account's IAM policy must allow the action.
EXAM TIP"Block Public Access is still on but the bucket policy allows public access" → the bucket is NOT public. Block Public Access always wins. For static website hosting, Block Public Access must be disabled on the bucket.
SECURITY WARNINGNever disable Block Public Access unless intentionally hosting a public resource. S3 data exposure from misconfigured buckets is one of the most common AWS security incidents.

MFA
S3 MFA Delete advanced
Require MFA to permanently delete object versions or suspend versioning
▶

MFA Delete adds an extra layer of protection — permanently deleting an object version or suspending versioning requires an MFA code in addition to credentials.

→ Can only be enabled/disabled by the bucket owner using the root account — no IAM user can enable it, even admins.
→ Requires Versioning to be enabled on the bucket first.
→ MFA is required for: permanently deleting an object version, and suspending versioning. It is NOT required for: uploading objects, creating delete markers, or listing versions.
→ Can only be configured via the AWS CLI — not the S3 console.
EXAM TIPOnly the root account can enable MFA Delete — not IAM admins. Versioning must be on first. MFA Delete protects against accidental or malicious permanent deletion of specific versions.
LOG
S3 Access Logs tool
Detailed audit trail of every request made to an S3 bucket
▶

S3 Access Logs record every request made to a bucket — who made it, what they requested, and the response. Logs are written to a separate S3 bucket in the same region.

→ Logs contain: requester, bucket name, request time, request action, response status, error code.
→ Never log to the same bucket you are monitoring — this creates an infinite logging loop and your bucket will grow indefinitely.
→ Logs may take up to a few hours to be delivered — they are not real-time.
→ Log files can be analyzed with Amazon Athena (SQL queries on S3) for security analysis or usage reporting.
EXAM TIPA classic trick question: if you set the access log destination to the same bucket being monitored, you create an infinite loop. Always use a separate dedicated logging bucket.
CORS
S3 CORS advanced
Cross-Origin Resource Sharing — allow web pages to access objects from other S3 buckets
▶

CORS (Cross-Origin Resource Sharing) is a browser security feature that blocks web pages from making requests to a different domain than the one that served the page. To allow it, the target server (S3) must respond with the correct CORS headers.

→ S3 CORS is configured on the bucket being requested (the resource bucket), not on the bucket hosting the website.
→ Common scenario: your website is hosted at www.example.com (bucket A) and fetches images from assets.example.com (bucket B). Bucket B needs a CORS rule allowing origin www.example.com.
→ CORS configuration specifies: AllowedOrigins, AllowedMethods (GET, POST, etc.), AllowedHeaders, MaxAgeSeconds.
EXAM TIPIf a JavaScript app hosted in one S3 bucket is getting CORS errors when fetching from another S3 bucket → add a CORS rule on the SOURCE bucket (the one being fetched FROM), not the bucket hosting the website.

LCY
Lifecycle Rules & S3 Analytics core
Automate object transitions and expirations across storage classes
▶

Lifecycle rules automatically transition objects to cheaper storage classes or expire/delete them after a defined number of days — no manual intervention needed.

→ Transition actions: move objects to a cheaper class after N days (e.g. Standard → Standard-IA after 30 days → Glacier after 90 days).
→ Expiration actions: permanently delete objects after N days. Can target: current versions, non-current versions, incomplete multipart uploads, and expired delete markers.
→ Rules can be scoped to a prefix (e.g. logs/) or object tags.
→ Transition constraints: Standard → Standard-IA requires ≥30 days. Standard-IA → Glacier requires ≥30 more days. Cannot skip a tier (must follow waterfall order).
→ S3 Analytics: analyzes access patterns and recommends when to transition from Standard to Standard-IA. Takes 24–48 hours to start generating recommendations. Does NOT recommend Glacier transitions.
EXAM TIP"Automatically move logs to Glacier after 90 days then delete after 365 days" → Lifecycle rule. S3 Analytics helps decide WHEN to set up transitions but only for Standard to Standard-IA. Use it to generate data, then configure lifecycle rules based on the output.
LCK
S3 Object Lock & Glacier Vault Lock advanced
WORM — Write Once Read Many for compliance
▶

Object Lock enforces WORM (Write Once Read Many) storage. Once locked, objects cannot be overwritten or deleted for a set period — even by root. Used for regulatory compliance (SEC, FINRA, HIPAA).

ModeWho can delete?Use case
Compliance ModeNobody — not even root — can delete or shorten the retention period.Strict regulatory compliance where immutability must be absolute.
Governance ModeRoot and users with special IAM permissions can delete.Protect against accidental deletion while allowing admin override.
→ Legal Hold: an alternative lock with no expiry — can be applied/removed by users with s3:PutObjectLegalHold permission. Useful during active litigation.
→ Object Lock requires Versioning to be enabled. It must be enabled at bucket creation — cannot be added to an existing non-versioned bucket.
→ Glacier Vault Lock: similar WORM concept for Glacier Vaults. Uses a Vault Lock Policy — once locked, the policy itself cannot be changed.
EXAM TIP"Prevent object deletion for compliance — not even root can delete" → Compliance Mode Object Lock. "Protect against deletion but allow admin override" → Governance Mode. "Financial/healthcare regulation requiring immutable storage" → Compliance Mode or Glacier Vault Lock.
S3 — Module 1.4
Data Management
Event notifications, batch operations, Storage Lens, and S3 Requester Pays.
EVT
S3 Event Notifications core
Trigger Lambda, SNS, or SQS on S3 object events
▶

S3 Event Notifications let you automatically trigger downstream services when objects are created, deleted, replicated, or have their storage class changed.

→ Supported destinations: Lambda, SNS, SQS, and Amazon EventBridge.
→ Event types include: s3:ObjectCreated:*, s3:ObjectRemoved:*, s3:ObjectRestore:*, s3:Replication:*, s3:LifecycleTransition.
→ Notifications can be filtered by prefix or suffix (e.g. only trigger on .jpg files in the uploads/ prefix).
→ Amazon EventBridge integration: route S3 events to EventBridge for more advanced routing, filtering, and fan-out to many destinations simultaneously. EventBridge also archives events and enables replay.
→ Delivery is typically within seconds but not guaranteed to be exactly once — for exactly-once processing, use SQS FIFO queues.
EXAM TIP"Process uploaded images automatically" → S3 Event Notification → Lambda. "Fan out S3 events to multiple targets" → use EventBridge (much more flexible than direct S3 → Lambda). EventBridge supports filtering, archiving, and replay — S3 direct notifications do not.
BAT
S3 Batch Operations advanced
Run operations on millions of S3 objects with a single request
▶

S3 Batch Operations executes a single operation across billions of S3 objects in bulk — useful for one-time migrations, bulk encryption, or mass tagging.

→ Supported operations: copy objects, replace object tags, set ACLs, restore from Glacier, invoke Lambda per object, set Object Lock, replicate existing objects (Batch Replication).
→ Requires a manifest file (CSV or S3 Inventory report) listing the objects to process.
→ AWS manages retries, tracks progress, sends completion reports, and generates CloudWatch metrics.
→ Use S3 Inventory to generate the list of objects, then feed that into a Batch Operations job.
EXAM TIP"Encrypt all existing unencrypted S3 objects at once" → S3 Batch Operations (copy objects with encryption enabled). "Replicate existing objects to a new bucket" → Batch Replication via Batch Operations. Batch Operations is for bulk one-time operations; Lifecycle rules are for ongoing automation.
LNS
S3 Storage Lens tool
Organization-wide S3 usage and activity visibility
▶

S3 Storage Lens gives you a unified, organization-wide view of S3 storage usage, activity trends, and cost optimization recommendations across all buckets and accounts.

→ Provides metrics across accounts, regions, and buckets — not just a single bucket. Aggregates at the organization level via AWS Organizations.
→ Metrics include: total storage size, object count, average object size, % data that could be moved to cheaper storage classes, replication status.
→ Free tier: 28 default usage metrics. Advanced metrics (activity metrics, CloudWatch integration) are paid.
→ Recommendations highlight cost savings opportunities (e.g. "40% of data in Standard hasn't been accessed in 90 days — consider Intelligent-Tiering").
EXAM TIPStorage Lens = visibility and recommendations across your entire S3 estate. S3 Analytics = transition recommendations for a single bucket. Storage Lens is the org-level tool; S3 Analytics is per-bucket.
RPY
S3 Requester Pays advanced
Shift data transfer costs to the requester instead of the bucket owner
▶

By default, the bucket owner pays for storage and data transfer costs. With Requester Pays, the requester (downloader) pays for the data transfer and request costs instead of the bucket owner.

→ The requester must be authenticated (cannot be anonymous) — they must have an AWS account and the request must include x-amz-request-payer: requester.
→ Use case: sharing large public datasets (e.g. genomics data, satellite imagery) where the bucket owner wants to offset download costs.
EXAM TIP"A company shares a large public dataset and doesn't want to pay for download costs" → S3 Requester Pays. The requester must be authenticated — anonymous access doesn't work with Requester Pays.
S3 — Module 1.5
Access Patterns
Pre-signed URLs, Access Points, Object Lambda, Static Website Hosting, and cross-account access patterns.
PSU
Pre-Signed URLs core
Time-limited URLs that grant temporary access to a private S3 object
▶

A pre-signed URL embeds the creator's credentials and an expiry time into a URL. Anyone with the URL can perform the allowed action (GET or PUT) on the object without needing AWS credentials — until the URL expires.

→ Generated by: an IAM user, an IAM role, or the bucket owner. The URL inherits the permissions of the signing identity at the time of the request.
→ Default expiry: 3,600 seconds (1 hour). Maximum: 604,800 seconds (7 days) when signed with SigV4 and an IAM role or instance profile.
→ Use cases: temporary download links for private content (e.g. paid media, personalized reports), pre-signed upload URLs to let users upload directly to S3 without going through your servers.
→ If the signing identity's permissions are revoked, the pre-signed URL stops working immediately (because access is evaluated at request time using the embedded credentials).
EXAM TIP"Allow a user to download a private S3 object for 2 hours without giving them AWS credentials" → pre-signed URL. "Allow a mobile app to upload a file directly to S3" → pre-signed PUT URL. Avoids your servers acting as a proxy for large uploads/downloads.
APT
S3 Access Points advanced
Named access endpoints with their own policies — simplify large-scale access management
▶

Access Points are named network endpoints attached to a bucket. Each has its own DNS name and access policy, allowing you to create separate permission policies for different applications or teams — without maintaining one giant bucket policy.

→ Each Access Point has its own unique DNS hostname and its own access point policy (like a bucket policy scoped to a prefix).
→ Access Points can be restricted to a specific VPC — traffic only accessible from within that VPC (requires a VPC endpoint).
→ The bucket policy must grant Access Points permission to access the bucket (or use aws:sourceAccount condition).
→ Example: one Access Point for the analytics team (read-only on data/ prefix), another for the finance team (read/write on reports/ prefix) — cleaner than one complex bucket policy.
EXAM TIP"Many teams need different access to different prefixes in one bucket — the bucket policy is getting unwieldy" → S3 Access Points. Each Access Point = separate policy scoped to a prefix. VPC-restricted Access Points ensure data never leaves the VPC.
OLM
S3 Object Lambda advanced
Transform S3 objects on-the-fly as they are retrieved
▶

S3 Object Lambda adds a Lambda function to the retrieval path of S3 objects. When an application requests an object via a special Object Lambda Access Point, the Lambda function intercepts the request, transforms the data, and returns the modified result.

→ Works via a special Object Lambda Access Point — the application uses this endpoint instead of the regular S3 endpoint.
→ The original object in S3 is never modified — the transformation happens in-flight.
→ Use cases: redact PII from CSVs before returning to analytics users, convert image formats on-the-fly, add watermarks, decompress files, enrich data from other systems.
→ Eliminates the need to maintain multiple copies of the same data in different formats.
EXAM TIP"Return PII-redacted version of S3 objects to the analytics team, and the full version to the data science team, without duplicating the data" → S3 Object Lambda with two access points. One Lambda redacts PII; the other passes through unmodified.
WEB
S3 Static Website Hosting core
Host a static website directly from an S3 bucket
▶

S3 can serve static websites (HTML, CSS, JS, images) directly from a bucket. It generates an HTTP endpoint but does not support HTTPS — use CloudFront for HTTPS.

→ URL format: http://<bucket-name>.s3-website.<region>.amazonaws.com
→ Requires: (1) static website hosting enabled on the bucket, (2) an index.html specified as the index document, (3) Block Public Access turned OFF, (4) a bucket policy granting s3:GetObject to everyone (*).
→ S3 website hosting is HTTP only — to add HTTPS or a custom domain, put CloudFront in front.
→ You can configure a custom error document (e.g. 404.html).
EXAM TIPIf you need HTTPS for an S3 static website → CloudFront + S3 (not S3 alone). If a 403 Forbidden is returned when visiting the website → Block Public Access is still on, or the bucket policy doesn't allow s3:GetObject to public.
S3 — Module 1.6
Performance & Operations
Multipart Upload, Transfer Acceleration, byte-range fetches, and maximizing S3 throughput.
PRF
S3 Performance Optimization core
Multipart Upload, Transfer Acceleration, byte-range fetches, and request rate limits
▶

S3 scales automatically to high request rates, but knowing the performance features and limits is frequently tested.

→ Request rate limits: S3 can handle 3,500 PUT/COPY/POST/DELETE and 5,500 GET/HEAD requests per second, per prefix. Spread objects across multiple prefixes to scale beyond these limits.
→ Multipart Upload: required for objects > 5 GB; recommended for objects > 100 MB. Splits a file into parts, uploads in parallel, then S3 assembles them. Parts can be retried individually on failure.
→ S3 Transfer Acceleration: uploads travel to the nearest AWS Edge Location (CloudFront PoP) over the public internet, then traverse AWS's private backbone to the target S3 region. Helps for uploads from geographically distant locations. Costs extra.
→ Byte-Range Fetches: request specific byte ranges of an object in parallel — like multipart for downloads. Dramatically improves download throughput for large files. Also useful to retrieve just the header of a file.
→ Old guidance about random key prefixes for performance is no longer relevant — S3 partitions automatically based on request load since 2018.
EXAM TIP"Speed up global uploads to S3" → Transfer Acceleration. "Upload a 10 GB file reliably" → Multipart Upload. "Speed up large object downloads" → Byte-Range Fetches. Transfer Acceleration uses the same endpoint format: bucket.s3-accelerate.amazonaws.com.
S3 — Module 1.7
Advanced S3
S3 Express One Zone, S3 Select, and the overall S3 architecture for exam scenarios.
EXP
S3 Express One Zone advanced
Highest-performance S3 — single-digit millisecond latency, single AZ
▶

S3 Express One Zone is a storage class for latency-sensitive applications requiring consistent single-digit millisecond response times. It uses a different bucket type — S3 Directory Buckets — and stores data in a single AZ.

→ Up to 10x faster than standard S3 with consistent sub-10ms latency. Request costs are lower; storage costs are higher than Standard.
→ Uses Directory Buckets — a different bucket type from standard S3 (general-purpose buckets). Directory Buckets have different naming conventions and limits.
→ Data is stored in one AZ only — no redundancy across AZs. Data is lost if the AZ fails.
→ Use cases: ML training data access, high-frequency analytics, interactive applications needing consistent low latency.
→ Works well co-located with compute (EC2, Lambda, ECS) in the same AZ for maximum performance.
EXAM TIP"Consistently sub-10ms S3 access for ML training or analytics" → S3 Express One Zone. Trade-off: single AZ (lower durability) for maximum performance. Contrast with One Zone-IA: both are single AZ, but Express One Zone is about latency; One Zone-IA is about cost savings for infrequent access.
SEL
S3 Select & Glacier Select advanced
Run SQL queries on S3 objects without downloading the whole file
▶

S3 Select lets you use SQL expressions to retrieve a subset of data from an S3 object (CSV, JSON, Parquet) — filtering happens on the S3 side, so you only transfer the data you need.

→ Supports CSV, JSON, and Parquet formats with GZIP or BZIP2 compression (CSV and JSON only).
→ Can reduce data transfer by up to 400x and improve performance by up to 400% compared to downloading the full object and filtering client-side.
→ Glacier Select: same concept for objects stored in Glacier Flexible Retrieval — run SQL queries during the retrieval process.
EXAM TIPS3 Select is about cost and performance optimization — retrieve only the rows/columns you need from a large CSV or Parquet file. Contrast with Athena: Athena runs SQL across many S3 objects (data lake queries); S3 Select queries a single object.

AWS Snow Family
Physical devices for offline data migration and edge computing when the network isn't fast enough.
SNW
Snow Family Overview core
Snowcone, Snowball Edge, Snowmobile — physical data transfer devices
▶

The Snow Family are physical appliances that AWS ships to your location. You load data onto them locally, ship them back to AWS, and AWS imports the data into S3. Used when transferring data over the internet would take too long or is not feasible.

Snowball Edge Storage Optimized80 TB usable40 vCPUs, 80 GB RAMLarge-scale data migrations. Edge computing with limited needs.

→ Snowball devices support EC2-compatible compute (AMIs) and Lambda functions at the edge — run local processing before uploading.
→ Data is automatically encrypted on the device using KMS. AWS erases devices after data is imported (NIST 800-88 compliant).
→ Snowball cannot import directly into Glacier — data goes into S3 Standard first, then you use a Lifecycle rule to transition to Glacier.
EXAM TIP"Migrate 50 TB of data, network bandwidth is limited" → Snowball Edge. "Migrate 100 PB" → Snowmobile. "Snowball → Glacier" → not possible directly; use S3 → Lifecycle rule → Glacier. Snowcone is the smallest form factor. Compute Optimized = most vCPUs/RAM for edge processing.

Storage Extras — Module 2.3
AWS Storage Gateway
Bridge between on-premises infrastructure and AWS cloud storage — three gateway types for different use cases.
GWY
Storage Gateway Types core
S3 File Gateway, FSx File Gateway, Volume Gateway, Tape Gateway
▶

Storage Gateway is a hybrid cloud storage service that connects on-premises environments to AWS storage (S3, FSx, EBS). It runs as a VM appliance (or hardware appliance) on-premises and caches frequently used data locally for low-latency access.

Storage Extras — Module 2.4
Transfer Family & DataSync
Managed file transfer protocols into S3/EFS, and high-speed automated data synchronization.
TRN
AWS Transfer Family core
Managed SFTP, FTPS, FTP, and AS2 endpoints for S3 and EFS
▶

AWS Transfer Family provides fully managed file transfer endpoints for SFTP, FTPS, FTP, and AS2 protocols. Files are transparently stored in S3 or EFS — your trading partners and legacy systems keep using the protocols they already know.

→ Supported protocols: SFTP (SSH File Transfer Protocol), FTPS (FTP over TLS), FTP (unencrypted — use only in private networks), AS2 (Applicability Statement 2 — EDI/B2B).
→ Storage backends: Amazon S3 or Amazon EFS. Files transferred via SFTP land directly in S3 or EFS.
→ Supports authentication via: service-managed (user/password or SSH keys stored in Transfer Family), Active Directory (via AWS Directory Service), LDAP, custom identity provider (Lambda).
→ Multi-AZ endpoints available for high availability.
→ Pricing: hourly per endpoint + per GB transferred.
EXAM TIP"Partners send files via SFTP to S3 without code changes" → AWS Transfer Family. "Replace on-premises SFTP server" → Transfer Family. "B2B EDI file exchange" → Transfer Family with AS2. Transfer Family = protocol compatibility bridge to S3/EFS.
DSY
AWS DataSync core
High-speed, scheduled, automated data transfer between on-premises and AWS
▶

DataSync is a fully managed data transfer service for moving large amounts of data between on-premises storage and AWS (S3, EFS, FSx), or between AWS storage services. It handles scheduling, monitoring, data validation, and network optimization automatically.

→ Can transfer data up to 10x faster than open-source tools by using parallel multi-threaded transfers and network optimization.
→ Supported sources and destinations: NFS shares, SMB shares, HDFS, S3 API-compatible storage, S3 (all classes), EFS, FSx for Windows, FSx for Lustre, FSx for ONTAP, FSx for OpenZFS.
→ Runs an agent (VM) on-premises to connect to your on-prem storage. Agent connects outbound to AWS — no inbound firewall rules needed.
→ Automatically handles: data integrity verification (checksums), encryption in transit (TLS), bandwidth throttling, scheduling, and retry logic.
→ Can also sync between AWS storage services (e.g. S3 to EFS, S3 to another S3 bucket in a different region) — no agent needed for AWS-to-AWS transfers.
EXAM TIPDataSync vs Storage Gateway: DataSync = one-time or scheduled migrations (move/sync data into AWS). Storage Gateway = ongoing hybrid access (on-prem servers continuously accessing cloud storage). DataSync vs S3 Replication: DataSync can transfer to EFS and FSx; S3 Replication only copies between S3 buckets.
