Module 1 — Encryption & KMS
Encryption 101 & AWS KMS
Encryption fundamentals and the AWS Key Management Service — the backbone of data protection across all AWS services.
101
Encryption Fundamentals core
At-rest vs in-transit, symmetric vs asymmetric, envelope encryption
▶

Encryption converts plaintext data into unreadable ciphertext using a key. Without the key, the data is worthless. AWS services apply encryption at two layers: at rest (stored data) and in transit (data moving over a network).

→ Encryption at rest: protects data stored on disk, in databases, or in object storage. Uses symmetric encryption (AES-256). The same key encrypts and decrypts. Examples: SSE-S3, SSE-KMS, EBS encryption, RDS encryption.
→ Encryption in transit: protects data moving over a network — between services, between client and server. Uses TLS/SSL (asymmetric for key exchange, then symmetric for bulk data). Examples: HTTPS, TLS between app and RDS, encrypted S3 API calls.
→ Symmetric encryption: one key for both encrypt and decrypt (AES-256). Fast, used for bulk data. The challenge: how to securely share the key with the other party.
→ Asymmetric encryption: a key pair — public key (encrypt only, can share freely) and private key (decrypt only, never share). RSA, Elliptic Curve. Slower than symmetric. Used for key exchange, digital signatures, TLS handshake.
→ Envelope encryption: the KMS pattern. A Data Encryption Key (DEK) encrypts the actual data. The DEK itself is encrypted by a KMS Master Key (CMK/KMS Key). Only the encrypted DEK is stored with the data — the plaintext DEK is never persisted. This way KMS only needs to handle small key operations, not large data volumes.
EXAM TIPEnvelope encryption = DEK encrypts data, KMS key encrypts the DEK. This is how S3, EBS, RDS, and virtually every AWS encryption works. "Why not just encrypt everything directly with KMS?" — KMS limits you to 4 KB payload. Envelope encryption lets you encrypt arbitrary-size data efficiently while keeping the master key centrally managed.
KMS
AWS KMS Overview core
Centrally manage encryption keys — audit usage, control access, rotate automatically
▶

AWS Key Management Service (KMS) is a managed service for creating, managing, and controlling cryptographic keys. It is the central key management layer used by virtually every AWS service that offers encryption. KMS integrates with CloudTrail — every key usage is logged.

→ Three KMS key types: AWS Managed Keys (created automatically by AWS services like S3, EBS — free, cannot customize key policy, auto-rotate every year, named aws/service), Customer Managed Keys (CMK) (you create, manage, define key policy, rotate — $1/month/key), AWS Owned Keys (internal AWS keys shared across accounts — free, no visibility).
→ Key material origins: KMS-generated (default — AWS generates and manages key material in FIPS 140-2 validated HSMs), External (you import your own key material — you manage rotation), CloudHSM (key material generated in your dedicated HSM cluster).
→ KMS key operations: Encrypt (up to 4 KB plaintext), Decrypt, GenerateDataKey (generate a DEK — returns plaintext + encrypted copy), GenerateDataKeyWithoutPlaintext (for deferred use), Sign/Verify (asymmetric keys only), GetPublicKey.
→ Key policies: resource-based policy attached to each KMS key. Controls who can use and manage the key. Unlike IAM (where the root account implicitly has all permissions), KMS requires an explicit key policy that grants root access — otherwise the root cannot use the key.
→ Automatic key rotation: Customer Managed Keys can be set to auto-rotate annually. AWS keeps old key material for decryption; new material is used for encryption going forward. The key ID stays the same — application code changes are not required.
→ KMS limits: 10,000 API requests per second per region (can be increased). Each Encrypt/Decrypt/GenerateDataKey call counts. At high S3 request rates with SSE-KMS, consider S3 Bucket Keys to reduce KMS API calls by up to 99%.
→ CloudTrail integration: every KMS API call is logged — who used the key, when, for what. Provides a full audit trail of cryptographic operations.
EXAM TIPKMS max direct encryption: 4 KB. Larger data → envelope encryption (GenerateDataKey). AWS Managed Keys = free, auto-rotate, cannot customize. Customer Managed Keys = $1/month, full control. "Audit who used an encryption key" → CloudTrail + KMS. Key policy must explicitly grant root access — root does NOT implicitly have KMS key access.
Module 2 — KMS Advanced
KMS Advanced Features
Multi-Region Keys, cross-account encryption patterns, S3 replication with KMS, and sharing encrypted AMIs.
MRK
KMS Multi-Region Keys advanced
Replicate keys across regions — decrypt in a different region without re-encryption
▶

KMS Multi-Region Keys are a set of interoperable KMS keys in different AWS regions that have the same key material and key ID. Data encrypted with the primary key in one region can be decrypted using the replica key in another region — without cross-region API calls.

→ Multi-Region Key IDs always start with mrk- (vs single-region keys that start with a UUID). The same key ID works in all regions where the key is replicated.
→ Each replica key is an independent KMS key in its region — has its own key policy, grants, and CloudTrail logs. They share key material but are managed separately.
→ Use cases: global DynamoDB tables (encrypt in us-east-1, replicate to eu-west-1, decrypt in eu-west-1 without cross-region KMS calls), cross-region disaster recovery, global client-side encryption.
→ Multi-Region Keys are NOT the default — most use cases don't need them. Use when: you need to decrypt data in a different region using the same key, or you're building globally distributed applications with local encryption/decryption.
EXAM TIP"Encrypt DynamoDB data in one region and decrypt in another without re-encrypting" → KMS Multi-Region Key. Multi-Region Keys share key material, have the same key ID with mrk- prefix, but each replica is independently managed. Without Multi-Region Keys, you'd need to call KMS in the original region to decrypt — adds latency and creates a dependency.
S3R
S3 Replication with Encryption advanced
Replicate SSE-KMS encrypted objects — key permissions and Multi-Region Key options
▶

S3 Replication works with SSE-S3 and SSE-KMS encrypted objects, but SSE-KMS replication requires explicit configuration because objects must be decrypted in the source region and re-encrypted in the destination region.

→ SSE-S3 encrypted objects: replicated by default — no extra configuration needed. AWS handles the key.
→ SSE-KMS encrypted objects: must explicitly opt in to replicate. The replication configuration must specify: (1) the KMS key to use for encryption in the destination, (2) the IAM role must be granted kms:Decrypt on the source key and kms:Encrypt on the destination key.
→ Replication with SSE-KMS is slower (throttled by KMS API limits) — each object requires a Decrypt + Encrypt KMS call. Use S3 Bucket Keys to reduce KMS call volume.
→ Multi-Region KMS Keys for S3 replication: objects are still re-encrypted (even with Multi-Region Keys, S3 replication always decrypts and re-encrypts), but the same key ID can be used in both regions, simplifying key management.
EXAM TIP"Replicate SSE-KMS encrypted S3 objects" → must configure SSE-KMS in the replication rule, must grant the replication IAM role Decrypt (source key) + Encrypt (destination key). SSE-C encrypted objects CANNOT be replicated — customer provides key per request, replication service can't provide it.
AMI
Encrypted AMI Sharing Process advanced
Share an AMI encrypted with a CMK across AWS accounts
▶

Sharing an encrypted AMI between AWS accounts is more complex than sharing an unencrypted one because the KMS key used to encrypt the underlying EBS snapshot must also be shared.

→ Step 1: The AMI must be encrypted with a Customer Managed Key (CMK) — not an AWS Managed Key. AWS Managed Keys cannot be shared cross-account.
→ Step 2: Modify the KMS key policy in Account A to grant Account B the permissions: kms:DescribeKey, kms:ReEncrypt*, kms:CreateGrant, kms:Decrypt.
→ Step 3: Share the AMI (modify launch permissions) with Account B's account ID.
→ Step 4: Account B can now launch instances from the shared AMI. Account B can optionally copy the AMI and re-encrypt it with their own CMK for independence from Account A's key.
EXAM TIPAMI encrypted with an AWS Managed Key cannot be shared across accounts — must use a Customer Managed Key. Both the AMI AND the underlying KMS key must be shared. The receiving account must have KMS key permissions granted via the source account's key policy.
Module 3 — Hardware Security
AWS CloudHSM
Dedicated hardware security modules — your own tamper-resistant cryptographic hardware in AWS.
HSM
AWS CloudHSM advanced
FIPS 140-2 Level 3 dedicated hardware — you manage the keys, AWS manages the hardware
▶

CloudHSM provides dedicated Hardware Security Modules (HSMs) in your AWS account. Unlike KMS (where AWS manages the HSMs and your key material), CloudHSM gives you exclusive access to the hardware — you manage the keys and users, AWS manages only the physical hardware and availability.

Module 4 — Secrets & Config
SSM Parameter Store
Centralized, hierarchical config and secrets storage — integrated with IAM, KMS, and CloudFormation.
SSM
SSM Parameter Store Overview core
Hierarchical key-value store for configuration data and secrets — Standard vs Advanced tiers
▶

AWS Systems Manager Parameter Store provides secure, hierarchical storage for configuration data and secrets. It is free for standard parameters, deeply integrated with IAM for access control, and natively integrated with KMS for encryption.

→ Hierarchical naming: parameters are organized in a path structure (e.g. /myapp/prod/db-password, /myapp/dev/db-password). You can retrieve all parameters under a path prefix in one API call — making per-environment config management clean.
→ Parameter types: String (plain text), StringList (comma-separated values), SecureString (encrypted with KMS — value is encrypted at rest, decrypted on access by authorized callers).
→ Standard vs Advanced tiers: Standard = free, up to 10,000 parameters, max 4 KB value, no parameter policies. Advanced = $0.05/parameter/month, up to 100,000 parameters, max 8 KB, supports parameter policies (TTL/expiry, notification on change).
→ Versioning: every change to a parameter creates a new version. You can retrieve a specific version (/myapp/db-password:3) or always get the latest. Useful for rollback and audit.
→ IAM access control: use IAM policies to grant access to specific parameter paths. E.g. allow EC2 instances in production to read only /myapp/prod/*.
→ EventBridge integration: trigger Lambda or SNS when a parameter changes — useful for automated redeployment when config changes.
→ Native integration with: EC2 (CloudWatch Agent config), CloudFormation (reference parameters in templates), Lambda (read config at startup), ECS Task Definitions (inject config as environment variables).
EXAM TIPSSM Parameter Store = free for basic use, hierarchical paths, versioning, KMS encryption option. No automatic rotation (use Lambda + EventBridge to rotate manually). If you need automatic rotation → Secrets Manager. "Store database passwords securely for Lambda to retrieve" → SSM SecureString or Secrets Manager. Cost difference: SSM = free (Standard), Secrets Manager = $0.40/secret/month.
Module 5 — Secrets Management
AWS Secrets Manager
Purpose-built secrets management with automatic rotation — the right choice for database credentials and API keys.
SEC
AWS Secrets Manager core
Automatic credential rotation, native RDS/Aurora/Redshift integration, cross-account sharing
▶

AWS Secrets Manager is a dedicated secrets management service designed specifically for storing, rotating, and accessing sensitive credentials — database passwords, API keys, OAuth tokens. Its key differentiator from SSM Parameter Store is automatic rotation.

→ Automatic rotation: the defining feature. Secrets Manager uses a Lambda function to automatically rotate credentials on a configurable schedule. For RDS, Aurora, Redshift, DocumentDB — AWS provides built-in rotation Lambda functions that update both the secret AND the database password simultaneously.
→ Native database integration: for RDS (MySQL, PostgreSQL, Oracle, SQL Server), Aurora, Redshift, and DocumentDB — Secrets Manager natively understands the credential format and how to rotate passwords. Rotation happens without application downtime.
→ Encryption: all secrets are encrypted at rest with KMS (using an AWS Managed Key aws/secretsmanager by default, or a CMK you specify).
→ Cross-account access: a resource-based policy on the secret can allow other AWS accounts to retrieve it. Cross-account rotation requires additional Lambda permissions.
→ Multi-Region Secrets: replicate secrets to multiple AWS regions. The replica is read-only and stays in sync with the primary. Used for multi-region applications that need local secret access (DR failover, global apps).
→ Cost: $0.40 per secret per month + $0.05 per 10,000 API calls. More expensive than SSM Parameter Store (free for Standard).

The automatic rotation capability is the key differentiator — if the question mentions rotation, it's almost always Secrets Manager.
Module 6 — TLS Certificates
AWS Certificate Manager (ACM)
Provision, manage, and auto-renew free TLS/SSL certificates for use with AWS services.
ACM
AWS Certificate Manager core
Free TLS certificates with automatic renewal for CloudFront, ALB, API Gateway, and more
▶

ACM provisions, manages, and auto-renews TLS/SSL certificates for use with integrated AWS services. Public certificates are completely free — no cost for the certificate itself, only for the AWS services using it.

→ Certificate types: Public certificates (free, issued by Amazon's Certificate Authority, auto-renewed, only usable with AWS services) and Private certificates (requires AWS Private CA, costs $400/month for the CA + $0.75/cert, use for internal services).
→ Supported services: ALB, NLB, CloudFront, API Gateway, Elastic Beanstalk, CloudFormation, AppSync. Cannot be used on EC2 instances directly — ACM certificates are not exportable (private key never leaves ACM).
→ Automatic renewal: ACM automatically renews certificates before expiry (typically 60 days before expiry). No manual renewal. Renewal is fully transparent — no downtime.
→ Domain validation methods: DNS validation (recommended — add a CNAME record to your DNS; ACM checks it; auto-renewable as long as CNAME stays) and Email validation (sends email to domain owner contacts — must manually re-validate at renewal).
→ CloudFront certificates MUST be in us-east-1 — regardless of the CloudFront distribution's origin region. All other services use certificates in the same region as the service.
→ Supports wildcard certificates (*.example.com) and multi-domain (SAN) certificates — one certificate for multiple domains.
→ Importing third-party certificates is supported — upload cert + private key + certificate chain to ACM. ACM does NOT auto-renew imported certificates — you must manually renew and re-import.
EXAM TIP"Add HTTPS to an ALB for free" → ACM public certificate on the ALB. CloudFront certificates → must be in us-east-1. "ACM certificate on EC2 directly" → NOT POSSIBLE — ACM certificates are not exportable. Use ACM on an ALB in front of EC2 instead. DNS validation is preferred over email for auto-renewal.
Module 7 — Network Security
WAF & AWS Shield
Layer 7 web application firewall and DDoS protection at the network and application layers.
WAF
AWS WAF core
Layer 7 web application firewall — protect against SQL injection, XSS, bad bots, and more
▶

AWS WAF (Web Application Firewall) filters HTTP/HTTPS traffic at Layer 7 using customizable rules. It protects web applications from common exploits like SQL injection, cross-site scripting (XSS), and bot traffic — before requests reach your application.

→ Deployable on: CloudFront (global, closest to users), ALB, API Gateway, AWS AppSync, Cognito User Pool. Not on NLB — WAF is Layer 7 only; NLB is Layer 4.
→ Web ACL (Access Control List): the WAF policy. Contains rules and rule groups. Rules are evaluated in priority order. Each rule returns Allow, Block, Count, or CAPTCHA.
→ Rule conditions: IP addresses/ranges, geographic origin (country), HTTP headers (User-Agent, Referer), URI strings, query string parameters, request body (first 64 KB inspected), HTTP method, size constraints.
→ Managed Rule Groups: pre-built rule sets from AWS and third-party Marketplace sellers — covering OWASP Top 10, known bad IPs, WordPress vulnerabilities, PHP vulnerabilities, bot detection, etc. No need to write rules from scratch.
→ Rate-based rules: block IPs that exceed a request threshold within a 5-minute window — protects against HTTP flood attacks and brute force. Set at any threshold above 100 requests per 5 minutes.
→ IP Sets: maintain lists of blocked or allowed IPs that can be shared across Web ACLs and updated dynamically (e.g. via Lambda that reads from a threat intelligence feed).
EXAM TIP"Block SQL injection and XSS" → WAF. "Block a specific country from accessing your API" → WAF Geo Match rule. "Rate-limit requests from a single IP" → WAF rate-based rule. WAF is NOT deployed on NLB (Layer 4). WAF on CloudFront = global protection at the edge.
SLD
AWS Shield core
DDoS protection — Standard (free) and Advanced (paid with response team)
▶

AWS Shield protects AWS resources against Distributed Denial of Service (DDoS) attacks. All AWS accounts get Shield Standard automatically. Shield Advanced is a paid tier for enhanced protection of critical workloads.

BPS
DDoS Protection Best Practices advanced
Architecture patterns that minimize DDoS attack surface
▶
→ Use CloudFront in front of everything: absorbs volumetric attacks at edge. Traffic scrubbed before reaching your origin. Protects origins by hiding their IPs.
→ Global Accelerator for non-HTTP: provides static IPs and reroutes around DDoS — traffic enters AWS backbone immediately.
→ Route 53 for DNS resilience: Route 53 is distributed across many PoPs — very hard to DDoS. Use Alias records to route to CloudFront/ALB, not directly to EC2 IPs.
→ WAF on CloudFront or ALB: rate-limit requests, block geographic regions, filter application-layer attack patterns before they reach origin servers.
→ Auto Scaling + ELB: absorb large traffic increases at the application layer. Scale out to handle load while Shield/WAF filters attack traffic.
→ Obfuscate origins: backend EC2 instances and ALBs in private subnets. Only CloudFront has the origin address — attackers can't directly target your servers.
Module 8 — Centralized Security
AWS Firewall Manager
Centrally configure and enforce security rules across all accounts in your AWS Organization.
FWM
AWS Firewall Manager advanced
Organization-wide security policy management — WAF, Shield, Security Groups, Network Firewall
▶

AWS Firewall Manager is a security management service that lets you centrally configure and enforce security policies across all accounts in your AWS Organization. New accounts automatically get the policies applied.

→ Manages: WAF rules (apply the same Web ACL across all CloudFronts/ALBs in the org), Shield Advanced (enable for all accounts centrally), Security Groups (enforce baseline SG rules org-wide), Network Firewall policies, Route 53 Resolver DNS Firewall rules.
→ Requires: AWS Organizations (must be the management account or delegated admin), AWS Config must be enabled in all accounts (Firewall Manager uses Config to assess compliance).
→ New resources and new accounts automatically inherit the Firewall Manager policies — enforcing security as teams create new ALBs, CloudFronts, or VPCs.
EXAM TIP"Enforce the same WAF rules across all ALBs in all AWS accounts in the organization" → Firewall Manager. "Automatically apply WAF to new CloudFront distributions created in any account" → Firewall Manager. Requires Organizations + Config enabled. WAF is the rule engine; Firewall Manager is the organization-wide deployment mechanism.
Module 9 — Threat Detection
Amazon GuardDuty
Intelligent threat detection using ML — find malicious behavior across your AWS account, no setup required.
GD
Amazon GuardDuty core
ML-powered threat detection — finds account compromise, crypto-mining, and unusual API calls
▶

Amazon GuardDuty is a threat detection service that continuously analyzes your AWS account activity using machine learning, anomaly detection, and integrated threat intelligence — without requiring agents or any infrastructure changes.

→ Data sources analyzed: CloudTrail Management Events (API calls), CloudTrail S3 Data Events (S3 access patterns), VPC Flow Logs (network traffic), DNS Logs (domain name queries), EKS Audit Logs, RDS Login Activity, Lambda Network Activity, Runtime Monitoring (EC2/EKS/ECS process behavior).
→ GuardDuty does NOT use its own log collection — it reads from the existing CloudTrail, VPC Flow Logs, and DNS logs your account already generates. Zero infrastructure impact.
→ Threat examples GuardDuty detects: cryptocurrency mining (EC2 communicating with known mining pools), compromised credentials (unusual API calls from unexpected geographies), data exfiltration (S3 data being accessed from unusual IPs), lateral movement (instance calling anomalous internal IPs), port scanning from EC2 instances, Tor network communication.
→ Findings are delivered as JSON to GuardDuty console and can be sent to EventBridge to trigger automated remediation (Lambda to revoke IAM credentials, isolate an EC2 instance, block an IP in WAF).
→ 30-day free trial. After trial, charged based on volume of data analyzed. Enable with one click — no agents, no log forwarding setup.
→ Multi-account: designate a GuardDuty administrator account (via Organizations) that aggregates findings from all member accounts in one view.
→ Trusted IP lists and Threat lists: add known-safe IP ranges (bypasses findings for those IPs) and known-bad IP lists (immediately flag any contact with those IPs).
EXAM TIP"Detect compromised EC2 instances communicating with known malware C&C servers" → GuardDuty. "Detect cryptocurrency mining in my account" → GuardDuty. "Automatically respond to GuardDuty findings" → GuardDuty → EventBridge → Lambda. GuardDuty = passive detection. To actively block → configure EventBridge + Lambda remediation. Disabling GuardDuty = disable the detector (don't just suspend — suspend keeps billing).
Module 10 — Vulnerability & Data Security
Amazon Inspector & Amazon Macie
Continuous vulnerability scanning for EC2/containers, and sensitive data discovery for S3.
INS
Amazon Inspector core
Automated CVE scanning for EC2 instances, Lambda, and container images in ECR
▶

Amazon Inspector is a continuous vulnerability management service that automatically scans EC2 instances, Lambda functions, and container images in ECR for known software vulnerabilities (CVEs) and unintended network exposure.

→ What it scans: EC2 instances (OS vulnerabilities in packages, unintended network exposure via Security Groups), ECR container images (OS and application packages in images, scanned when pushed), Lambda functions (software libraries in the deployment package).
→ Inspector uses the SSM Agent on EC2 instances for software inventory — no separate Inspector agent needed. Agentless scanning is also available for some checks.
→ Findings are centralized in the Inspector console and sent to EventBridge and AWS Security Hub. Each finding has a risk score based on CVE severity + exploitability + exposure.
→ Inspector continuously re-evaluates findings as new CVEs are published — no need to re-trigger scans manually. If a new CVE is disclosed and your instance is vulnerable, Inspector surfaces it automatically.
→ Multi-account: delegate administration to a central account via AWS Organizations — aggregate findings from all accounts.
EXAM TIP"Scan EC2 instances for software vulnerabilities" → Amazon Inspector. "Scan ECR container images for CVEs on push" → Amazon Inspector (ECR Enhanced Scanning). "Scan Lambda function code for vulnerabilities" → Amazon Inspector. Inspector = vulnerability scanning (software CVEs + network exposure). GuardDuty = behavioral threat detection (runtime + network anomalies).
MAC
Amazon Macie core
ML-powered sensitive data discovery in S3 — find PII before it becomes a compliance problem
▶

Amazon Macie is a data security service that uses machine learning to automatically discover, classify, and protect sensitive data stored in Amazon S3. It identifies Personally Identifiable Information (PII) and other sensitive data types across your S3 estate.

→ Macie scans S3 objects and automatically classifies data containing: PII (names, email addresses, SSNs, passport numbers, credit card numbers, bank account numbers, driver's license numbers), health information, intellectual property, and custom data types you define.
→ Also identifies S3 bucket security risks: publicly accessible buckets, buckets not encrypted, buckets shared with external accounts, buckets without logging enabled.
→ Findings are sent to EventBridge — trigger Lambda to quarantine or re-encrypt discovered sensitive data, or notify a compliance team via SNS.
→ Use cases: GDPR compliance (find all S3 objects containing EU citizen data), PCI-DSS (find credit card numbers stored in S3), HIPAA (find PHI in S3 that shouldn't be there).
→ Multi-account: delegate Macie administration to one account to see findings across the entire organization's S3 data.
EXAM TIP"Find PII stored in S3 buckets" → Amazon Macie. "Detect credit card numbers accidentally uploaded to S3" → Macie. "Identify publicly accessible S3 buckets containing sensitive data" → Macie. Macie is S3-only — it doesn't scan databases or other storage. For scanning RDS for sensitive data, use a third-party tool or Comprehend on exported data.

PAIR
Confusing Pairs core
The most common exam distractors in this module
▶
→ KMS vs CloudHSM: KMS = multi-tenant, AWS manages HSMs, deep AWS service integration, FIPS 140-2 L2. CloudHSM = dedicated single-tenant, you manage keys exclusively, FIPS 140-2 L3, limited native AWS integration.
→ SSM Parameter Store vs Secrets Manager: SSM = free (Standard), no auto-rotation, config + secrets. Secrets Manager = $0.40/secret/month, automatic rotation for RDS/Aurora/Redshift. Choose by: need rotation? → Secrets Manager. Cost-sensitive? → SSM.
→ WAF vs Shield: WAF = Layer 7 application firewall (SQL injection, XSS, rate limiting, geo-blocking). Shield = DDoS volumetric protection (Layer 3/4 with Standard, Layer 7 with Advanced). They are complementary — both can be used together.
→ GuardDuty vs Inspector vs Macie: GuardDuty = behavioral threat detection (runtime anomalies, compromised accounts). Inspector = vulnerability scanning (CVEs in software). Macie = sensitive data discovery (PII in S3). Three different security domains.
→ WAF vs Firewall Manager: WAF = the actual rule engine (Web ACL). Firewall Manager = the deployment mechanism that applies WAF rules (and other policies) across all accounts in an organization. Firewall Manager manages WAF at scale.
→ AWS Managed Key vs Customer Managed Key: AWS Managed = free, auto-rotates, cannot customize policy, cannot share cross-account. CMK = $1/month, full control, shareable. For cross-account encryption → must use CMK.
