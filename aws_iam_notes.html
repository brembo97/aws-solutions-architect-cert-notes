Module 1 — Foundations
Users, Groups & Policies
The core building blocks of AWS access control. Everything in IAM starts here.
IAM
What is IAM? core
Identity and Access Management — the AWS access control service
▶

IAM is a global AWS service (not region-specific) that controls authentication (who you are) and authorization (what you can do) across your AWS account. It is completely free to use — you are only charged for the AWS resources accessed through it.

→ The root account is created automatically when you set up AWS. It has unrestricted access to everything and should never be used for daily operations — protect it with MFA and lock it away.
→ IAM operates globally — a user, group, role, or policy created in one region is accessible everywhere.
→ IAM applies the principle of least privilege: only grant what is explicitly needed, nothing more.
EXAM TIPYou will be asked whether IAM is global or regional — it is always global. Also remember: by default, a new IAM user has zero permissions (implicit deny).
USR
IAM Users core
Individual identities for people or applications
▶

An IAM user is an entity that represents a person or service that interacts with AWS. Each user has a unique name within the account, and can have credentials: a password for console access and/or access keys for programmatic access.

→ Users start with no permissions — permissions must be explicitly granted via policies.
→ Users can belong to multiple groups and inherit permissions from all of them.
→ Users can also have inline policies attached directly to them (not recommended — use group policies instead).
→ AWS recommends creating one IAM user per person, not shared credentials.
IMPORTANTDo not use the root account for day-to-day tasks. Create an admin IAM user instead. The root account should only be used for initial setup and account-level actions (like changing billing settings or closing the account).
GRP
IAM Groups core
Collections of users that share permissions
▶

Groups let you apply the same set of permissions to multiple users at once. Instead of attaching policies to every user individually, you assign them to a group and add users to it.

→ Groups cannot contain other groups — they can only contain users.
→ A user can belong to multiple groups simultaneously — their effective permissions are the union of all attached policies.
→ Groups are a best practice for managing permissions at scale. Examples: Developers, Admins, ReadOnlyUsers.
→ Users don't have to belong to a group, but it's recommended.
EXAM TIPA common exam trap: groups cannot contain other groups. Also, a user not in any group can still have direct (inline) policies attached.
POL
IAM Policies core
JSON documents defining permissions
▶

Policies are JSON documents that specify what actions are allowed or denied on which AWS resources. They are attached to users, groups, or roles. AWS also provides hundreds of pre-built AWS Managed Policies.

→ Version: always "2012-10-17" (the current policy language version).
→ Statement: array of permission rules. Each has: Effect (Allow/Deny), Action (e.g. s3:GetObject), Resource (ARN of the target), and optionally Condition.
→ Managed policies are created/maintained by AWS or by you (customer managed) and can be reused across multiple identities.
→ Inline policies are embedded directly into a single user/group/role and are deleted with it. Avoid for maintainability.
→ Wildcard * means "all" — "Action": "*" with "Resource": "*" is full admin access.
EXAM TIPAWS evaluates all applicable policies and the result is: if any statement says Deny → denied. Otherwise, if any says Allow → allowed. If neither → implicit deny.
SGN
Simultaneous Sign-in core
Managing multiple AWS accounts at once
▶

You can be logged into multiple AWS accounts at the same time by using different browsers or a private/incognito window for each. This is useful when switching between environments like development and production.

→ Each browser session maintains an independent AWS session — no interference between accounts.
→ The account ID or alias is visible in the top-right of the AWS console, helping distinguish which account you're in.
Module 2 — Security
Security & MFA
Protecting accounts with multi-factor authentication, auditing tools, and best practices.
MFA
Multi-Factor Authentication core
Second layer of protection beyond passwords
▶

MFA requires a user to present two forms of identity: something they know (password) and something they have (a device). Even if a password is compromised, an attacker cannot access the account without the second factor.

MFA TypeExamplesNotes
Virtual MFA appGoogle Authenticator, Authy, DuoGenerates time-based one-time codes (TOTP). Software on a smartphone. Supports multiple accounts per device.
Universal 2nd Factor (U2F)YubiKeyPhysical hardware key plugged into USB. Supports multiple root/IAM users on one key.
Hardware OTP tokenGemalto, SurePassIDPhysical device that generates a numeric code. Used for governments/enterprises requiring dedicated hardware.
EXAM TIPMFA should always be enabled on the root account. If asked which MFA type works for multiple accounts on one device: Virtual MFA apps. If asked which is a physical key: YubiKey (U2F).
CR
IAM Credentials Report tool
Account-level security audit
▶

The Credentials Report is an account-wide CSV file listing all IAM users and the status of their various credentials. Generated from the IAM console under "Credential Report".

→ Shows: password last used, password last changed, MFA active status, access key age, access key last used.
→ Useful for compliance audits and identifying stale/unused credentials at the account level.
→ Scope is account-wide — covers all users.
AA
IAM Access Advisor tool
Per-user permission usage analysis
▶

Access Advisor is available within an individual user's IAM page. It shows the list of services the user has permission to access, and when those services were last accessed.

→ Helps enforce least privilege by identifying permissions that are granted but never used.
→ Scope is per user (as opposed to the Credentials Report which is account-wide).
→ Data has a delay of up to 4 hours before reflecting recent activity.
EXAM TIPCredentials Report = account-level, all users, credential status. Access Advisor = per user, service access history. Know which tool answers which question.
BP
IAM Best Practices core
AWS-recommended security guidelines
▶
→ Never use the root account except for initial account setup and billing/account changes.
→ Create individual IAM users — do not share credentials between people.
→ Assign permissions via groups, not directly to users.
→ Enforce strong password policies and require MFA for all users.
→ Use IAM Roles for applications and AWS services — never embed access keys in code.
→ Rotate access keys regularly and delete unused ones.
→ Audit with the Credentials Report and Access Advisor periodically.
→ Never share access keys — if compromised, deactivate immediately.
Module 3 — Programmatic Access
CLI, SDK & CloudShell
Three ways to interact with AWS beyond the web console.
KEY
Access Keys core
Credentials for programmatic access
▶

Access keys are long-term credentials used for programmatic access (CLI and SDK). Each key pair consists of an Access Key ID (like a username) and a Secret Access Key (like a password).

→ Generated under a user's IAM page → Security credentials tab.
→ The Secret Access Key is shown only once at creation — if lost, you must create a new key pair.
→ Each user can have a maximum of 2 active access keys at a time (allows rotation without downtime).
→ Never embed access keys in code, commit them to git, or share them — use environment variables or the credentials file.
→ Access keys can be deactivated (not just deleted) — useful when rotating.
SECURITY WARNINGExposed access keys are one of the most common AWS security incidents. Always use IAM Roles for EC2/Lambda instead of embedding keys. If a key is compromised, deactivate it immediately.
CLI
AWS CLI tool
Command-line access to AWS APIs
▶

The AWS CLI (Command Line Interface) is an open-source tool that lets you interact with AWS services directly from your terminal. It calls the same underlying APIs as the web console — anything you can do in the console, you can do with the CLI.

→ Available for Windows, macOS, and Linux. Configured with aws configure (prompts for Access Key, Secret, default region, output format).
→ Configuration is stored in ~/.aws/credentials and ~/.aws/config.
→ Supports multiple named profiles: aws configure --profile myprofile.
→ Useful for automation, scripting, and CI/CD pipelines.
SDK
AWS SDK tool
Language-specific AWS API libraries
▶

The SDK (Software Development Kit) lets you call AWS APIs from within application code. AWS provides SDKs for Python (boto3), JavaScript/Node.js, Java, .NET, Ruby, Go, PHP, C++, and more.

→ Credentials are read from environment variables, the ~/.aws/credentials file, or an IAM Role attached to the compute resource — in that order of precedence.
→ The most popular SDK is boto3 (Python). The AWS CLI itself is built on boto3.
→ Never hardcode credentials in application code — use environment variables or IAM Roles.
EXAM TIPIf asked the best way for an EC2 instance to access S3 — use an IAM Role attached to the EC2 instance, not access keys. Roles provide temporary, automatically rotating credentials.
CS
AWS CloudShell tool
Browser-based, pre-authenticated terminal
▶

CloudShell is a browser-based shell available directly in the AWS console. It automatically uses the credentials of the currently logged-in IAM user — no configuration needed.

→ Comes with AWS CLI, Python, Node.js, and other tools pre-installed.
→ Provides 1 GB of persistent storage per region — files you create are saved between sessions.
→ Not available in all AWS regions — check the region availability page before relying on it.
→ No SSH keys or credentials needed — great for quick testing or learning.
Module 4 — Delegation
IAM Roles for AWS Services
Granting permissions to AWS services so they can act on your behalf.
ROL
What are IAM Roles? core
Temporary identities for services and cross-account access
▶

An IAM Role is an identity with permission policies, but it is not tied to a specific person. Instead, it is assumed by trusted entities — AWS services, other AWS accounts, or identity providers (federated users).

→ Roles provide temporary security credentials (via AWS STS), not long-term access keys.
→ Common use cases: EC2 Instance Role (app on EC2 accesses S3), Lambda Execution Role, ECS Task Role, CloudFormation Role.
→ A role has two parts: a trust policy (who can assume it) and a permissions policy (what they can do once they assume it).
→ When you assume a role, you temporarily give up your original permissions and gain only the role's permissions.
EXAM TIPThe exam often asks: how should an EC2 instance access S3 securely? Answer: attach an IAM Role to the EC2 instance. Never use access keys embedded in the instance.
Module 5 — Advanced IAM
Advanced Policies & Evaluation Logic
Deep dive into how AWS evaluates permissions across multiple policy types.
EV
Policy Evaluation Logic advanced
How AWS decides Allow or Deny
▶

When a principal (user, role, service) makes an API call, AWS evaluates all applicable policies in a specific order. The result is always either Allow or Deny.

1. Explicit Deny always wins — overrides any Allow from any policy.
2. Organizations SCPs — if the account is in an Org and an SCP restricts an action, it is denied even if IAM allows it. Management account is never affected by SCPs.
3. Resource-based policies — e.g. S3 bucket policy. If a resource policy explicitly allows a principal, access may be granted even without an identity-based allow (cross-account).
4. Permissions boundaries — set the maximum permissions a user/role can have, even if identity policies grant more.
5. Session policies — applied when using STS AssumeRole. Further limit the role's permissions for that session.
6. Identity-based policies — the standard policies attached to users, groups, roles. Default is implicit deny if no allow found anywhere.
EXAM TIPExplicit Deny ALWAYS wins. If no explicit Allow is found anywhere in the chain, the result is implicit Deny. These are two different denies — explicit (a policy says Deny) vs implicit (silence = deny).
RB
Resource-based Policies vs IAM Roles advanced
Two ways to grant cross-account access
▶

There are two patterns for allowing a principal in Account A to access a resource in Account B.

Resource-based PolicyIAM Role
Attached toThe resource itself (S3, SQS, Lambda, etc.)A role in the target account

EXAM TIPKey difference: with a resource-based policy you keep your original permissions. With a role assumption, you temporarily trade your permissions for the role's. This matters for actions that require permissions from both accounts simultaneously (e.g., replication across accounts).
AP
Advanced Policy Conditions advanced
Fine-grained control using context keys
▶

Policy conditions add extra requirements for an Allow or Deny to take effect. They use condition operators and context keys from the request context.

→ aws:RequestedRegion — restrict which regions a user can operate in.
→ aws:PrincipalOrgID — allow access only to principals within your Organization.
→ aws:MultiFactorAuthPresent — require MFA to be active for sensitive actions (e.g., deleting S3 buckets).
→ aws:SourceIp / aws:VpcSourceIp — restrict access to specific IP ranges or VPC endpoints.
→ aws:CalledVia — allow only when the call is made through a specific AWS service (e.g., CloudFormation).
→ Permissions Boundary: a managed policy attached to a user/role that sets the maximum permissions. Even if identity policies grant more, the effective permissions are the intersection.
EXAM TIPPermissions boundary = ceiling on permissions. If a user has AdministratorAccess policy but a permissions boundary allowing only S3, the effective permissions are only S3.
Module 6 — Multi-Account
AWS Organizations
Centrally manage, govern, and bill multiple AWS accounts.
ORG
Organizations Overview enterprise
Account hierarchy and consolidated billing
▶

AWS Organizations lets you group multiple AWS accounts into a managed hierarchy. You get centralized billing, volume discounts, and tools to enforce policies across all accounts.

→ Management account (formerly master): the root account that creates the org. Has full control. Is never affected by SCPs.
→ Member accounts: all other accounts in the org. Can be organized into Organizational Units (OUs).
→ OUs are containers for accounts. SCPs applied to an OU cascade to all accounts and nested OUs within it.
→ Consolidated billing: one payment method for all accounts. Volume discounts (EC2, S3) are shared across the org.
→ Reserved Instance and Savings Plan benefits are shared across accounts in the org.
→ Offers two feature sets: Consolidated Billing only, or All Features (includes SCPs).
SCP
Service Control Policies (SCPs) enterprise
Permission guardrails across accounts
▶

SCPs are a type of Organizations policy applied at the root, OU, or account level. They define the maximum permissions available to any principal in an account — they act as guardrails, not grants.

→ SCPs do not grant permissions — they only restrict. A principal must still have an IAM policy that allows the action AND the SCP must not deny it.
→ SCPs apply to all principals in the account including the account's own root user (but not the Organization's management account).
→ An SCP of Deny ec2:* at the OU level prevents all EC2 actions in all accounts in that OU, regardless of their IAM policies.
→ SCPs are evaluated before IAM policies in the permission evaluation chain.
EXAM TIPManagement account is NEVER restricted by SCPs. SCPs are guardrails — they narrow the permission space but don't grant access on their own. Both SCP + IAM policy must allow an action for it to succeed.
TAG
Tag Policies enterprise
Enforce consistent tagging across the organization
▶

Tag Policies are a type of Organizations policy that standardize the tags applied to resources across all accounts in the org. They define which tags are required, their allowed values, and which resource types they apply to.

→ Enforce consistent naming conventions (e.g. Environment must be one of prod, dev, staging).
→ Help with cost allocation reporting, compliance, and resource management.
→ Non-compliant resources are flagged — tag policies report violations, they do not prevent resource creation by default.
Module 7 — Enterprise Identity
AWS IAM Identity Center
Single Sign-On (SSO) for all AWS accounts and business applications.
SSO
IAM Identity Center enterprise
Formerly AWS SSO — one login for everything
▶

IAM Identity Center (successor to AWS SSO) provides a single point of login to access all AWS accounts in an Organization, as well as SAML 2.0-integrated business applications like Salesforce, Microsoft 365, Slack, and more.

→ Built-in identity store: Identity Center has its own directory — you can create users and groups directly. No external IdP required.
→ External IdP integration: Connect to Active Directory (via AWS Directory Service), Okta, Azure AD, or any SAML 2.0 / SCIM provider. Users authenticate with their corporate credentials.
→ Permission sets: IAM policies bundled together and assigned to users/groups for specific AWS accounts. One permission set can be deployed across multiple accounts.
→ Users get a personalized portal showing all their assigned accounts and applications. One click to switch between them.
→ Integrated with AWS Organizations — must be enabled at the management account level.
EXAM TIPWhen asked about centralizing access to multiple AWS accounts or integrating with an external identity provider (Okta, Azure AD), the answer is IAM Identity Center. Permission sets = the mechanism for assigning permissions per account.
Module 8 — Directory Integration
AWS Directory Services
Connecting Windows/Active Directory workloads with AWS.
AD
Three Directory Service Flavors enterprise
AWS Managed AD vs AD Connector vs Simple AD
▶
ServiceDescriptionOn-premises AD needed?
AWS Managed Microsoft AD Full Microsoft Active Directory hosted by AWS in the cloud. Can establish a trust relationship with your on-premises AD. Supports MFA. Users and groups exist both on-prem and in the cloud independently. Optional (trust relationship)
AD Connector A directory gateway — a proxy that redirects authentication requests to your existing on-premises AD. No data is cached in AWS. All auth goes through the on-prem AD. Yes (required)

EXAM TIPKey differentiators: AD Connector = just a proxy, on-prem AD required, no caching. AWS Managed AD = real AD in cloud, works independently, can trust on-prem. Simple AD = basic, standalone, no on-prem support, no trust relationships.
Module 9 — Governance Automation
AWS Control Tower
Automated setup and governance of a secure, multi-account AWS environment.
CT
AWS Control Tower enterprise
Landing Zone — automated multi-account setup
▶

Control Tower automates the setup of a secure, well-architected multi-account AWS environment called a Landing Zone. It sits on top of AWS Organizations and automates the baseline configuration that would otherwise take days to set up manually.

→ Landing Zone: the pre-configured multi-account environment. Includes a management account, a log archive account, and an audit account out of the box.
→ Guardrails: automated rules applied across the org. Two types:
• Preventive guardrails — implemented as SCPs. They prevent non-compliant actions (e.g., prevent disabling CloudTrail).
• Detective guardrails — implemented as AWS Config rules. They detect and alert on non-compliant configurations (e.g., detect if S3 buckets are public).
→ Account Factory: a self-service portal (or automated pipeline) for provisioning new AWS accounts that are pre-configured to meet organizational standards.
→ Built on top of: AWS Organizations (account structure), AWS Service Catalog (Account Factory), AWS Config (detective guardrails), SCPs (preventive guardrails), CloudTrail (centralized logging).
→ Provides a dashboard showing compliance status of all accounts and guardrails.
EXAM TIPControl Tower = automated Landing Zone setup. Preventive guardrails = SCPs. Detective guardrails = AWS Config rules. Account Factory = automated account provisioning. It orchestrates Organizations, Config, and Service Catalog together.
