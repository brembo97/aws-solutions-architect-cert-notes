Module 1 — Foundations
VPC, CIDR & Subnets
The building blocks: IP addressing, Virtual Private Clouds, subnets, and the default VPC that every account starts with.
CID
CIDR & IP Addressing core
How IP ranges work in AWS — private vs public, CIDR notation, and reserved addresses
▶

CIDR (Classless Inter-Domain Routing) notation defines a block of IP addresses. Understanding CIDR is essential because every VPC, subnet, Security Group rule, and NACL rule uses CIDR blocks.

→ Format: 10.0.0.0/16 — the /16 is the prefix length (how many bits are the network part). The remaining bits are for hosts. /16 = 65,536 IPs. /24 = 256 IPs. /28 = 16 IPs (minimum allowed in AWS). /32 = 1 IP (single host).
→ Private IP ranges (RFC 1918): 10.0.0.0/8 (10.0.0.0 – 10.255.255.255), 172.16.0.0/12 (172.16.0.0 – 172.31.255.255), 192.168.0.0/16 (192.168.0.0 – 192.168.255.255). VPC CIDRs must fall within these ranges.
→ VPC CIDR range: minimum /28 (16 IPs), maximum /16 (65,536 IPs). You can add up to 5 secondary CIDRs to a VPC after creation.
→ 5 reserved IPs per subnet: AWS reserves the first 4 and last 1 address in every subnet. Example in 10.0.0.0/24: .0 = network, .1 = VPC router, .2 = DNS, .3 = future use, .255 = broadcast. A /24 subnet has 251 usable IPs, not 256.
EXAM TIPQuick CIDR math: /16 = 65,536 IPs, /24 = 256 IPs, /28 = 16 IPs (minus 5 reserved = 11 usable). The exam often asks "how many usable IPs in a /24 subnet?" → 251 (not 256). CIDRs must not overlap between VPCs you want to peer. 0.0.0.0/0 = all IPs (used in route tables and SG rules for "anywhere").
VPC
VPC Overview core
Your own isolated virtual network inside AWS — the foundation of all networking
▶

A VPC (Virtual Private Cloud) is a logically isolated section of the AWS cloud where you define your own network — IP ranges, subnets, route tables, gateways, and security rules. Every resource you launch (EC2, RDS, Lambda in VPC) lives inside a VPC.

→ VPCs are regional — a VPC exists in one AWS region and spans all AZs in that region. Subnets are AZ-specific.
→ Max 5 VPCs per region (soft limit, can be increased). Each VPC has a primary CIDR block + up to 4 secondary CIDRs.
→ VPCs have DNS resolution (enabled by default) and DNS hostnames (disabled by default in custom VPCs). Enable DNS hostnames to get public DNS names for EC2 instances with public IPs.
→ Default VPC: every account has one per region. CIDR 172.31.0.0/16. Has an Internet Gateway, a default route table routing 0.0.0.0/0 to the IGW, and public subnets in every AZ (instances get public IPs by default). Do not delete it — some AWS services require it.
→ Custom VPC: you create it. No IGW by default. No public IPs by default. You control everything — CIDR, subnets, route tables, gateways.
EXAM TIPDefault VPC = public subnets, IGW attached, instances get public IPs. Custom VPC = private by default, no IGW, no public IPs. "Instance launched in a VPC has no internet access" → check: is there an IGW? Is the route table routing 0.0.0.0/0 to it? Does the instance have a public IP?
SUB
Subnets core
Sub-ranges of VPC CIDR, each in one AZ — public vs private
▶

A subnet is a range of IP addresses within a VPC, confined to a single Availability Zone. Subnets are where you actually place resources. Whether a subnet is "public" or "private" depends on its route table — not a property of the subnet itself.

→ Public subnet: has a route table entry sending 0.0.0.0/0 to an Internet Gateway. Instances with public IPs in this subnet can reach and be reached from the internet.
→ Private subnet: no route to an IGW. Instances cannot be reached from the internet directly. Outbound internet access (if needed) goes through a NAT Gateway in a public subnet.
→ Each subnet is associated with exactly one route table and exactly one NACL. If not explicitly associated, it uses the VPC's main route table and default NACL.
→ A subnet can only be in one AZ — it cannot span multiple AZs. Best practice: create at least 2 subnets per tier (public + private) across 2+ AZs for high availability.
→ Auto-assign public IPv4: a subnet-level setting. When enabled, instances launched in this subnet automatically get a public IPv4. Typically enabled for public subnets, disabled for private subnets.
EXAM TIP"What makes a subnet public?" → a route to an IGW (0.0.0.0/0 → IGW) in its route table, NOT a label. A subnet with no internet route is private regardless of its name. Remember: 5 IPs reserved per subnet (first 4 + last 1).

Module 2 — Internet Access
IGW, Route Tables, Bastion Hosts & NAT
How traffic reaches the internet — and how private instances access the internet without being directly reachable.
IGW
Internet Gateway & Route Tables core
The gateway between your VPC and the public internet
▶

An Internet Gateway (IGW) is a horizontally scaled, redundant, highly available VPC component that allows communication between your VPC and the internet. It performs NAT for instances with public IPv4 addresses.

→ One IGW per VPC. An IGW cannot be attached to multiple VPCs. A VPC can have only one IGW.
→ For internet access you need: (1) an IGW attached to the VPC, (2) a route table entry 0.0.0.0/0 → IGW on the subnet, (3) a public IPv4 or Elastic IP on the instance, (4) Security Group allowing the traffic.
→ Route Tables: each subnet is associated with one route table. Route tables contain rules (routes) that determine where network traffic is directed. Most specific route wins (longest prefix match). The local route (VPC CIDR → local) enables all subnets within the VPC to communicate — it cannot be removed.
→ Best practice: create separate route tables for public subnets (routes to IGW) and private subnets (routes to NAT Gateway or no internet route). Do not use the main route table for public routing.
EXAM TIP"EC2 has a public IP but cannot reach the internet" → check 4 things: (1) IGW attached? (2) Route table has 0.0.0.0/0 → IGW? (3) SG allows outbound? (4) NACL allows traffic? Missing any one = no internet. The local route is automatic and ensures intra-VPC communication.
BST
Bastion Hosts core
Jump box for SSH/RDP access to instances in private subnets
▶

A Bastion Host (jump box) is an EC2 instance in a public subnet that you SSH into, and from there SSH into instances in private subnets. It is the only entry point for administrative access to your private infrastructure.

→ Bastion SG: allow inbound SSH (port 22) only from your corporate IP or specific CIDR range — never 0.0.0.0/0 in production.
→ Private instance SG: allow inbound SSH only from the Bastion's Security Group or private IP — not from the internet.
→ Modern alternative: AWS Systems Manager Session Manager — provides shell access to private instances without needing a bastion host, SSH keys, or port 22 open. Runs via SSM Agent + IAM role. No public IP needed on the target. Preferred for new architectures.
EXAM TIP"SSH into private instances without a bastion" → SSM Session Manager (no port 22, no bastion, no public IP). "Traditional jump box" → Bastion in public subnet. The exam increasingly favors Session Manager over bastion hosts for security reasons.
NAT
NAT Gateway vs NAT Instance core
Allow private instances to reach the internet for updates — without being reachable from the internet
▶

NAT (Network Address Translation) lets instances in private subnets initiate outbound internet connections (software updates, API calls) while preventing inbound connections from the internet. NAT Gateway is the managed solution; NAT Instance is the legacy self-managed approach.

→ NAT Gateway is AZ-specific. For HA: deploy one NAT Gateway per AZ, each in a public subnet of that AZ, with the corresponding private subnet routing through its own AZ's NAT GW.
→ Regional NAT Gateway: AWS now supports a single NAT Gateway that can serve multiple AZs — but for maximum HA, one per AZ is still recommended.
EXAM TIP"Private instances need internet for OS updates but must not be reachable from the internet" → NAT Gateway in public subnet. Always prefer NAT Gateway over NAT Instance. NAT Instance requires disabling Source/Dest Check — this is a classic exam detail. For HA: one NAT Gateway per AZ.

Module 3 — Network Security
NACLs & Security Groups
Two firewall layers — stateful vs stateless, allow vs deny, instance-level vs subnet-level. The most-tested networking comparison.
CMP
Security Groups vs NACLs core
The #1 most-tested VPC comparison on every AWS exam
▶
Security GroupNetwork ACL (NACL)
Operates atInstance level (ENI)Subnet level
Stateful / StatelessStateful — if inbound is allowed, outbound response is automatically allowedStateless — must explicitly allow both inbound AND outbound traffic separately
Rule typesAllow rules ONLY — no deny rulesAllow AND Deny rules
Rule evaluationAll rules evaluated together (union of all allows)Rules evaluated in numbered order — first match wins, then stops evaluating
Default behaviorAll inbound denied, all outbound allowedDefault NACL: all traffic allowed in/out. Custom NACL: all traffic denied in/out.

→ Ephemeral ports and NACLs: when a client connects to a server, the server responds on a random high port (1024–65535) called an ephemeral port. Because NACLs are stateless, the outbound NACL must allow ephemeral ports or the response will be blocked. SGs handle this automatically (stateful).
→ NACL rule numbering: lower number = higher priority. Rule 100 is evaluated before Rule 200. The * (asterisk) rule is the implicit deny-all at the end — matches only if no numbered rule matched. Best practice: number rules in increments of 100 (100, 200, 300) to leave room for insertions.
→ Blocking a specific IP: you CANNOT block a specific IP with a Security Group (no deny rules). Use a NACL deny rule to block traffic from a specific IP address or CIDR range.
EXAM TIP"Block a specific IP address" → NACL (deny rule). Security Groups cannot deny. "Connection timeout" → SG is blocking (no response received). "Connection refused" → app is rejecting (traffic reached the instance). NACLs = stateless (must allow ephemeral ports for return traffic). This comparison appears on EVERY AWS exam.
CLASSIC TRAPDefault NACL allows ALL traffic. Custom NACL denies ALL traffic. If you create a new NACL and associate it with a subnet, all traffic will be blocked until you add allow rules.

Module 4 — VPC Connectivity
VPC Peering & VPC Endpoints
Connect VPCs together and access AWS services privately — without going over the internet.
PER
VPC Peering core
Private network connection between two VPCs — same or different accounts/regions
▶

VPC Peering creates a private network connection between two VPCs, allowing instances in both to communicate using private IP addresses as if they were in the same network. Traffic stays on the AWS backbone — never traverses the public internet.

→ Non-transitive: if VPC A peers with VPC B, and VPC B peers with VPC C, A and C can NOT communicate through B. You must create a separate peering connection between A and C.
→ CIDRs must NOT overlap: if VPC A uses 10.0.0.0/16 and VPC B also uses 10.0.0.0/16, they cannot peer. Plan CIDRs carefully across your organization.
→ Works cross-region and cross-account. The owner of the accepter VPC must accept the peering request.
→ Route table update required: after creating the peering connection, you must add a route in each VPC's route table pointing the other VPC's CIDR to the peering connection ID. Security Groups in the peer VPC can be referenced cross-account.
→ VPC peering scales well for a small number of VPCs. For many VPCs (10+), consider Transit Gateway instead.
EXAM TIPVPC Peering is NOT transitive — this is the #1 peering exam question. Overlapping CIDRs = cannot peer. "10 VPCs that all need to communicate" → Transit Gateway (not 45 peering connections). Always update route tables on BOTH sides after creating a peering connection.

EPT
VPC Endpoints core
Access AWS services privately from your VPC — no IGW, NAT, VPN, or internet needed
▶

VPC Endpoints allow you to privately connect your VPC to supported AWS services without leaving the AWS network. Traffic between your VPC and the service does not traverse the internet — it stays on the AWS backbone.

→ Gateway Endpoints are preferred for S3 and DynamoDB — free, simple, and fast. Interface Endpoints are required for everything else.
→ PrivateLink: the underlying technology behind Interface Endpoints. Also allows you to expose your own service from one VPC to another — the consumer VPC creates an Interface Endpoint to your Network Load Balancer (NLB) without VPC peering.
→ S3 Interface Endpoint: you CAN also create an Interface Endpoint for S3 (for use cases requiring a private IP in the subnet, like on-premises access via Direct Connect). But for standard VPC → S3 traffic, Gateway Endpoint is preferred (free).
EXAM TIP"Lambda in VPC needs to call S3 without internet" → S3 Gateway Endpoint (free) OR NAT Gateway (costs money). Gateway Endpoint is preferred. "Access SQS privately from a VPC" → Interface Endpoint (SQS doesn't support Gateway). "Expose your service to another VPC without peering or internet" → PrivateLink (NLB + Interface Endpoint).

Module 5 — Hybrid Connectivity
Site-to-Site VPN & Direct Connect
Connect your on-premises network to AWS — encrypted VPN over the internet or dedicated private fiber.
VPN
Site-to-Site VPN core
Encrypted IPsec tunnel over the public internet — quick to set up, variable performance
▶

Site-to-Site VPN creates an encrypted IPsec tunnel between your on-premises network and your AWS VPC. Traffic travels over the public internet but is encrypted end-to-end. Fast to provision (minutes), low cost, but subject to internet performance variability.

→ AWS side: Virtual Private Gateway (VGW) — a gateway attached to your VPC that terminates the VPN connection. Alternatively, attach to a Transit Gateway instead.
→ On-premises side: Customer Gateway (CGW) — represents your on-premises VPN device (router or software VPN appliance). You provide the public IP of your device to AWS.
→ Each VPN connection has two IPsec tunnels for redundancy — each terminates at a different AWS endpoint in different AZs. If one tunnel fails, traffic fails over to the other.
→ Route propagation: VGW can automatically propagate on-premises routes to VPC route tables via BGP. Or you can use static routes if your on-premises device doesn't support BGP.
→ Max throughput: ~1.25 Gbps per VPN tunnel. For higher throughput, use multiple VPN connections with ECMP (Equal Cost Multi-Path) on a Transit Gateway — not on a VGW.
→ Setup time: minutes to hours. Cost: ~$0.05/hr per VPN connection + data transfer.
EXAM TIP"Quick, encrypted connection to AWS from on-premises" → Site-to-Site VPN. "Backup path when Direct Connect fails" → VPN over internet. VGW = AWS side. CGW = on-premises side. Two tunnels per connection for HA. Max 1.25 Gbps per tunnel. ECMP for higher throughput requires Transit Gateway.
DX
AWS Direct Connect (DX) core
Dedicated private fiber connection — consistent bandwidth, low latency, no internet
▶

Direct Connect provides a dedicated, private network connection from your on-premises data centre to AWS. Traffic does NOT traverse the public internet — it uses dedicated fiber through an AWS Direct Connect Partner facility. Higher bandwidth, lower latency, and consistent performance compared to VPN.

→ Connection speeds: Dedicated connections: 1 Gbps, 10 Gbps, or 100 Gbps. Hosted connections (via partners): 50 Mbps to 10 Gbps.
→ Virtual Interfaces (VIFs): Public VIF — access AWS public services (S3, DynamoDB) over DX. Private VIF — access VPC resources over DX. Transit VIF — access VPCs via a Transit Gateway over DX.
→ Direct Connect Gateway: a global resource that connects one DX connection to multiple VPCs across different regions. Without it, a DX connection reaches only VPCs in the same region. With DX Gateway: one physical connection → access VPCs in any region.
→ Setup time: 1+ months. A physical fiber cable must be installed from your data centre to the nearest DX location. This is NOT quick like VPN.
→ Not encrypted by default. Traffic is private (dedicated fiber) but not encrypted. For encryption over DX: establish a Site-to-Site VPN over the DX connection (IPsec on top of DX). This gives you dedicated + encrypted.
→ DX is NOT inherently HA: a single DX connection is a single point of failure. For HA: (1) two DX connections at different DX locations, or (2) one DX + Site-to-Site VPN as backup.
→ DX SiteLink: send data between two DX locations via the AWS backbone without going through a VPC — useful for connecting two on-premises data centres via AWS.
EXAM TIP"Consistent, high-bandwidth, low-latency private connection to AWS" → Direct Connect. "DX takes 1+ month to provision" — use VPN as interim. "DX is not encrypted" → add VPN over DX for encryption. "Access VPCs in multiple regions from one DX" → DX Gateway. "HA for hybrid connectivity" → two DX or DX + VPN backup. Single DX is a single point of failure — always pair it.

Module 6 — Hub Networking
AWS Transit Gateway
Regional hub for interconnecting VPCs, VPNs, and Direct Connect — replaces complex peering meshes.
TGW
Transit Gateway core
Hub-and-spoke network — connect thousands of VPCs and on-premises through a single gateway
▶

AWS Transit Gateway acts as a regional network hub. You attach VPCs, VPN connections, and Direct Connect gateways to a single Transit Gateway. All attached networks can communicate through the hub — replacing the need for complex VPC peering meshes.

→ Transitive routing: unlike VPC peering, Transit Gateway supports transitive connectivity. VPC A ↔ TGW ↔ VPC B = A and B can communicate via the TGW without a direct connection between them.
→ Attachments: VPCs, Site-to-Site VPN connections, Direct Connect Gateways, other Transit Gateways (for inter-region peering), Connect (SD-WAN/third-party appliances).
→ Route tables: TGW has its own route tables (separate from VPC route tables). You can create multiple TGW route tables to segment traffic — e.g. isolate production VPCs from dev VPCs even though they share the same TGW.
→ Inter-region peering: Transit Gateways in different regions can be peered. Traffic flows over the AWS backbone — not the internet. Useful for global network architectures without complex VPN meshes.
→ ECMP: Transit Gateway supports Equal Cost Multi-Path routing for VPN connections — aggregate multiple VPN tunnels for higher throughput. Not supported on VGW. This is the standard answer for "increase VPN throughput beyond 1.25 Gbps."
→ Multicast: TGW is the only AWS service that supports IP multicast — useful for media streaming, financial market data distribution.
→ Works with AWS RAM (Resource Access Manager): share a Transit Gateway across multiple AWS accounts in an Organization.
EXAM TIP"Connect 10+ VPCs that all need to communicate" → Transit Gateway (not 45 peering connections). "Increase VPN throughput beyond 1.25 Gbps" → multiple VPN connections to TGW with ECMP. "Hub-and-spoke network topology" → Transit Gateway. "IP multicast on AWS" → Transit Gateway. TGW is regional — use inter-region peering for cross-region.
Module 7 — Network Monitoring
VPC Flow Logs & Traffic Mirroring
Capture and analyze network traffic metadata and content for security analysis and troubleshooting.
FLW
VPC Flow Logs core
Capture IP traffic metadata going to/from ENIs, subnets, or entire VPCs
▶

VPC Flow Logs capture information about IP traffic going to and from network interfaces in your VPC. They record metadata (source/dest IP, port, protocol, action, bytes) — NOT packet contents.

→ Three levels: VPC (all ENIs in the VPC), Subnet (all ENIs in the subnet), ENI (specific network interface). More specific levels inherit from broader ones.
→ Log destinations: CloudWatch Logs (query with Logs Insights) or S3 (query with Athena — cheaper for large-scale analysis).
→ Each flow log record includes: ENI ID, source IP, destination IP, source port, dest port, protocol, packets, bytes, start/end time, and action (ACCEPT or REJECT). The action tells you whether the SG/NACL allowed or blocked the traffic.
→ Not captured: DNS traffic to the Amazon DNS server (.2), DHCP traffic, instance metadata (169.254.169.254), NTP traffic, Windows license activation.
→ Troubleshooting pattern: if you see REJECT on inbound → SG or NACL is blocking the request. If inbound shows ACCEPT but outbound shows REJECT → NACL is blocking the response (NACLs are stateless, SGs are stateful so SGs wouldn't block the response).
EXAM TIP"Inbound ACCEPT + Outbound REJECT" → NACL is blocking the response (stateless). If it were a SG issue, you'd see inbound REJECT (stateful — response auto-allowed). "Analyze network traffic patterns at scale" → Flow Logs to S3 → Athena SQL queries. Flow Logs do NOT capture packet content — for deep packet inspection use Traffic Mirroring.
MIR
VPC Traffic Mirroring advanced
Copy actual network packets for deep inspection — IDS/IPS, forensics, compliance
▶

Traffic Mirroring copies actual network packets (not just metadata) from ENIs and sends them to an appliance or NLB target for deep packet inspection, content analysis, or forensics. It is the AWS equivalent of a network TAP or SPAN port.

→ Source: ENI (network interface of EC2, RDS, etc.). Target: another ENI or an NLB (behind which sits IDS/IPS appliances).
→ Captures full packet content using VXLAN encapsulation. Can filter which traffic to mirror by protocol, port, or direction.
→ Use cases: network security monitoring (IDS/IPS), regulatory content inspection, troubleshooting application-level issues, compliance with packet capture requirements.
EXAM TIPFlow Logs = metadata (IPs, ports, accept/reject). Traffic Mirroring = actual packet content (deep inspection). "Capture packet content for IDS analysis" → Traffic Mirroring. "See which IPs are being blocked by SGs" → Flow Logs.
Module 8 — IPv6
IPv6 & Egress-Only Internet Gateway
Dual-stack VPCs and the IPv6-specific gateway for outbound-only internet access.
IP6
IPv6 in VPC advanced
Dual-stack mode — every IPv6 address in AWS is public, no NAT needed
▶

AWS VPCs support dual-stack networking — IPv4 and IPv6 running simultaneously. Every resource can have both an IPv4 private address and an IPv6 address. Unlike IPv4, IPv6 addresses in AWS are all globally unique and publicly routable — there are no "private" IPv6 ranges in VPC.

→ All IPv6 addresses are public (globally routable). There is no concept of "private IPv6" in AWS VPCs. Security is enforced entirely through SGs and NACLs, not by network isolation.
→ VPC IPv6 CIDR is assigned by AWS (Amazon-provided /56 block) or you bring your own (BYOIP). Subnets get a /64 block from the VPC's /56.
→ IPv4 cannot be disabled — the VPC always has IPv4. IPv6 is added alongside. This is called dual-stack mode.
→ For IPv6 internet access: route ::/0 (all IPv6) to the IGW in the subnet's route table. SGs must also allow IPv6 traffic.
→ "EC2 can't connect to IPv4 internet" → check that the subnet's route table has 0.0.0.0/0 → IGW (separate from ::/0 → IGW). IPv4 and IPv6 routes are independent entries.
EXAM TIP"IPv6 addresses in AWS VPC are all public" — this is counterintuitive but true. No NAT for IPv6 (no private addresses to translate). Security relies 100% on SGs + NACLs. IPv4 cannot be disabled — dual-stack is mandatory.
EGW
Egress-Only Internet Gateway advanced
NAT Gateway equivalent for IPv6 — outbound only, blocks all inbound
▶

An Egress-Only Internet Gateway allows IPv6 instances to initiate outbound connections to the internet while preventing inbound connections from the internet. It is the IPv6 equivalent of a NAT Gateway (which only works for IPv4).

→ Used for IPv6 only. NAT Gateway is for IPv4 only. They solve the same problem for different protocols.
→ Route: ::/0 → eigw-xxx in the private subnet's route table. Instance can call external APIs over IPv6 but cannot be reached from the internet.
→ Stateful: like a NAT Gateway, it remembers outbound connections and allows the corresponding inbound responses.
EXAM TIP"Private subnet with IPv6 instances needs outbound internet access only" → Egress-Only Internet Gateway. "Private subnet with IPv4 instances needs outbound internet" → NAT Gateway. They are distinct — one for each protocol. A NAT Gateway does NOT work for IPv6; an Egress-Only IGW does NOT work for IPv4.
Module 9 — Advanced Security
AWS Network Firewall
Managed, stateful network firewall for VPCs — deep packet inspection, IDS/IPS, domain filtering.
NFW
AWS Network Firewall advanced
Layer 3–7 managed firewall — goes far beyond SGs and NACLs
▶

AWS Network Firewall is a managed, stateful firewall service that provides fine-grained control over network traffic at the VPC level. It supports deep packet inspection, intrusion detection/prevention (IDS/IPS), and domain-based filtering — capabilities that Security Groups and NACLs cannot provide.

→ Operates at Layer 3 through Layer 7. Inspects IP headers, TCP/UDP headers, AND application-layer content (HTTP, TLS SNI, DNS domain names).
→ Rule types: (1) Stateless rules (like NACLs — fast, no connection tracking). (2) Stateful rules (like SGs — understands connection state, supports Suricata-compatible IDS/IPS rules). (3) Domain list rules (allow/deny traffic to specific FQDNs — e.g. allow *.amazonaws.com, deny everything else).
→ Deployment: Network Firewall deploys firewall endpoints in your VPC subnets. Route traffic through the firewall endpoints using VPC route tables — typically all traffic entering or leaving the VPC passes through the firewall first.
→ Integration: managed via AWS Firewall Manager for organization-wide policies. Logs to CloudWatch, S3, or Kinesis Firehose. Works with Gateway Load Balancer for third-party appliance scaling.
→ Use cases: enterprise-grade firewall compliance (PCI, HIPAA), blocking traffic to known malicious domains, IDS/IPS rule enforcement, egress domain filtering (allow only approved external domains).
EXAM TIP"Block traffic to specific domains from a VPC" → Network Firewall (domain list rules). "IDS/IPS within a VPC" → Network Firewall (Suricata rules). "Inspect VPC traffic at Layer 7" → Network Firewall. SGs/NACLs cannot filter by domain name or inspect application-layer content. Network Firewall fills that gap. For third-party appliances in the traffic path → Gateway Load Balancer.

NUM
Key Numbers & Facts core
Hard limits and facts tested directly in exam questions
▶
→ VPC CIDR range: min /28 (16 IPs), max /16 (65,536 IPs). Up to 5 secondary CIDRs.
→ Reserved IPs per subnet: 5 (first 4 + last 1). /24 = 251 usable IPs.
→ Max VPCs per region: 5 (soft limit). Max subnets per VPC: 200.
→ Max SGs per instance: 5. Max rules per SG: 60 inbound + 60 outbound.
→ NACL rules: evaluated in numbered order, first match wins. SG rules: all evaluated together (union).
→ Default NACL: allows all. Custom NACL: denies all.
→ VPC Peering: non-transitive. CIDRs must not overlap.
→ S3/DynamoDB Gateway Endpoint: free. Interface Endpoint: ~$0.01/hr/AZ + $0.01/GB.
→ VPN max throughput: 1.25 Gbps per tunnel. ECMP via TGW for more.
→ Direct Connect setup time: 1+ months. Dedicated: 1/10/100 Gbps. Hosted: 50 Mbps–10 Gbps.
→ Direct Connect: NOT encrypted by default. Add VPN over DX for encryption.
→ NAT Gateway: $0.045/hr + $0.045/GB processed. AZ-specific. Deploy one per AZ for HA.
→ IPv6 in VPC: all addresses are public. IPv4 cannot be disabled (dual-stack only).
→ Flow Logs: metadata only (IPs, ports, action). Traffic Mirroring: full packet content.
→ Ephemeral ports (NACLs): 1024–65535 must be allowed in outbound rules for return traffic.
