Module 1 — Foundations
Why Decouple Applications?
The problem with tightly coupled synchronous systems — and how async messaging solves it.
DCL
Synchronous vs Asynchronous Communication core
Why tight coupling breaks at scale — and the three decoupling patterns
▶

In a synchronous (tightly coupled) architecture, Service A calls Service B and waits for a response. If B is slow or unavailable, A is blocked — and any traffic spike that overwhelms B cascades directly into A. Decoupling adds an intermediary so services operate independently.

PatternServiceModelBest for
QueueSQSOne producer, one consumer (pull model). Messages wait to be processed.Work queues, task offloading, load leveling, decoupling services
Pub/SubSNSOne producer, many consumers (push model). All subscribers receive every message.Event notifications, fan-out, broadcasting to multiple systems
StreamKinesisMany producers, many consumers. Messages retained and replayable. Ordered within shard.Real-time analytics, log ingestion, event sourcing, IoT telemetry
→ Decoupling provides: resilience (producer keeps working if consumer is down), elasticity (scale consumers independently), load leveling (smooth out traffic spikes), and fault isolation (failures don't cascade).
→ All three AWS services (SQS, SNS, Kinesis) are fully managed — no servers to provision, no message brokers to operate.
EXAM TIPQueue (SQS) = point-to-point, consumer deletes messages after processing. Pub/Sub (SNS) = broadcast to all subscribers. Stream (Kinesis) = ordered, replayable, retained. When the exam describes a "traffic spike overwhelming a downstream service" → SQS queue in between is almost always the answer.
Module 2 — SQS
SQS — Standard Queue Core
Amazon's oldest service — fully managed message queuing with at-least-once delivery and best-effort ordering.
SQS
SQS Standard Queue core
Fully managed queue — unlimited throughput, at-least-once delivery
▶

SQS (Simple Queue Service) is AWS's oldest service and the foundational tool for decoupling components. Producers send messages to a queue; consumers poll the queue, process messages, and delete them. SQS is the buffer between services.

→ Unlimited throughput and unlimited number of messages in the queue. Fully serverless — no provisioning required.
→ Default message retention: 4 days. Maximum retention: 14 days. Messages not processed within this window are permanently deleted.
→ Maximum message size: 256 KB. For larger payloads, store the data in S3 and send a reference in the SQS message (use the SQS Extended Client library).
→ At-least-once delivery: a message may be delivered more than once. Your consumer must be idempotent — processing the same message twice should produce the same result.
→ Best-effort ordering: messages are generally delivered in the order they were sent, but this is NOT guaranteed in a Standard queue. Use FIFO queue for strict ordering.
→ Consumers poll the queue (pull model) — SQS doesn't push to consumers. Consumers must actively request messages.
→ Multiple consumers can poll the same queue — each message is delivered to only one consumer (competing consumers pattern). Messages are not broadcast to all consumers (that's SNS).
EXAM TIPSQS is the #1 answer for "decouple two services" or "buffer a traffic spike." Key limits: 256 KB max message, 14 days max retention, at-least-once delivery (idempotent consumers required). Standard queue = unlimited throughput, no ordering guarantee. Need ordering? → FIFO queue.
DLQ
Dead Letter Queue (DLQ) core
Capture messages that repeatedly fail processing — prevent poison pill loops
▶

A Dead Letter Queue is a separate SQS queue that receives messages the main queue has failed to process successfully after a configured number of attempts. Prevents one bad message from blocking the entire queue indefinitely.

→ Configure a maxReceiveCount on the source queue (e.g. 5). If a message is received and not deleted after 5 attempts, SQS moves it to the DLQ automatically.
→ The DLQ must be the same type as the source queue: Standard → Standard DLQ, FIFO → FIFO DLQ.
→ Set a long retention period on the DLQ (up to 14 days) so you have time to debug failed messages before they expire.
→ DLQ Redrive: after fixing the bug, you can replay messages from the DLQ back to the source queue for reprocessing — without re-ingesting them from the producer.
→ Use CloudWatch alarms on ApproximateNumberOfMessagesNotVisible and NumberOfMessagesSent on the DLQ to detect processing failures early.
EXAM TIP"Messages keep failing and blocking the queue" → configure a DLQ with maxReceiveCount. "Replay failed messages after fixing a bug" → DLQ Redrive. maxReceiveCount is the maximum number of times a message can be received before being moved to DLQ.
Module 3 — SQS
SQS Advanced Features
Visibility timeout, long polling, delay queues, and message attributes.
VIS
Visibility Timeout core
How SQS prevents double-processing — the invisible window
▶

When a consumer receives a message from SQS, the message becomes invisible to all other consumers for the visibility timeout period. This prevents two consumers from processing the same message simultaneously. The consumer must delete the message before the timeout expires, or the message becomes visible again.

→ Default visibility timeout: 30 seconds. Range: 0 seconds to 12 hours. Set at the queue level; can be overridden per message via ChangeMessageVisibility API.
→ If the consumer processes the message successfully → calls DeleteMessage → message permanently removed. If the consumer crashes or times out → visibility timeout expires → message becomes visible again → another consumer picks it up (at-least-once delivery).
→ If visibility timeout is too short: message becomes visible before processing completes → processed twice → duplicates. If too long: a failed message is hidden for too long → delays retry → longer recovery time.
→ For long processing jobs, consumers should call ChangeMessageVisibility to extend the timeout before it expires — like a heartbeat to SQS.
EXAM TIP"Messages are being processed twice" → visibility timeout is shorter than processing time. Increase it. "Message stuck invisible for too long after consumer crash" → visibility timeout is too long. The sweet spot = slightly longer than your maximum expected processing time.
POL
Long Polling core
Reduce empty responses and API costs by waiting for messages to arrive
▶

By default, SQS uses short polling — the consumer sends a request, SQS checks a subset of its servers immediately, and returns any available messages (or an empty response if none exist). Long polling keeps the connection open until a message arrives or the wait time expires.

→ Long polling: consumer waits up to 20 seconds for a message to arrive. If a message arrives during the wait → returned immediately. If not → empty response after 20s.
→ Long polling reduces the number of API calls (fewer empty responses) → lower costs. It also reduces latency slightly — message is returned as soon as it arrives rather than waiting for the next poll cycle.
→ Enable via: setting WaitTimeSeconds to 1–20 on the ReceiveMessage API call, or enabling it at the queue level (queue attribute ReceiveMessageWaitTimeSeconds).
→ Long polling is almost always preferable to short polling — lower cost, lower latency, same functionality.
EXAM TIP"Reduce SQS API costs" or "reduce empty poll responses" → enable long polling. Long polling waits up to 20 seconds. Short polling returns immediately even if the queue is empty. Long polling is preferred in almost all scenarios.
EXT
Extended Client & Other Features advanced
Large messages via S3, delay queues, and message attributes
▶
→ SQS Extended Client Library: for messages larger than 256 KB, the producer stores the payload in S3 and sends a pointer in the SQS message. The consumer retrieves the pointer, downloads the payload from S3, and processes it. Messages can be up to 2 GB using this approach.
→ Delay Queues: delay delivery of new messages by 0–15 minutes. Messages are invisible to consumers for the delay period. Useful for scheduling retries or introducing processing lag.
→ Message Attributes: key-value metadata attached to a message (up to 10 attributes per message). Consumers can filter or route based on attributes without parsing the body.
→ Purge Queue: delete all messages in a queue instantly. Use with caution — cannot be undone.
EXAM TIP"Send a message larger than 256 KB through SQS" → SQS Extended Client Library (payload goes to S3, pointer goes to SQS). Delay queue = new message delay, up to 15 minutes. Visibility timeout = per-message processing window. Different concepts!
Module 4 — SQS
SQS FIFO & Auto Scaling
Strict ordering, exactly-once processing, deduplication — and scaling EC2 consumers based on queue depth.
FIFO
SQS FIFO Queue core
First-In-First-Out, exactly-once processing, strict ordering
▶

FIFO (First-In-First-Out) queues guarantee that messages are processed in exactly the order they were sent, and that each message is processed exactly once. This comes at the cost of lower throughput compared to Standard queues.

Standard QueueFIFO Queue
OrderingBest-effort (not guaranteed)Strict FIFO — guaranteed
DeliveryAt-least-once (duplicates possible)Exactly-once (deduplication built-in)
ThroughputUnlimited300 msg/sec (3,000 with batching)
Name suffixAny nameMust end in .fifo
DLQ supportYes (Standard DLQ)Yes (FIFO DLQ)
→ Message Deduplication: FIFO queues deduplicate within a 5-minute window. Two methods: Content-based deduplication (SHA-256 hash of body) or a Deduplication ID you provide. If the same ID arrives within 5 minutes, the second message is discarded.
→ Message Groups: messages with the same MessageGroupId are processed in order, one at a time. Different group IDs can be processed in parallel — enabling partitioned ordering within the same queue.
→ FIFO queues must have a name ending in .fifo — this is enforced by SQS.
→ Use cases: financial transactions (order matters), inventory management, user command processing, any workflow where sequence is business-critical.
EXAM TIP"Messages must be processed in order" → FIFO queue. "Prevent duplicate processing" → FIFO queue with deduplication. "Throughput > 3,000 msg/sec with ordering" → not possible with FIFO — must architect differently. FIFO queue name MUST end in .fifo.
ASG
SQS + Auto Scaling Group core
Scale EC2 consumers based on queue depth — the canonical decoupling pattern
▶

Combining SQS with an ASG creates the most powerful decoupling pattern: the queue absorbs bursts of messages, and the ASG scales EC2 consumer instances up/down based on how many messages are waiting. Perfect for variable workloads.

→ The ASG scales based on a CloudWatch metric: ApproximateNumberOfMessagesVisible — the number of messages in the queue waiting to be processed.
→ Use a Target Tracking policy targeting a queue depth per instance (e.g. "keep ~100 messages per consumer"). ASG adds instances when depth rises and removes them when it falls.
→ Architecture: Producer → SQS → ASG (EC2 consumers). The queue is the buffer — producers never need to know about consumer capacity.
→ Common use case: image/video processing pipeline. Upload triggers → message in SQS → EC2 fleet scales up → processes jobs → scales down when queue empties.
EXAM TIP"Front-end spikes overloading the backend" → add SQS between them + ASG scaling on queue depth. "Scale processing workers based on workload" → SQS + ASG with ApproximateNumberOfMessagesVisible metric. This is one of the most common architectural patterns on the exam.
Module 5 — SNS
SNS & Fan-Out Pattern
Simple Notification Service — publish once, deliver to many subscribers simultaneously.
SNS
Amazon SNS Overview core
Pub/Sub messaging — one message, many subscribers, push delivery
▶

SNS (Simple Notification Service) implements the publish/subscribe messaging pattern. A producer publishes a message to an SNS Topic, and all subscribers to that topic receive the message simultaneously — pushed to them automatically.

→ Up to 12,500,000 subscriptions per topic. Up to 100,000 topics per account (soft limit).
→ Supported subscriber types (SNS pushes to all of them): SQS, Lambda, HTTP/HTTPS endpoints, Email / Email-JSON, SMS, Mobile Push (iOS, Android), Kinesis Data Firehose.
→ SNS is a push model — unlike SQS where consumers poll. SNS delivers to all subscribers as soon as the message is published.
→ Maximum message size: 256 KB (same as SQS).
→ SNS does not persist messages — if a subscriber is unavailable, the message is lost for that subscriber (unless it's an SQS subscriber with its own retention). No replay capability.
→ Message filtering: subscribers can apply a filter policy (JSON) so they only receive messages matching specific attributes. A subscriber without a filter policy receives all messages.
EXAM TIPSNS = broadcast to all subscribers (push). SQS = one consumer per message (pull). "Send the same event to multiple services simultaneously" → SNS. "Send an email + trigger a Lambda + write to SQS with one event" → publish to SNS with three subscribers. SNS does not persist messages — no replay.
FAN
SNS + SQS Fan-Out Pattern core
Combine SNS broadcast with SQS persistence — the canonical pub/sub + queue pattern
▶

Fan-Out combines SNS with multiple SQS queues as subscribers. One SNS message fans out to all SQS queues simultaneously. Each queue is consumed independently by a different service — giving you SNS broadcast + SQS durability + independent scaling.

→ Why SNS → SQS instead of direct SNS → service: SQS adds persistence (message retained up to 14 days), retry logic, and allows the consuming service to scale independently at its own pace.
→ Classic architecture: Producer → SNS Topic → [SQS Queue A (email service), SQS Queue B (analytics), SQS Queue C (fraud detection)]
→ Adding a new consumer is as simple as subscribing a new SQS queue to the topic — no changes to the producer.
→ S3 event fan-out limitation: an S3 bucket can only trigger one SNS topic per event type. To fan-out to multiple services, have S3 notify SNS → SNS fans out to multiple SQS queues.
→ SNS FIFO + SQS FIFO: for ordered fan-out with deduplication, SNS FIFO topics deliver to SQS FIFO queues in order. SNS FIFO has the same 300 msg/sec limit as SQS FIFO.
EXAM TIP"S3 event notification needs to trigger multiple downstream services" → S3 → SNS → multiple SQS queues (fan-out). "Add a new consumer without changing existing producers" → add SQS queue subscription to existing SNS topic. Fan-out = SNS + SQS — the most commonly tested integration pattern involving both services.
Module 6 — Kinesis
Kinesis Data Streams
Real-time data streaming — ordered, replayable, multi-consumer event ingestion at massive scale.
KDS
Kinesis Data Streams Overview core
Sharded real-time stream — ordered within shards, replayable for 1–365 days
▶

Kinesis Data Streams is a real-time data streaming service designed to ingest and process large volumes of data (logs, events, IoT telemetry, clickstreams) with low latency. Unlike SQS, data is retained for replay and multiple consumers can read the same stream independently.

→ A stream is made up of shards — the fundamental unit of capacity. Each shard provides: 1 MB/s write (1,000 records/s), 2 MB/s read per consumer.
→ Default data retention: 24 hours. Can be extended up to 365 days (additional cost). Data can be replayed — multiple consumers can read the same data at different positions.
→ Records are ordered within a shard. Records with the same partition key always go to the same shard — maintaining ordering for related events.
→ Max record size: 1 MB. Records contain: partition key, sequence number, and data blob.
→ Two capacity modes: Provisioned (manually set shard count, pay per shard-hour) and On-Demand (auto-scales, pay per GB throughput — no shard management needed).
→ Consumers: standard (2MB/s per shard shared across all consumers — polling) or Enhanced Fan-Out (2MB/s per shard per consumer — push via HTTP/2, dedicated throughput per consumer).
EXAM TIPKDS vs SQS: KDS = ordered within shards, replayable, many consumers read same data, producers/consumers manage position (sequence number). SQS = unordered (Standard), message deleted after processing, one consumer gets each message. "Multiple applications need to process the same event stream" → KDS (or SNS fan-out). "Real-time analytics dashboard" → KDS.
SCL
Shard Scaling & Capacity Planning advanced
Resharding, hot shards, and the On-Demand vs Provisioned choice
▶
→ Shard splitting: divide one shard into two — increases throughput. Use when a shard is at capacity (ProvisionedThroughputExceededException).
→ Shard merging: combine two shards into one — reduces cost when traffic is low.
→ Hot shard problem: if all records share the same partition key, all go to one shard → that shard is overwhelmed while others are idle. Use highly random or distributed partition keys (e.g. user ID, device ID) to spread load evenly.
→ On-Demand mode: auto-scales shards based on observed peak throughput from the past 30 days. No shard management — best for unpredictable workloads. Higher per-GB cost.
→ Provisioned mode: you manually specify and pay for shards. Best for predictable high-throughput workloads where you can plan capacity in advance. Lower per-shard cost at scale.
EXAM TIP"Getting ProvisionedThroughputExceededException" → increase shard count (shard splitting) or switch to On-Demand. "All records going to one shard" → hot shard problem, use more distributed partition keys. On-Demand = hands-off, higher cost. Provisioned = manual, cheaper at scale.
Module 7 — Kinesis
Amazon Data Firehose
Fully managed ETL streaming delivery — load data to S3, Redshift, OpenSearch, and Splunk with zero code.
FH
Amazon Data Firehose core
Near-real-time delivery to storage and analytics destinations — fully serverless
▶

Amazon Data Firehose (formerly Kinesis Data Firehose) is a fully managed service for loading streaming data into destinations like S3, Redshift, Amazon OpenSearch, Splunk, and custom HTTP endpoints. It handles buffering, batching, compression, and encryption automatically — no consumer code to write.

→ Destinations: Amazon S3, Amazon Redshift (via S3 intermediate), Amazon OpenSearch Service, Splunk, Datadog, MongoDB, any custom HTTP endpoint.
→ Firehose is near-real-time, not real-time — it buffers data before delivery. Buffer size: 1–128 MB. Buffer interval: 60–900 seconds. Data is delivered when either threshold is hit first.
→ Fully serverless — no shards to manage, no consumer code. Scales automatically. You pay per GB of data processed.
→ Supports optional data transformation via Lambda before delivery — convert formats, filter records, enrich data.
→ Supports compression (GZIP, Snappy, ZIP) and encryption at rest before writing to S3.
→ No data storage or replay — Firehose delivers and forgets. If you need replay, use Kinesis Data Streams (which can feed Firehose).
→ Data sources: can receive data directly from producers, from Kinesis Data Streams, CloudWatch Logs, CloudWatch Events, IoT, or Kinesis Agent.
EXAM TIPFirehose vs KDS: KDS = real-time, ordered, replayable, you manage consumers. Firehose = near-real-time, automatic delivery to destination, no code, no replay. "Load logs into S3 or Redshift automatically" → Firehose. "Multiple applications need to process the same stream in real time" → KDS. They are often chained: KDS (real-time processing) → Firehose (delivery to S3).

Module 8 — Amazon MQ
Amazon MQ
Managed message broker for legacy AMQP/MQTT/STOMP workloads migrating to AWS — not a serverless service.
MQ
Amazon MQ Overview core
Managed ActiveMQ and RabbitMQ — lift-and-shift migration of on-premises message brokers
▶

Amazon MQ is a managed message broker service that runs Apache ActiveMQ or RabbitMQ on AWS. It is designed specifically for organizations migrating existing on-premises messaging applications that use standard broker protocols — without rewriting the application to use SQS or SNS.

→ Supports open standard protocols: AMQP, MQTT, STOMP, OpenWire, and WebSocket. These are the protocols used by ActiveMQ and RabbitMQ.
→ Amazon MQ is NOT serverless — it runs on provisioned EC2-like instances. You choose the instance size and pay per broker-hour. You also manage broker configuration.
→ Does not scale as massively as SQS/SNS — limited by the broker instance size. Not designed for millions of messages per second.
→ Supports both queues (point-to-point) and topics (pub/sub) — same as ActiveMQ/RabbitMQ behavior on-premises.
→ High availability: deploy in Multi-AZ active/standby configuration with automatic failover. Standby takes over if the primary broker fails.
→ Amazon MQ runs inside your VPC — not publicly accessible by default. Accessed via private endpoints from within the VPC or over VPN/Direct Connect.
EXAM TIPThe exam trigger for Amazon MQ is always migration: "existing application uses ActiveMQ/RabbitMQ/AMQP/MQTT on-premises, wants to migrate to AWS without rewriting." → Amazon MQ. If building a new application → use SQS (simpler, serverless, more scalable). Amazon MQ = migration path, not the greenfield choice.
KEY DISTINCTIONAmazon MQ is not serverless and does not scale like SQS. It's a managed version of existing broker software. Choose SQS/SNS for new cloud-native applications. Choose Amazon MQ only when protocol compatibility with existing on-premises brokers is required.
