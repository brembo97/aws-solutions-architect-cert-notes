Module 1 — CloudWatch
Metrics & Alarms
Collect, visualize, and react to performance data from every AWS service and your own applications.
MET
CloudWatch Metrics core
Time-series data points from AWS services and custom sources
▶

CloudWatch Metrics are time-stamped data points that represent measurements of your AWS resources and applications over time. Every AWS service automatically publishes metrics to CloudWatch at no extra charge. You can also publish your own custom metrics from applications or on-premises servers.

→ Metrics are organized by Namespace (logical container, e.g. AWS/EC2, AWS/RDS, MyApp/Payments) and Dimensions (key-value attributes that uniquely identify a metric, e.g. InstanceId=i-1234).
→ Default metric resolution: standard metrics are published in 1-minute granularity (some services publish every 5 minutes). High-Resolution Metrics: custom metrics can be published at 1-second granularity (additional cost).
→ Metric retention: data points <60s resolution: retained for 3 hours. 1-minute: retained for 15 days. 5-minute: retained for 63 days. 1-hour: retained for 15 months. Older data is automatically aggregated to lower resolution.
→ Key EC2 metrics (built-in): CPUUtilization, NetworkIn/Out, DiskReadOps/WriteOps, StatusCheckFailed. These come automatically — no agent needed.
→ EC2 metrics NOT available without the CloudWatch Agent: RAM/memory utilization, disk space used, process-level metrics, swap usage. These are OS-level metrics — AWS doesn't have access to them without an agent.
→ Custom Metrics: publish from application code via the CloudWatch API (PutMetricData). Include custom dimensions (e.g. per user, per transaction type). Standard resolution ($0.30/metric/month) or High Resolution ($0.30/metric/month + higher API cost).
→ Dashboards: create CloudWatch Dashboards with graphs, numbers, alarms, and text widgets. Dashboards can span multiple regions and accounts. Global dashboards don't auto-update region — you must add each region.
EXAM TIP"Monitor EC2 memory utilization" → requires CloudWatch Agent (RAM is NOT a default metric). Default EC2 metrics: CPU, network, disk I/O. "High-Resolution Metrics" = 1-second granularity custom metrics. Metric retention varies — 1-second data only kept 3 hours. Long-term metric storage → export to S3 or use metrics math with longer storage settings.
ALM
CloudWatch Alarms core
Trigger actions when metrics cross thresholds — notification, scaling, recovery
▶

CloudWatch Alarms watch a single metric (or metric math expression) and trigger actions when the metric crosses a defined threshold. Alarms are the primary way to automate responses to operational events in AWS.

→ Three alarm states: OK (metric is within threshold), ALARM (metric has breached the threshold), INSUFFICIENT_DATA (not enough data to evaluate — common on new alarms or during gaps).
→ Alarm actions: SNS notification (email, Lambda, SQS, HTTP webhook), EC2 actions (stop, terminate, reboot, or recover an instance), Auto Scaling actions (scale out/in an ASG), Systems Manager actions.
→ Period: the length of time (in seconds) over which the metric is evaluated. Must be at least as long as the metric's resolution. For 1-minute metrics, minimum period is 60 seconds.
→ Evaluation periods + Datapoints to alarm: configure "M out of N" threshold — e.g. alarm if 3 out of 5 consecutive data points breach the threshold. Reduces false positives from momentary spikes.
→ Composite Alarms: combine multiple alarms using AND/OR logic. Create a high-level "system health" alarm that triggers only when multiple component alarms are in ALARM state simultaneously — reduces alert noise.
→ EC2 Instance Recovery: when an instance fails system status checks, a CloudWatch Alarm action can automatically recover the instance — moves it to new hardware, preserving the private IP, Elastic IP, and EBS volumes.
→ Alarms can also be triggered on Anomaly Detection — CloudWatch uses ML to model the expected range for a metric and fires the alarm when the metric is outside the "normal" band (no fixed threshold needed).
EXAM TIP"Receive a notification when CPU exceeds 80%" → CloudWatch Alarm → SNS. "Auto-recover an EC2 instance after hardware failure" → CloudWatch Alarm → EC2 Recover action. Composite Alarms reduce alert noise by requiring multiple conditions. INSUFFICIENT_DATA ≠ OK — it means the alarm can't evaluate yet.
SYN
CloudWatch Synthetics & Network Monitor advanced
Proactive monitoring — simulate user traffic and test network paths
▶
→ CloudWatch Synthetics: configurable scripts (Node.js or Python) that run on a schedule to simulate user interactions — test API endpoints, website flows, UI interactions. Detect broken links, slow responses, or outages before real users encounter them.
→ Synthetics canaries run every minute (or longer). Results are stored in S3 (screenshots, HAR files). Integrates with CloudWatch Alarms — alert when a canary fails.
→ CloudWatch Network Synthetic Monitor: monitor network performance between your VPC resources and internet destinations. Sends synthetic probe traffic and measures latency, packet loss, and route changes — without deploying your own monitoring infrastructure.
EXAM TIP"Detect API or website outages before users do" → CloudWatch Synthetics (canaries). Synthetics are scheduled test scripts; regular CloudWatch Alarms react to real traffic. Network Monitor = measure network performance to and from your VPC endpoints.
Module 2 — CloudWatch
Logs & Agents
Centralize application and system logs from EC2, Lambda, ECS, on-premises, and every AWS service.
LOG
CloudWatch Logs core
Centralized log storage, search, and streaming for all AWS and custom sources
▶

CloudWatch Logs is the central log aggregation service in AWS. It ingests logs from EC2 (via agent), Lambda (automatically), ECS containers, EKS, API Gateway, RDS, VPC Flow Logs, CloudTrail, Route 53, and custom applications.

→ Log Groups: a container for log streams representing the same application/resource (e.g. /aws/lambda/my-function, /var/log/messages). Retention period is set at the log group level (default: never expire — you pay indefinitely).
→ Log Streams: a sequence of log events from a single source within a log group (e.g. one log stream per EC2 instance, one per Lambda function invocation context).
→ Log retention: configurable from 1 day to 10 years (or never expire). Set at the log group level. Important for cost control — storing logs indefinitely can become expensive.
→ CloudWatch Logs Insights: interactive query language for log analysis. Ad-hoc queries across multiple log groups simultaneously. Visualize results as time series or bar charts. Common queries for Lambda errors, API Gateway 5xx rates, custom application events.
→ Metric Filters: extract metric values from log events using pattern matching. E.g. count occurrences of "ERROR" in application logs → publish as a CloudWatch metric → trigger an alarm. Converts unstructured logs into quantitative metrics.
→ Subscription Filters: stream log events in real-time to Kinesis Data Streams, Kinesis Firehose, or Lambda for real-time processing or archiving. One subscription filter per log group to the same destination.
→ Export to S3: batch export of log data to S3 for long-term archiving or Athena analysis. Export takes up to 12 hours — not real-time. For real-time, use Subscription Filters to Firehose → S3.
→ Live Tail: real-time streaming view of incoming log events in the console — like tail -f for CloudWatch. Filter by log group and pattern. Useful for debugging live issues without querying.
→ Encryption: logs are encrypted at rest by default. Can configure KMS customer-managed key at the log group level for additional compliance requirements.
EXAM TIP"Stream CloudWatch Logs to Elasticsearch/OpenSearch in real-time" → Subscription Filter → Lambda → OpenSearch. "Export logs to S3 for Athena analysis" → Export to S3 (up to 12 hours delay, not real-time). "Count ERROR occurrences and trigger alarm" → Metric Filter → CloudWatch Metric → Alarm. Lambda logs to CloudWatch automatically — no agent needed.
AGT
CloudWatch Agent core
Collect OS-level metrics and logs from EC2 and on-premises servers
▶

The CloudWatch Agent is software you install on EC2 instances (Linux or Windows) and on-premises servers. It enables collection of metrics and logs that are not available through the default AWS hypervisor-level monitoring.

→ Unified CloudWatch Agent (current, recommended): single agent that collects both metrics and logs. Replaces the older separate "CloudWatch Logs Agent" (logs only) and "CloudWatch Monitoring Scripts" (metrics). Install the unified agent for all new deployments.
→ Metrics it enables: RAM/memory utilization, disk space used/free per partition, disk I/O at the file system level, swap usage, process-level metrics, network connections. These cannot be obtained without the agent.
→ Logs it can collect: any log file on the system — application logs (/var/log/app.log), system logs (/var/log/syslog), Windows Event Logs. Each file maps to a CloudWatch log stream.
→ Configuration is stored in SSM Parameter Store — centrally manage agent configuration across all instances. Push config changes without SSH-ing into each instance.
→ The agent needs an IAM role attached to the EC2 instance with permissions: CloudWatchAgentServerPolicy (publish metrics + send logs) and optionally AmazonSSMReadOnlyAccess (read config from SSM Parameter Store).
→ Works on: Amazon Linux 2/2023, Ubuntu, CentOS, RHEL, Windows Server. Also works on on-premises servers — configure with IAM credentials.
EXAM TIP"Monitor EC2 memory/RAM" → must install CloudWatch Agent. This is the #1 CloudWatch exam trap — memory is NOT a default metric. "Collect custom application log files from EC2" → CloudWatch Agent. "Centrally configure agents across a fleet" → store agent config in SSM Parameter Store.
Module 3 — CloudWatch
Insights & Operational Visibility
Container Insights, Lambda Insights, Contributor Insights, and Application Insights — operational observability at scale.
INS
CloudWatch Insights Services advanced
Four specialized observability tools for containers, Lambda, contributors, and applications
▶

CloudWatch offers four "Insights" features that provide deeper operational visibility beyond raw metrics and logs. Each is purpose-built for a specific observability use case.

FeatureWhat it monitorsKey capability
Container InsightsECS, EKS, Kubernetes on EC2, FargateCollects metrics and logs from containers — CPU, memory, disk, network per task/pod/service. Aggregates performance data at the cluster, node, pod, and container level. Requires CloudWatch Agent (as a DaemonSet on EKS/K8s or built into Fargate). Visualized in pre-built dashboards.
Lambda InsightsLambda functionsDetailed performance metrics for Lambda — cold starts, duration, memory usage, concurrent executions, errors, and throttles. Provided as a Lambda Layer. Goes beyond the basic Lambda metrics in CloudWatch.
Contributor InsightsCloudWatch LogsAnalyze log data to identify the top N contributors to a metric — e.g. top 10 source IPs generating the most traffic, top URLs returning 404s, top users causing errors. Runs continuously (not ad-hoc). Useful for identifying noisy neighbors or bad actors.
Application InsightsApplication stacks (.NET, SQL Server, IIS, Java, Linux)Automatically discover application dependencies, configure monitoring, surface operational insights and anomalies. Sets up recommended metrics and alarms for your application stack automatically. Reduces time to diagnose application issues.
→ CloudWatch Logs Insights (separate from the four above): interactive SQL-like query language for ad-hoc log analysis. Query multiple log groups in a single request. Results can be exported or visualized.
EXAM TIPContainer Insights = containers (ECS/EKS). Lambda Insights = Lambda function performance. Contributor Insights = top-N log analysis (who's causing the most errors/traffic). Application Insights = automated monitoring for .NET/Java apps. Each has a very specific use case — the exam describes the scenario and you pick the right Insights tool.
Module 4 — EventBridge
Amazon EventBridge
Serverless event bus — route events from AWS services, SaaS apps, and custom sources to any target.
EVB
EventBridge Overview core
Formerly CloudWatch Events — event-driven routing at scale with filtering and transformation
▶

Amazon EventBridge (formerly CloudWatch Events) is a serverless event bus that makes it easy to connect applications using events from AWS services, integrated SaaS applications, and your own custom applications. Events are matched against rules and routed to targets.

→ Event buses: three types — Default bus (receives events from all AWS services automatically), Partner bus (receives events from SaaS partners — Zendesk, Datadog, Salesforce, GitHub, etc.), Custom bus (events from your own applications).
→ Rules: match incoming events against a pattern (JSON filter) and route matching events to one or more targets. A rule can have up to 5 targets.
→ Event pattern matching: filter on any field in the event JSON — source, detail-type, account, region, specific field values. Supports prefix, suffix, wildcard, numeric range, and IP address matching. Only matching events invoke the rule.
→ Targets: Lambda, SQS, SNS, Kinesis Data Streams, Step Functions, ECS tasks, CodeBuild, CodePipeline, API Gateway, Batch, another EventBridge bus (cross-account/region), and more.
→ Input transformation: transform and reshape event data before sending to a target — extract specific fields, add constants, rename keys. No Lambda needed for simple transformations.
→ Scheduled rules: trigger targets on a schedule using cron (cron(0 12 * * ? *)) or rate expression (rate(5 minutes)). Replaces Lambda scheduled events and CloudWatch Events scheduled rules. EventBridge Scheduler is the dedicated scheduling service.
→ Cross-account event routing: send events from one AWS account to an EventBridge bus in another account. Configure resource-based policies on the target bus to allow inbound events.
→ Schema Registry: EventBridge can infer and store the schema of events. Schemas can be used to generate code bindings (TypeScript, Python, Java, Go) for strongly-typed event handling.
→ Event archiving and replay: archive all or filtered events to an archive indefinitely. Replay archived events to a bus (useful for re-processing events after a bug fix or to test new rules).
EXAM TIP"Trigger a Lambda when an EC2 instance is stopped" → EventBridge rule on EC2 state change event. "Run a Lambda every hour" → EventBridge scheduled rule (rate expression). "React to SaaS app events (Zendesk ticket created)" → EventBridge Partner Event Bus. "Fan out one event to multiple targets" → EventBridge rule with multiple targets. EventBridge replaced CloudWatch Events — they are the same service, new name.
SCH
EventBridge Scheduler advanced
Dedicated scheduling service — one-time or recurring invocations at scale
▶

EventBridge Scheduler is a standalone scheduling service (separate from EventBridge rules) for creating, running, and managing scheduled invocations of Lambda, SQS, Step Functions, and other AWS targets.

→ Supports three schedule types: Rate-based (every N minutes/hours/days), Cron-based (specific times, timezone-aware), and One-time (run exactly once at a specific datetime).
→ Scales to millions of schedules. Each schedule has its own IAM role for invoking the target. Supports retry policies and dead-letter queues for failed invocations.
→ Use over EventBridge rules when: you need one-time schedules, timezone-aware cron, or massive scale of individual schedules (e.g. per-user scheduled tasks).
Module 5 — Audit
AWS CloudTrail
API call history for your AWS account — who did what, from where, and when. Enabled by default.
CT
CloudTrail Overview core
Full API audit trail — every AWS API call recorded with user identity, time, source IP, and parameters
▶

AWS CloudTrail records every API call made in your AWS account — through the Console, CLI, SDKs, or other services. It answers the question "Who did what in my AWS account?" and is the foundation of AWS compliance, forensics, and security auditing.

→ Enabled by default for all AWS accounts. The last 90 days of management events are available free in the CloudTrail Event History — no trail setup required. Beyond 90 days or for data events → create a Trail.
→ Event types: Management Events (control plane operations — CreateBucket, RunInstances, AttachRolePolicy — on by default in trails), Data Events (data plane operations — S3 GetObject/PutObject, Lambda invocations — NOT logged by default, high volume, additional cost), Insight Events (unusual API call patterns detected by CloudTrail Insights).
→ Trails: configuration to persistently log events. Can be single-region or multi-region. Multi-region trail applies to all regions in the account from one configuration — recommended for comprehensive auditing. Logs delivered to S3 (JSON format, up to 15 minutes delivery lag).
→ Log integrity validation: CloudTrail uses SHA-256 hashing and digital signatures to verify that log files have not been tampered with. Enable on the trail configuration — produces a digest file for each hour.
→ CloudTrail Lake: managed, immutable data lake for CloudTrail events. Query with SQL using Athena-like syntax directly in CloudTrail. Retention up to 7 years. No need to manage S3 + Athena separately.
→ Organization Trail: a trail that applies across all accounts in an AWS Organization — created from the management account. Centralize all account audit logs into one S3 bucket.
→ Encryption: CloudTrail logs in S3 can be encrypted with SSE-S3 (default) or SSE-KMS (for additional key management control and CloudTrail access auditing).
EXAM TIP"Who deleted this S3 bucket?" → CloudTrail (check Management Events). "Who terminated this EC2 instance?" → CloudTrail. "Detect unusual API patterns" → CloudTrail Insights. "Who invoked this Lambda function?" → CloudTrail Data Events (must be enabled — not on by default). Default 90-day history = free, in console only. For S3 persistence → create a Trail. Delivery lag = up to 15 minutes.
SECURITY WARNINGCloudTrail logs themselves must be protected — restrict S3 bucket access, enable MFA Delete, enable log file validation, and consider KMS encryption. Attackers who compromise an account often try to delete CloudTrail logs to cover their tracks.
INT
CloudTrail + EventBridge Integration advanced
React in near-real-time to any API call in your account
▶

CloudTrail sends management events to EventBridge's default event bus within seconds. This enables near-real-time automated responses to API calls — much faster than waiting for logs to arrive in S3 (up to 15 minutes).

→ EventBridge rule pattern: match on source: aws.cloudtrail and specific API calls in the event detail. Example: match on eventName: DeleteBucket to trigger a Lambda that re-creates or alerts on deletion.
→ Use cases: alert when root account is used, notify on IAM policy changes, trigger remediation when security groups are opened to 0.0.0.0/0, alert on high-cost API calls (RunInstances with large instance types).
→ Latency: CloudTrail → EventBridge is near-real-time (seconds). CloudTrail → S3 is eventually consistent (up to 15 minutes). Use EventBridge for real-time response; S3 for long-term archiving and forensics.
EXAM TIP"Alert immediately when someone uses the root account" → CloudTrail sends event to EventBridge → SNS notification. "Automatically remediate when a security group is opened to 0.0.0.0/0" → CloudTrail → EventBridge → Lambda (remove the rule). This pattern = CloudTrail (detect) + EventBridge (route) + Lambda (remediate).
Module 6 — Compliance
AWS Config
Continuous resource configuration assessment — track changes, evaluate compliance, and remediate automatically.
CFG
AWS Config Overview core
Configuration history and compliance evaluation for AWS resources
▶

AWS Config continuously records the configuration of your AWS resources and evaluates them against desired-state rules. It answers "What did my infrastructure look like at a specific point in time?" and "Is my infrastructure compliant with my policies?"

→ Configuration recording: whenever a resource is created, modified, or deleted, Config records the configuration state as a configuration item. The full history of configuration changes is stored in S3. Delivered within minutes of a change.
→ Config Rules: evaluate resources against desired configurations. Two types: AWS Managed Rules (150+ pre-built rules — e.g. s3-bucket-public-read-prohibited, ec2-instance-no-public-ip, iam-password-policy) and Custom Rules (Lambda function you write, evaluated on configuration change or schedule).
→ Compliance evaluation: each rule marks resources as COMPLIANT or NON_COMPLIANT. View compliance on a dashboard, query with Athena (Config data in S3), or integrate with Security Hub.
→ Config does NOT prevent changes — it records and evaluates. It is a compliance monitor, not a firewall. To block non-compliant actions, use IAM policies or Service Control Policies (SCPs).
→ Remediation: Config can trigger automatic remediation via SSM Automation documents when a rule is violated. E.g. automatically revoke public access on S3 buckets that become non-compliant. Remediation actions can be triggered immediately or with a scheduled retry.
→ Configuration snapshots: point-in-time snapshots of all recorded resources. Delivered to S3 on demand or on a schedule. Useful for periodic compliance reports.
→ Config Aggregator: aggregate Config data from multiple accounts and regions into a single account — view organization-wide compliance posture in one dashboard. Works with AWS Organizations.
→ Cost: $0.003 per configuration item recorded. $0.001 per rule evaluation. Config is NOT free — costs accumulate with many resources and rules.
EXAM TIP"Was this S3 bucket publicly accessible 3 weeks ago?" → AWS Config (configuration history). "Alert when a security group allows unrestricted SSH" → Config Rule (securitygroup-restricted-ssh). "Automatically remediate non-compliant resources" → Config Rule + SSM Automation remediation. Config records configuration state; CloudTrail records API calls. Both together give full auditability.
IMPORTANT GOTCHAConfig records configuration changes, not API calls. It answers "what was the configuration?" CloudTrail answers "who changed it and when via API?" For complete forensics: use BOTH. Config = what changed. CloudTrail = who changed it.

Key Numbers to Memorize core
Hard limits and defaults tested directly in exam questions
▶
→ CloudWatch default metric resolution: 1 minute. High-Resolution: 1 second.
→ CloudWatch metric retention: 1s data = 3 hours. 1-min = 15 days. 5-min = 63 days. 1-hour = 15 months.
→ CloudWatch Logs export to S3: up to 12 hours delay (not real-time).
→ EventBridge rule targets: up to 5 targets per rule.
→ CloudTrail default Event History: 90 days, free, management events only.
→ CloudTrail Trail log delivery to S3: up to 15 minutes.
→ CloudTrail Lake retention: up to 7 years.
→ CloudTrail Data Events: NOT logged by default — must be explicitly enabled (additional cost).
→ EC2 metrics NOT available by default (need CloudWatch Agent): Memory/RAM, disk space, swap.
→ AWS Config cost: $0.003 per configuration item recorded (not free).
→ CloudWatch Alarms: INSUFFICIENT_DATA ≠ OK — it means not enough data points to evaluate.
