Module 1 — Core Concepts
RPO, RTO & DR Scope
The two metrics that define every DR strategy, disaster scopes, and the cost vs recovery speed spectrum.
RPO
RPO — Recovery Point Objective core
How much data can you afford to lose? The maximum acceptable data loss window
▶

RPO is the maximum amount of data loss your business can tolerate, expressed as a time window. It answers: "If a disaster occurs right now, how old can the most recent backup be?" Lower RPO = more frequent backups = higher cost.

→ RPO = 1 hour: if a database is backed up every hour, in the worst case you lose up to 60 minutes of transactions. Acceptable for many business applications.
→ RPO = 0: no data loss. Requires synchronous replication — every write must be committed to both primary and standby simultaneously. Very expensive, may impact latency.
→ RPO = 24 hours: acceptable for non-critical systems. Daily backups. Low cost.
→ RPO drives backup frequency, replication lag tolerance, and whether synchronous vs asynchronous replication is required.
EXAM TIPRPO = data loss tolerance (time between last backup and disaster). Higher RPO = more data loss acceptable = cheaper. Lower RPO = less data loss = more expensive (more frequent snapshots, synchronous replication).
RTO
RTO — Recovery Time Objective core
How long can the business be down? The maximum acceptable downtime window
▶

RTO is the maximum amount of time your application can be offline after a disaster before unacceptable business impact occurs. Lower RTO = faster recovery = more pre-provisioned infrastructure = higher cost.

→ RTO = 4 hours: the application can be down up to 4 hours while teams restore from backup. Acceptable for internal tools.
→ RTO = minutes: requires pre-provisioned standby infrastructure that can be activated quickly. DNS failover to a warm standby region.
→ RTO ≈ 0: zero downtime. Active/active multi-region where traffic shifts automatically. Maximum cost.
→ RTO drives infrastructure pre-provisioning, automation of failover, DNS TTL settings, and runbook complexity.
EXAM TIPRPO = data loss tolerance (backward in time). RTO = downtime tolerance (forward in time). They are independent — you can have low RPO but high RTO (frequent snapshots but slow restoration). The exam gives RPO/RTO requirements and asks which strategy fits. Map: high RPO + high RTO → Backup & Restore. Low RPO + Low RTO → Multi-Site.

DR Strategy 1 — Lowest Cost
Backup & Restore
Store backups off-site and restore from scratch on disaster. High RPO and RTO, minimal ongoing cost.
B&R
Backup & Restore Strategy core
RPO: hours. RTO: hours to days. Cost: $. No standby infrastructure — provision from scratch.
▶

In Backup & Restore, you regularly back up data and configuration to a different region or account. On disaster, you provision new infrastructure from scratch and restore data from the backup. There is no standby — all resources are created at recovery time.

→ Lowest cost: pay only for backup storage. No compute running in the DR region until a disaster occurs.
→ Highest RPO and RTO: restoring data and provisioning infrastructure takes hours to days depending on volume and complexity.
→ AWS Backup: centralized backup service for EC2, EBS, RDS, DynamoDB, EFS, S3, FSx. Define backup plans (schedule, retention, lifecycle). Cross-region and cross-account copy rules. Vault Lock = WORM immutable backups even root cannot delete.
→ Infrastructure as Code: store CloudFormation/CDK/Terraform templates in a separate region/account. At recovery time, run the template to re-provision all infrastructure. This is what makes Backup & Restore feasible at scale — no manual rebuilding.
→ S3 CRR: replicate S3 data automatically to DR region. EBS snapshots: copy to DR region. RDS snapshots: copy to DR region or use automated backup cross-region copy. AMI copies: replicate to DR region so EC2 can launch identically configured instances quickly.
→ Glacier: very cheap long-term archive storage. Higher restore time (minutes to hours). Use for compliance archives or rarely-accessed backups.
EXAM TIP"Lowest cost DR — acceptable to be down for several hours" → Backup & Restore. "Immutable backups for compliance" → AWS Backup Vault Lock. "How to quickly re-provision infrastructure after disaster?" → IaC (CloudFormation) stored in another region. Cross-account backups protect against account-level disasters.
DR Strategy 2
Pilot Light
Keep the critical data core running in DR — scale the rest from zero when disaster strikes.
PLT
Pilot Light Strategy core
RPO: minutes. RTO: tens of minutes to hours. Cost: $$. Database replica runs, compute does not.
▶

The "pilot light" analogy: a tiny flame always burns, ready to ignite the full furnace immediately when needed. In DR, you keep the most critical component — the database — continuously running and replicated in the DR region. On disaster, provision the remaining layers (app servers, load balancers) around it.

→ Always running in DR: the database tier — RDS cross-region Read Replica, Aurora Global Database secondary, or DynamoDB Global Table replica. Data is always replicated → very low RPO (seconds to minutes of async lag).
→ NOT running in DR: application servers, web servers, load balancers. These are provisioned on demand from AMIs and CloudFormation when disaster is declared.
→ Failover steps: (1) detect disaster, (2) promote database replica to primary, (3) provision app/web servers from AMIs via CloudFormation or SSM runbook, (4) update Route 53 DNS to DR region.
→ RTO limited by compute provisioning time — typically 10–30 minutes. Faster with pre-configured AMIs and automated runbooks (SSM Automation Documents).
→ Much cheaper than Warm Standby — only the database tier runs continuously. Compute in DR region runs only during the disaster.
EXAM TIPPilot Light = data always replicated, compute not running. "Critical data tier is always replicated, app tier scaled from zero on failover" → Pilot Light. The key distinction from Warm Standby: with Pilot Light you PROVISION new instances (slower); with Warm Standby you SCALE existing instances (faster).
DR Strategy 3
Warm Standby
A fully functional but under-provisioned copy in DR — scale it out on failover, not up from nothing.
WRM
Warm Standby Strategy core
RPO: seconds. RTO: minutes. Cost: $$$. Full stack at minimum capacity — scale on failover.
▶

Warm Standby runs a complete but scaled-down version of the production system in the DR region — all tiers present (web, application, database), but at minimum/reduced capacity. On disaster, scale out the DR environment to full production capacity via Auto Scaling.

→ Always running in DR: the entire stack — ELB, EC2 ASG (minimum 1–2 instances), database replica. Everything is running at reduced scale.
→ Failover steps: (1) detect disaster, (2) promote database replica, (3) scale out EC2 ASG to full production capacity (fast — no provisioning, just scaling), (4) update Route 53 to route all traffic to DR region.
→ RTO measured in minutes — app tier already running at low capacity, scale-out is fast. No instance provisioning required — just increase ASG desired count.
→ Optional dual use: the warm standby can serve a fraction (e.g. 10%) of production traffic continuously — validates it actually works and reduces failover from 0% → 100% to 10% → 100%. Reduces cost by making the standby do useful work.
→ Cost optimization: use smaller instance types in DR (t3.small instead of m5.large). On failover, use a launch template that specifies larger instances or scale out the smaller ones.
EXAM TIPWarm Standby = full stack, scaled down. Pilot Light = data only, no compute. "RTO of minutes, infrastructure already running but not at full scale" → Warm Standby. "RTO near-zero, always active in all regions" → Multi-Site Active/Active.
DR Strategy 4 — Maximum Resilience
Multi-Site Active/Active
Production traffic in multiple regions simultaneously — zero downtime, near-zero data loss, maximum cost.
A/A
Multi-Site Active/Active Strategy core
RPO: near-zero. RTO: near-zero. Cost: $$$$. Full production in multiple regions simultaneously.
▶

Multi-Site Active/Active runs full production workloads in two or more AWS regions simultaneously. Traffic is distributed across regions via Route 53 or Global Accelerator. If one region fails, the others absorb all traffic automatically — no failover procedure, no downtime.

→ Both regions serve live production traffic: no standby. Route 53 Weighted or Latency routing distributes users. Each region is sized to handle 100% of traffic if the other fails.
→ Database — the hardest part: multiple active regions writing to databases requires active-active replication. Options: DynamoDB Global Tables (writes in any region, multi-master, ~1s replication), Aurora Global Database (one primary for writes — technically active/passive for writes but active/active for reads; secondary can be promoted in <1 minute), CockroachDB/Cassandra/Keyspaces for true distributed SQL.
→ Conflict resolution: with multi-master writes (DynamoDB Global Tables), last-writer-wins based on timestamp. Application must be designed to handle potential conflicts or use partition-based routing to avoid same-record conflicts.
→ Automatic failover: Route 53 health checks + failover routing or Global Accelerator health checks automatically stop sending traffic to the failed region. No manual procedure to execute.
→ Cost: full production infrastructure in multiple regions 24/7. Double or triple compute, database, and data transfer costs vs single-region.
→ Use cases: mission-critical platforms ($X million/minute downtime cost), financial trading systems, globally-distributed consumer apps, regulatory zero-data-loss requirements.
EXAM TIP"Zero downtime and zero data loss requirement" → Multi-Site Active/Active. "Writes accepted in any region with no downtime on region failure" → DynamoDB Global Tables. Note: Aurora Global Database is often called "active/active" but writes go to one primary — secondaries serve reads. For true multi-master writes → DynamoDB Global Tables.
Module 5 — AWS Tools
Key AWS Services for DR
The services that appear most in DR architectures — what they do, when to use them.
BAK
AWS Backup core
Centralized managed backup across all AWS services — cross-account and cross-region
▶
→ Supports: EC2, EBS, RDS (all engines), Aurora, DynamoDB, EFS, FSx for Windows/Lustre, S3, Storage Gateway, DocumentDB, Neptune.
→ Backup Plans: define frequency (hourly/daily/weekly/monthly), retention, lifecycle (move to cold storage after N days), and copy rules (copy to another region or account).
→ Backup Vault Lock (WORM): once enabled, backups cannot be deleted by anyone — including root. Required for PCI, HIPAA, and other compliance frameworks that mandate immutable retention.
→ Cross-account backups: copy backups to a separate isolated AWS account for protection against account compromise, ransomware, or admin error in the primary account.
EXAM TIP"Centralized backup policy across all AWS services and accounts" → AWS Backup. "Backups that not even root can delete" → AWS Backup Vault Lock. Cross-account backup = defense-in-depth against account-level disasters.
DRS
AWS Elastic Disaster Recovery (DRS) & MGN advanced
Continuous block-level replication for fast failover of on-premises and cloud workloads
▶
→ AWS Elastic Disaster Recovery (DRS): formerly CloudEndure Disaster Recovery. Installs a lightweight replication agent on source servers (on-premises or other cloud). Continuously replicates block-level changes to low-cost staging EBS volumes in AWS. On disaster, launch recovery EC2 instances in minutes. RPO: seconds. RTO: minutes.
→ Staging area: low-cost EBS volumes in the target AWS region, continuously updated with changes. Full EC2 instances only launch during an actual DR event or test drill — keeping standby costs low.
→ AWS MGN (Application Migration Service): same continuous replication technology but for migration (lift-and-shift to EC2), not ongoing DR. Performs a final cutover when migration is ready.
→ DRS supports: physical servers, VMware, Hyper-V, other clouds (Azure, GCP). Any server that can run the replication agent.
→ DRS supports failback — after on-premises is rebuilt, replicate from AWS back to on-premises and cut over.
EXAM TIPDRS = DR use case (continuous protection, failover on disaster). MGN = migration use case (one-time lift-and-shift). Both use block-level continuous replication agents. "Replicate on-premises servers to AWS for DR with RPO of seconds" → AWS Elastic Disaster Recovery.
R53
Route 53 & Global Accelerator for DR core
DNS-based and anycast-based traffic failover between regions
▶
→ Route 53 Failover routing: primary record points to main region (with health check). Secondary record points to DR region. On primary health check failure, Route 53 returns DR endpoint. Subject to DNS TTL caching — set low TTL (60s) on failover records to minimize switchover time.
→ Route 53 health check timing: checks every 30s (or 10s with fast health check option). Fails after 3 consecutive failures (~90s detection). Total failover time = detection + TTL expiry. For aggressive RTO, use fast health checks + low TTL.
→ Global Accelerator: anycast routing — no DNS TTL delay. Failover is at the network layer, propagates in under 30 seconds regardless of client DNS caching. Better RTO for non-HTTP workloads or when DNS delay is unacceptable.
→ For multi-site active/active: Route 53 Weighted (split traffic by percentage) or Latency (route to nearest region) — both with health checks that automatically remove failed endpoints.
EXAM TIP"Automatic DNS-based region failover" → Route 53 Failover routing + health check. "Sub-minute RTO without DNS delay" → Global Accelerator (anycast, no TTL). Route 53 failover = fast but constrained by DNS TTL. Global Accelerator failover = faster, no DNS, better for strict RTO.

Module 7 — Hybrid DR
On-Premises to AWS DR
Using AWS as the disaster recovery site for on-premises workloads.
HYB
On-Premises → AWS DR Patterns core
Storage, compute replication, and network connectivity for hybrid DR
▶
→ Server replication: AWS Elastic Disaster Recovery (DRS) installs an agent on on-premises servers, continuously replicates block-level data to staging EBS volumes in AWS. On disaster, launch recovery EC2 instances in minutes. Near-zero RPO, minutes RTO.
→ File/object replication: AWS DataSync — scheduled, high-speed sync of on-premises NFS/SMB files to S3/EFS/FSx. Storage Gateway (Volume Gateway, Stored mode) — primary data on-premises, async backup to S3 for snapshot-based recovery.
→ Network resilience: use both Direct Connect (primary — dedicated, fast, low latency) AND Site-to-Site VPN (backup — internet-based, lower cost). VPN activates automatically if Direct Connect fails. A single Direct Connect connection is a single point of failure — always pair with VPN.
→ DNS cutover: on failover, update public DNS to point to AWS endpoints (ALB DNS, CloudFront, Route 53 records). Pre-lower DNS TTLs before planned maintenance windows for faster cutover.
→ Failback: after on-premises is rebuilt, replicate from AWS back to on-premises using DRS failback, DataSync reverse sync, or Storage Gateway. Test failback as part of your DR runbook — it's often overlooked.
EXAM TIP"Replicate on-premises servers to AWS for DR with RPO seconds" → AWS Elastic Disaster Recovery (DRS). "Sync on-premises files to S3 for DR" → DataSync. "Network resilience for hybrid connectivity" → Direct Connect + VPN as failover. Direct Connect alone has NO built-in HA.

KEY
Critical Distinctions & Exam Traps core
The most common exam mistakes on DR
▶
→ Multi-AZ ≠ Multi-Region: Multi-AZ = HA within one region (AZ failure). Multi-Region = DR across regions (region failure). You need both for complete resilience. They are not interchangeable.
→ RDS Multi-AZ ≠ Read Replica: Multi-AZ = synchronous, auto-failover, same region, no reads from standby. Read Replica = asynchronous, manual promotion, cross-region capable, serves reads. Completely different features.
→ Pilot Light vs Warm Standby: Pilot Light = data tier only running. Warm Standby = full stack running at min scale. Warm Standby is faster to recover because app tier is already running (scale vs provision).
→ RPO vs RTO: RPO = data loss (backward — last backup). RTO = downtime (forward — recovery time). Independent: you can have low RPO + high RTO (frequent backups, slow restoration).
→ DNS failover delay: Route 53 failover takes time equal to detection delay + TTL cache expiry. Clients may hit failed region for up to TTL seconds. Set low TTL (60s). For zero-delay → Global Accelerator.
→ DRS vs MGN: DRS = ongoing DR protection + continuous replication + failover when disaster occurs. MGN = one-time migration to EC2 + final cutover. Different purposes, same underlying technology.
→ Direct Connect is NOT HA alone: a single Direct Connect connection is a single point of failure. HA hybrid connectivity = two Direct Connect connections in different locations OR Direct Connect + VPN failover.
→ Aurora Global Database writes: writes go to ONE primary region. Secondaries serve reads and can be promoted. This is technically active/passive for writes — not the same as DynamoDB Global Tables which accept writes in any region (true active/active).
