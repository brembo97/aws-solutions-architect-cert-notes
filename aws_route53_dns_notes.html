Module 1 — Foundations
DNS Fundamentals
How DNS works, the hierarchy of servers, and the vocabulary the exam assumes you know cold.
DNS
What is DNS? core
The internet's phone book — translates domain names to IP addresses
▶

DNS (Domain Name System) translates human-readable domain names (e.g. www.example.com) into machine-routable IP addresses (e.g. 1.2.3.4). Without DNS, every application would need to hardcode IP addresses.

→ Domain hierarchy (right to left): Root (.) → TLD (.com, .org, .io) → Second-level domain (example) → Subdomain (www). Full domain: www.example.com. (trailing dot = root).
→ DNS resolution flow: browser → Local DNS Resolver (ISP/company) → Root Name Server → TLD Name Server → Authoritative Name Server → IP returned → cached at each level by TTL.
→ Root Name Servers: 13 sets of root servers managed by ICANN. They know which name servers are authoritative for each TLD.
→ TLD Name Servers: managed by registries (e.g. Verisign for .com). They know which name servers are authoritative for each second-level domain.
→ Authoritative Name Server: the definitive source for a domain's DNS records. Route 53 is AWS's authoritative DNS service.
→ Local DNS Resolver (Recursive Resolver): caches responses for the TTL duration and handles the recursive lookup on behalf of the client. Run by ISPs or companies (e.g. Google's 8.8.8.8).
EXAM TIPKnow the resolution order: client → recursive resolver → root NS → TLD NS → authoritative NS. Route 53 is the authoritative name server for domains you register or host in it. The recursive resolver does the heavy lifting and caches results.

Module 2 — Route 53
Route 53 Overview
AWS's managed DNS service — authoritative DNS, domain registration, health checks, and routing policies.
R53
Route 53 Overview core
Highly available, globally distributed authoritative DNS + domain registrar
▶

Route 53 is AWS's managed DNS service. It provides authoritative DNS, domain registration, health checking of endpoints, and intelligent routing policies. The name comes from port 53, the standard DNS port.

→ Route 53 is 100% availability SLA — the only AWS service with a full 100% uptime guarantee.
→ Route 53 serves three functions: Domain Registrar (buy/manage domain names), DNS Service (host authoritative DNS records), and Health Checker (monitor endpoints and route away from unhealthy ones).
→ DNS records are organized into Hosted Zones — a container for records belonging to a domain.
→ Public Hosted Zone: answers queries from the internet. Used for publicly accessible resources (websites, APIs).
→ Private Hosted Zone: answers queries only from within associated VPCs. Used for internal services and split-horizon DNS (same domain name resolves differently inside vs outside the VPC).
→ Cost: $0.50/month per hosted zone. No charge for DNS queries to Alias records pointing to AWS resources. Standard DNS query charges apply otherwise.
EXAM TIPRoute 53 = 100% SLA (unique in AWS). Public hosted zone = internet-facing. Private hosted zone = VPC-internal only. You can use Route 53 as just a DNS service even if you registered the domain with GoDaddy — update the NS records at the registrar to point to Route 53's name servers.
3RD
3rd-Party Domains & Route 53 core
Use Route 53 as DNS for a domain registered elsewhere
▶

Route 53 can host the DNS records for a domain registered with any third-party registrar (GoDaddy, Namecheap, Google Domains, etc.). You don't need to transfer the domain — just update the NS records.

→ Process: create a Public Hosted Zone in Route 53 for your domain → Route 53 assigns 4 NS (name server) records → log in to your registrar → update the domain's name servers to Route 53's NS values.
→ DNS resolution now flows through Route 53, even though the domain is registered elsewhere.
→ Alternatively, you can transfer domain registration to Route 53 — this moves both registration and DNS to AWS.
EXAM TIP"Domain registered with GoDaddy, wants to use Route 53 for DNS" → create hosted zone in Route 53 → update NS records at GoDaddy. You do NOT need to transfer the domain to use Route 53 for DNS.
Module 3 — Records
Records, TTL & Alias
How Route 53 records work, TTL caching behavior, and the critical CNAME vs Alias distinction.
TTL
TTL — Time To Live core
How long DNS resolvers cache a record before re-querying
▶

TTL is the number of seconds a DNS resolver will cache a record before discarding it and making a fresh query. It is set on each individual DNS record by the zone owner.

→ High TTL (e.g. 86,400 seconds = 24 hours): fewer DNS queries → lower Route 53 costs → but changes propagate slowly. Use for stable records that rarely change.
→ Low TTL (e.g. 60 seconds): changes propagate quickly → higher Route 53 query costs → more load on DNS infrastructure. Use when you expect to change a record soon (e.g. before a migration).
→ Migration strategy: before changing a record's IP, lower the TTL to 60 seconds → wait for the old high TTL to expire → make the DNS change → all clients will pick up the change within 60 seconds.
→ Clients (browsers, OS) respect TTL and will not re-query until it expires — you cannot force a client to re-query before the TTL expires (except by flushing local DNS cache).
→ Alias records managed by Route 53 have a TTL set by AWS — you cannot configure the TTL on Alias records.
EXAM TIP"Why are users still seeing the old IP after I updated DNS?" → TTL hasn't expired yet — clients are using the cached value. "How to minimize downtime during a DNS migration?" → lower TTL well in advance, then make the change.
ALI
CNAME vs Alias core
The most tested Route 53 distinction — know when to use each
▶

Both CNAME and Alias records map a hostname to another hostname. But Alias is a Route 53 extension of A/AAAA records with important advantages over standard CNAME.

Module 4 — Routing Policies
Simple & Weighted Routing
The two most basic routing policies — single target and proportional traffic distribution.
SMP
Simple Routing Policy core
One record → one or more IPs, no health checks, random selection
▶

The default routing policy. Routes traffic to a single resource. If you specify multiple values in one record, Route 53 returns all values to the client and the client picks one at random.

→ Can specify multiple IP addresses in a single record — Route 53 returns all of them, and the client's resolver picks one (client-side load balancing).
→ Cannot be associated with health checks when multiple values are specified in one record.
→ If using Alias records with Simple routing, you can only specify one AWS resource.
→ Use when: you have a single resource handling a domain and don't need health checks or intelligent routing.
EXAM TIPSimple routing = no health checks, no failover, no weighted traffic split. If the question involves health checks or any intelligent routing → it's not Simple routing. Random client-side selection when multiple IPs are returned.
WGT
Weighted Routing Policy core
Control what % of traffic goes to each resource — A/B testing, blue/green
▶

Weighted routing lets you assign a relative weight to multiple records with the same name. Traffic is distributed proportionally to the weights. Each record points to a different resource.

→ Weights are relative — a weight of 10 and 90 means 10% and 90% of traffic respectively. Weights don't have to add up to 100.
→ If all records are set to weight 0, Route 53 returns all records equally (traffic split evenly).
→ If one record has weight 0 and others have non-zero weights, the zero-weight record receives no traffic.
→ Can be associated with health checks — if a resource is unhealthy, its weight is excluded from the calculation and all traffic goes to healthy records.
→ Use cases: A/B testing, blue/green deployments (gradually shift 100% of traffic from old to new version), canary releases, gradual load shifting during migrations.
EXAM TIP"Send 10% of traffic to a new version for testing" → Weighted routing (weight 10 on new, 90 on old). "Gradually migrate traffic from one region to another" → Weighted routing. Setting all records to weight 0 distributes traffic equally — this is a common trick question.
Module 5 — Routing Policies
Latency & Geolocation Routing
Route users to the fastest region or to a specific region based on their physical location.
LAT
Latency Routing Policy core
Route users to the AWS region with the lowest network latency from their location
▶

Latency routing directs each user to the record with the lowest measured latency from their location to the target AWS region. Route 53 measures latency between thousands of locations and maintains an internal latency database.

→ Each latency record specifies an AWS region as its target. Route 53 picks the region with the lowest latency from the user's measured location.
→ Latency is measured between the user and the AWS region endpoint, not the individual EC2 instance. Different users near the same city may get different results.
→ Can be combined with health checks — if the lowest-latency region is unhealthy, traffic fails over to the next best region.
→ Latency ≠ geographic proximity. A user in Brazil might be routed to us-east-1 if that has lower latency than sa-east-1 at that moment.
EXAM TIP"Route users to the fastest AWS region" → Latency routing. Latency routing is based on measured network performance, NOT physical distance. Do not confuse with Geolocation (location-based) or Geoproximity (distance-based with bias). Latency = performance-optimized.
GEO
Geolocation Routing Policy core
Route based on where the user IS — continent, country, or US state
▶

Geolocation routing routes traffic based on the geographic location of the DNS query's origin — i.e. where the user is physically located. Unlike Latency routing, it is purely location-based, not performance-based.

→ Granularity levels: Continent (e.g. Europe), Country (e.g. Germany), US State (most specific). More specific records take precedence.
→ A Default record is required — it catches traffic from locations not covered by any specific geolocation record. Without a default, unmatched locations get a "no answer" response.
→ Can be combined with health checks — if the target for a location is unhealthy, traffic falls back to the default (not to the best latency option).
→ Use cases: content localization (different languages per region), regulatory compliance (GDPR — keep EU user data in EU), restricting content to specific countries (geo-blocking).
EXAM TIPGeolocation ≠ Latency. Geolocation = "where is the user from" (compliance, content localization). Latency = "which region is fastest for this user" (performance). "EU users must be served from EU data centers for GDPR" → Geolocation. Always create a Default record or unmatched users get no response.
Module 6 — Routing Policies
Failover Routing & Health Checks
Active-passive DNS failover and how Route 53 health checks power intelligent routing.
HC
Route 53 Health Checks core
Monitor endpoints and trigger routing changes when they fail
▶

Route 53 health checks continuously monitor the health of your endpoints. Unhealthy endpoints are automatically removed from DNS responses, preventing traffic from being routed to failed resources.

→ Endpoint health checks: probe a public IP/hostname directly (HTTP, HTTPS, TCP). Threshold: healthy if ≥ 18 of 15 global health checkers report healthy (>18% by default). Interval: 30 seconds (Standard) or 10 seconds (Fast — higher cost).
→ Calculated health checks: combine the results of multiple individual health checks using Boolean logic (AND, OR, NOT). Mark parent as healthy only if N of M children are healthy.
→ CloudWatch Alarm health checks: monitor a CloudWatch alarm state (OK vs ALARM). Allows health checking of private resources (inside a VPC) — Route 53 health checkers cannot reach private IPs directly.
→ Health checkers come from AWS global IP ranges — ensure your firewall/security groups allow inbound traffic from Route 53 health checker IP ranges on the monitored port.
→ Health checks can monitor a string in the first 5,120 bytes of the HTTP response body — useful for application-level health verification.
EXAM TIP"Health check a private EC2 instance" → create a CloudWatch metric/alarm for it → Route 53 CloudWatch Alarm health check. Route 53 health checkers are all public — they cannot reach private IPs. Calculated health checks let you define complex health logic (e.g. "healthy if 2 of 3 regions are up").
FOV
Failover Routing Policy core
Active-passive — automatically switch to a secondary when primary fails
▶

Failover routing implements active-passive DNS failover. Route 53 always routes to the primary record unless it fails its health check, at which point traffic is automatically redirected to the secondary (passive) record.

→ Two records with the same name: one marked Primary, one marked Secondary.
→ The primary record must have an associated health check. Secondary does not require one (but can have one).
→ When primary becomes unhealthy: Route 53 stops returning the primary record and returns the secondary. When primary recovers: Route 53 resumes routing to it.
→ Common pattern: primary = production EC2/ALB in primary region, secondary = S3 static error page or standby in another region.
→ Failover is DNS-based — it only takes effect when the cached DNS record expires (subject to TTL). Not instant like an ELB health check.
EXAM TIPFailover routing = active-passive DNS DR. Primary must have a health check. Secondary is the fallback. "Show a maintenance page if the app goes down" → Failover routing with S3 static page as secondary. Note: DNS failover is not instant — TTL determines how quickly clients switch.
Module 7 — Routing Policies
Geoproximity, IP-Based & Multi-Value
Advanced routing policies for traffic bias control, IP-based segmentation, and lightweight multi-target routing.
GPX
Geoproximity Routing advanced
Route based on geographic distance — with optional bias to expand/shrink regions
▶

Geoproximity routing routes traffic based on the geographic distance between users and your resources. The key differentiator from Geolocation is the bias — you can expand (+) or shrink (-) the effective "pull radius" of a resource to deliberately route more or less traffic to it.

→ Resources can be AWS resources (specify AWS region) or non-AWS resources (specify latitude/longitude).
→ Bias values: range from -99 to +99. Positive bias → expands the effective region (more users routed here). Negative bias → shrinks the effective region (fewer users routed here).
→ Requires Traffic Flow — Route 53's visual traffic policy editor (not available through standard console record creation).
→ Use cases: gradually shift traffic from one region to another by increasing the target region's bias, balancing load across regions with different capacities.
EXAM TIPGeoproximity vs Geolocation: Geoproximity = distance-based with bias control (Traffic Flow required). Geolocation = exact location-based (country/continent), no bias. "Shift traffic from us-east-1 to eu-west-1 gradually using geographic routing" → Geoproximity with increasing bias on eu-west-1.
IP
IP-Based Routing advanced
Route based on the client's IP CIDR block
▶

IP-based routing (also called CIDR-based routing) allows you to define routing rules based on the client's IP address — specifically, CIDR blocks mapped to specific endpoints.

→ You provide a list of CIDR blocks (IP ranges) → each CIDR maps to a specific Route 53 record/endpoint.
→ Use cases: route traffic from your ISP's IP range to a specific endpoint for cost optimization, route corporate office IP ranges to internal resources, route known customer IP ranges to dedicated servers.
→ When you know your users' IPs (e.g. corporate offices, ISPs), IP-based routing provides deterministic routing based on that knowledge — more precise than geolocation.
EXAM TIP"Route users from a specific ISP to a specific endpoint" or "route corporate VPN IP range to internal servers" → IP-based routing. More precise than Geolocation when you know exact IP ranges. Geolocation uses AWS's IP-to-geography mapping; IP-based uses your own CIDR definitions.
MV
Multi-Value Routing Policy core
Return multiple healthy IPs — client-side load balancing with health checks
▶

Multi-Value routing returns multiple values (up to 8) in response to DNS queries. Unlike Simple routing with multiple IPs, Multi-Value routing can associate each record with a health check — only healthy records are returned.

→ Can return up to 8 healthy records per query. The client picks one randomly (client-side load balancing).
→ Each record in a Multi-Value group can have its own health check — only healthy records are included in responses. If a resource fails, its record is excluded.
→ Not a substitute for a load balancer — it is DNS-based and doesn't have ALB's layer 7 capabilities. But it does provide basic availability at the DNS level.
→ Difference from Simple: Simple returns all values with no health filtering. Multi-Value filters out unhealthy records before returning results.
EXAM TIP"Return multiple IPs but only healthy ones — no load balancer involved" → Multi-Value routing. Multi-Value ≠ replacement for ELB. ELB distributes connections across targets with health awareness and persistence. Multi-Value just returns up to 8 healthy IPs for client-side selection.
Module 8 — Integration
Hybrid DNS & Route 53 Resolvers
DNS resolution between on-premises networks and AWS VPCs — inbound and outbound resolver endpoints.
RES
Route 53 Resolver advanced
Bidirectional DNS resolution between VPCs and on-premises networks
▶

By default, DNS queries from within a VPC are handled by the VPC's built-in resolver (at the +2 IP address of the VPC CIDR, e.g. 10.0.0.2). For hybrid environments, Route 53 Resolver Endpoints allow DNS queries to flow between on-premises networks and AWS VPCs.

Endpoint typeDirectionWhat it doesUse case
Inbound EndpointOn-prem → AWSOn-premises DNS servers can forward queries to AWS. Route 53 resolves Private Hosted Zone names and returns answers to on-prem clients.On-prem servers need to resolve app.internal.example.com which is in a Route 53 Private Hosted Zone.
Outbound EndpointAWS → On-premEC2 instances can forward specific DNS queries to on-premises DNS servers via Resolver Rules.EC2 instances need to resolve on-prem hostnames like server.corp.internal that are managed by the on-prem DNS server.
→ Resolver Endpoints are deployed in a VPC across multiple AZs for high availability. Each endpoint gets ENIs with private IP addresses.
→ Outbound endpoints use Resolver Rules to define which domain names should be forwarded to which on-premises DNS server IP. Rules can be shared across VPCs via AWS RAM (Resource Access Manager).
→ Requires connectivity between the VPC and on-premises network — either Site-to-Site VPN or Direct Connect.
→ DNS Firewall: Route 53 Resolver DNS Firewall lets you block DNS queries to known malicious domains and allow only approved domains — works at the VPC resolver level.
EXAM TIP"On-prem servers need to resolve AWS Private Hosted Zone records" → Inbound Endpoint. "EC2 instances need to resolve on-prem hostnames" → Outbound Endpoint + Resolver Rule. Both require VPN or Direct Connect connectivity. Share Resolver Rules across multiple VPCs using RAM.
