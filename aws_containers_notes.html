Container fundamentals, the Docker workflow, and why containers are the foundation of modern AWS architectures.
DOC
Docker Fundamentals core
Images, containers, Dockerfile, and the build-push-run workflow
▶

Docker packages an application and all its dependencies (libraries, runtime, config) into a standardized unit called a container. Containers run identically on any host that has Docker — eliminating "works on my machine" problems.

→ Docker Image: a read-only blueprint for a container. Built from a Dockerfile. Stored in a container registry (Docker Hub, ECR). Immutable — any change creates a new image layer.
→ Docker Container: a running instance of a Docker image. Isolated process with its own filesystem, networking, and process space. Ephemeral — data is lost when the container stops unless mounted to a volume.
→ Dockerfile: a text file of instructions (FROM, RUN, COPY, EXPOSE, CMD) that define how to build an image layer by layer.
→ Workflow: Dockerfile → docker build → Image → docker push → Registry (ECR) → docker pull on host → docker run → Container.
→ Containers vs VMs: containers share the host OS kernel — much lighter than VMs (milliseconds to start vs minutes). Multiple containers run on one EC2 instance. VMs each have their own OS.
→ On AWS, Docker containers run on: ECS (AWS-managed orchestration), EKS (Kubernetes), Fargate (serverless containers), App Runner (fully managed), or directly on EC2 instances.
EXAM TIPContainers = lightweight, fast, portable. The key AWS container question is about choosing the right orchestration layer: ECS (AWS-native, simpler), EKS (Kubernetes, more control), Fargate (serverless, no EC2 management), App Runner (simplest — source code to running app). Know when each is appropriate.
Module 2 — ECS
ECS Core Concepts
Clusters, task definitions, services, and tasks — the building blocks of Amazon ECS.
ECS
ECS Architecture core
Cluster → Service → Task → Container — the four-level hierarchy
▶

Amazon ECS (Elastic Container Service) is AWS's native container orchestration platform. It manages where and how Docker containers run — scheduling, scaling, health checking, and networking — without you managing a Kubernetes cluster.

ConceptWhat it isAnalogy
ClusterLogical grouping of compute resources (EC2 instances or Fargate capacity) where tasks run.The data centre / pool of servers
Task DefinitionA JSON blueprint specifying: container images, CPU/memory, ports, IAM roles, logging config, volumes, environment variables. Like a Dockerfile but for the entire task.The recipe / spec sheet
TaskA running instance of a Task Definition. One or more containers running together. Ephemeral — stops when done.The actual running job
ServiceMaintains a desired number of tasks running at all times. Handles restarts on failure, integrates with ELB, and manages rolling deployments.The process supervisor / systemd
→ A Task Definition can define multiple containers — a primary application container and sidecar containers (log forwarder, proxy, agent). All containers in a task share network namespace and storage volumes.
→ ECS Service is for long-running applications (web servers, APIs). ECS Task run-once is for batch jobs, migrations, or scheduled work (like a cron job).
→ ECS integrates natively with ALB for service discovery and load balancing — each task registered as a target with its dynamic port (dynamic port mapping).
EXAM TIPTask Definition = the config (what to run). Task = the running instance (one execution). Service = keeps N tasks running (long-lived apps). Cluster = the compute pool. "Configure CPU, memory, ports, and IAM role for a container" → Task Definition. "Keep 3 replicas running with auto-restart" → ECS Service.
IAM
ECS IAM Roles core
Task Role vs Task Execution Role — two different IAM roles, two different purposes
▶

ECS uses two separate IAM roles that are frequently confused on the exam. Understanding which role grants what permissions is critical.

RoleUsed byGrants permissions forExample
Task RoleThe container/application running inside the taskAWS API calls the application makes (S3, DynamoDB, SQS, etc.)App needs to read from S3 → attach S3 read policy to the Task Role
Task Execution RoleECS agent on the host (Fargate or EC2)ECS infrastructure operations: pulling the container image from ECR, writing logs to CloudWatch, reading Secrets Manager/SSM Parameter StoreECS needs to pull image from private ECR → attach ECR pull policy to Task Execution Role
→ Each task can have its own unique Task Role — enabling least-privilege per microservice. Task A gets S3 access; Task B gets DynamoDB access; neither gets the other's permissions.
→ The Task Execution Role is required for Fargate — without it, Fargate cannot pull images or send logs. It is defined in the Task Definition.
EXAM TIP"Container can't access S3" → missing or insufficient Task Role. "ECS task fails to start — can't pull image from ECR" → missing or insufficient Task Execution Role (needs ecr:GetAuthorizationToken and ecr:BatchGetImage). Two roles, two different jobs — don't mix them up.

EC2 launch type vs Fargate — and how containers connect to each other and the outside world.

EC2 Launch Type vs Fargate core
Self-managed instances vs serverless containers — the fundamental ECS choice
▶
EC2 Launch TypeFargate Launch Type
InfrastructureYou provision and manage EC2 instances in the cluster. You install and run the ECS agent on each instance.AWS manages all underlying infrastructure. No EC2 instances to manage, patch, or scale.

→ Both launch types can coexist in the same ECS cluster — you can run some services on EC2 and others on Fargate within the same cluster.
EXAM TIP"No servers to manage for containers" or "serverless containers" → Fargate. "Need GPU for ML container workload" → EC2 launch type (Fargate doesn't support GPU). "Reduce cost for predictable container workloads" → EC2 with Reserved Instances. Fargate is almost always the preferred answer for new applications on the exam.

awsvpc, bridge, host — how containers get IP addresses
▶
ModeHow it worksRequired for

DAT
ECS Data Volumes — EFS Integration advanced
Persistent shared storage for ECS tasks across AZs
▶

ECS tasks are ephemeral — storage inside a container is lost when the task stops. For persistent or shared data, mount an Amazon EFS file system directly into ECS tasks.

→ EFS mounts work with both EC2 and Fargate launch types — the only persistent shared storage option for Fargate.
→ Multiple tasks across multiple AZs can all mount the same EFS file system — shared access to the same data.
→ EFS + Fargate = fully serverless persistent storage: no EC2 to manage, no storage to provision. Pay only for what you use.
→ S3 cannot be mounted as a file system by ECS tasks — use EFS for file-system access, S3 via SDK/API calls for object storage.
EXAM TIP"Fargate tasks need to share persistent storage across AZs" → EFS. Fargate cannot use EBS (EBS is AZ-locked and attaches to instances, not tasks). EFS + Fargate = the serverless persistent storage pattern for containers.
Module 4 — ECS
ECS Auto Scaling & Architectures
Scale ECS services automatically and wire ECS into common solution architecture patterns.
ASG
ECS Auto Scaling core
Scale task count — and for EC2 launch type, scale the underlying cluster too
▶

ECS Service Auto Scaling automatically adjusts the desired task count up or down based on CloudWatch metrics. For Fargate, this is single-tier. For EC2 launch type, you must also scale the underlying EC2 cluster — two separate scaling mechanisms.

→ ECS Service Auto Scaling uses AWS Application Auto Scaling (not EC2 Auto Scaling). Three policy types: Target Tracking (keep a metric at a target value), Step Scaling (alarm-based graduated response), Scheduled Scaling (scale at known times).
→ Useful scaling metrics: ECSServiceAverageCPUUtilization, ECSServiceAverageMemoryUtilization, ALBRequestCountPerTarget (from ALB).
→ Fargate Auto Scaling: straightforward — scale tasks up/down. AWS handles the underlying compute capacity automatically.
→ EC2 launch type — two-tier scaling: scale ECS service tasks (task count ↑) AND scale the EC2 ASG (add EC2 instances to have capacity for new tasks). Use ECS Cluster Capacity Providers to automate EC2 scaling based on task demand.
→ ECS Cluster Capacity Providers (EC2 launch type): automatically provision and scale EC2 instances when ECS tasks can't be scheduled due to insufficient cluster capacity. Replaces manual EC2 ASG management for ECS.
EXAM TIPFargate = scale tasks only (single tier, simple). EC2 = scale tasks + scale EC2 cluster (two tiers, more complex). "ECS tasks failing to schedule — not enough cluster capacity" → EC2 cluster needs more instances → use Capacity Providers or manually scale the ASG.
ARC
ECS Solution Architectures core
Event-driven, scheduled tasks, and CI/CD patterns with ECS
▶

ECS appears in several recurring exam architecture patterns — knowing the common wiring between ECS and other services lets you answer scenario questions quickly.

→ Event-driven ECS tasks: S3 upload → EventBridge rule → trigger ECS task (run-once batch job). Each file upload spins up a task to process it, then the task exits. No persistent service needed.
→ SQS + ECS workers: messages arrive in SQS → ECS Service polls SQS → tasks process messages → scale ECS service based on queue depth. Classic decoupled worker pattern.
→ ECS behind ALB: ALB distributes HTTP traffic across ECS tasks. Each task registers with the ALB target group dynamically. ALB health checks replace individual instance health checks.
→ CI/CD with ECS: CodePipeline → CodeBuild (build image, push to ECR) → CodeDeploy (rolling or blue/green deployment to ECS). Blue/green on ECS uses two target groups — ALB switches from blue to green with no downtime.
→ Scheduled tasks: EventBridge Scheduler → run ECS task on a cron schedule. Replaces cron jobs on EC2 with serverless container execution.
EXAM TIP"Process S3 uploads asynchronously using containers" → S3 → EventBridge → ECS Task (Fargate). "Replace EC2 cron jobs with containers" → EventBridge Scheduler → ECS Task. "Zero-downtime container deployments" → ALB blue/green with ECS + CodeDeploy.
Module 5 — ECR
Amazon ECR — Elastic Container Registry
Managed Docker image registry — store, version, scan, and replicate container images securely on AWS.
ECR
Amazon ECR Overview core
Private Docker registry — integrated with ECS, EKS, and IAM
▶

ECR (Elastic Container Registry) is AWS's fully managed Docker (and OCI) container image registry. It replaces Docker Hub for private images on AWS, integrating natively with ECS, EKS, CodeBuild, and IAM for access control.

→ Two repository types: Private (IAM-controlled access, your images) and Public (ECR Public Gallery — public images anyone can pull, like Docker Hub).
→ Images are stored in S3 behind the scenes — backed by S3's durability and availability.
→ IAM access control: repository policies and IAM policies control who can push/pull. The Task Execution Role of ECS tasks must have ecr:GetAuthorizationToken, ecr:BatchGetImage, and ecr:GetDownloadUrlForLayer to pull images.
→ Image scanning: ECR scans images for known CVEs (vulnerabilities). Two modes: Basic scanning (on push or on-demand, uses Clair) and Enhanced scanning (continuous, uses Amazon Inspector, covers OS and application dependencies).
→ Lifecycle policies: automatically expire and delete old/untagged images to control storage costs (e.g. keep only the last 10 tagged images per repo).
→ Cross-region and cross-account replication: replicate images to other regions and accounts automatically — useful for multi-region deployments.
→ ECR images are immutable by default (optional) — once a tag is pushed it cannot be overwritten. Ensures production images are never accidentally replaced.
EXAM TIP"ECS task fails to start — cannot pull image" → ECS Task Execution Role is missing ECR permissions (or the image doesn't exist in ECR). "Scan container images for vulnerabilities automatically" → ECR Enhanced Scanning with Amazon Inspector. "Store private Docker images on AWS" → ECR.
Module 6 — EKS
Amazon EKS — Elastic Kubernetes Service
Managed Kubernetes on AWS — when you need Kubernetes control, without operating the control plane.
EKS
Amazon EKS Overview core
Managed Kubernetes control plane — you bring the workers, AWS runs the master
▶

Amazon EKS (Elastic Kubernetes Service) runs a managed Kubernetes control plane on AWS. AWS manages the Kubernetes API server, etcd, and control plane availability. You manage (or let AWS manage) the worker nodes where your containers actually run.

→ EKS supports standard Kubernetes — if you have existing Kubernetes manifests, Helm charts, or tooling, they work on EKS with minimal changes.
→ Three node group types: Managed Node Groups (AWS provisions and manages EC2 instances as Kubernetes nodes, handles AMI updates), Self-Managed Nodes (you provision and manage EC2 instances yourself), Fargate (serverless — AWS runs each pod on its own isolated Fargate compute, no node management at all).
→ Kubernetes terminology: Pod = one or more containers (equivalent to ECS Task). Node = EC2 instance running pods (equivalent to ECS container instance). Deployment = desired state for pods (equivalent to ECS Service). Namespace = virtual cluster for isolation.
→ EKS integrates with: ALB/NLB (via AWS Load Balancer Controller), EBS/EFS (via CSI drivers for persistent volumes), IAM (via IRSA — IAM Roles for Service Accounts, equivalent to ECS Task Role), ECR, CloudWatch Container Insights.
→ Use EKS when: your team already uses Kubernetes, you're migrating existing Kubernetes workloads to AWS, you need advanced Kubernetes features (custom schedulers, operators, CRDs), or you require multi-cloud portability.
→ Use ECS when: building new AWS-native applications, your team doesn't know Kubernetes, you want simpler operations and tighter AWS integration, or you want lower management overhead.
EXAM TIPEKS vs ECS: EKS = Kubernetes API (portable, complex, powerful). ECS = AWS proprietary API (simpler, tighter AWS integration, less portable). "Company already uses Kubernetes and wants to migrate to AWS" → EKS. "New microservices app, wants simplest container orchestration on AWS" → ECS with Fargate. Both support Fargate for serverless compute.
IRSA
EKS IAM — IRSA advanced
IAM Roles for Service Accounts — grant AWS permissions to specific Kubernetes pods
▶

IRSA (IAM Roles for Service Accounts) allows you to associate an IAM role with a Kubernetes Service Account, and pods using that Service Account assume that IAM role. This is the EKS equivalent of ECS Task Roles.

→ Avoids the need to embed AWS credentials in pods or attach broad EC2 instance profiles to worker nodes. Each pod gets only the permissions it needs.
→ Works via OIDC (OpenID Connect) — EKS cluster has an OIDC provider, IAM trusts it, pods receive short-lived credentials injected as environment variables.
→ EKS IRSA = ECS Task Role concept, applied to Kubernetes pods.
EXAM TIP"Give a Kubernetes pod on EKS permission to access S3 without storing credentials" → IRSA (IAM Roles for Service Accounts). Same principle as ECS Task Role but the Kubernetes-native implementation.
Module 7 — Managed Platforms
App Runner & App2Container
The simplest path to running containers on AWS — source code or image to production with zero infrastructure config.
APR
AWS App Runner core
Deploy containerized apps and APIs in minutes — no infrastructure, no orchestration
▶

AWS App Runner is a fully managed service for deploying containerized web applications and APIs. You provide a container image or source code — App Runner handles everything else: building, deploying, load balancing, TLS, auto-scaling, and health checks.

→ Two input sources: Container image (from ECR or Docker Hub) or Source code (GitHub — App Runner builds the image automatically using a detected runtime or a Dockerfile).
→ Fully managed: no ECS clusters, no Fargate task definitions, no ALB to configure, no ACM certificate to attach. App Runner provisions all of this transparently.
→ Auto-scaling: scales based on concurrent requests. Scales to zero when idle (optional — pay only when requests are being handled). Scales back up instantly on new requests.
→ Each App Runner service gets a unique HTTPS endpoint (custom domain support available). TLS is handled automatically.
→ Supports VPC connectivity — App Runner services can reach private resources in your VPC (RDS, ElastiCache) via VPC connectors.
→ Target audience: developers who want to run web apps and APIs without learning ECS, Fargate, or Kubernetes. Ideal for startups, microservices, APIs, and internal tools.
EXAM TIP"Deploy a web application with no infrastructure management — simplest possible container deployment" → App Runner. App Runner sits above ECS/Fargate in the abstraction hierarchy: more opinionated, less control, zero ops. Compare: EC2 (most control) → ECS EC2 → ECS Fargate → App Runner (least control, easiest).
A2C
AWS App2Container (A2C) tool
Automatically containerize existing Java and .NET applications running on EC2
▶

AWS App2Container is a command-line tool that analyzes running Java (Spring, Tomcat) and .NET (IIS) applications on EC2 or on-premises and automatically generates Docker images, ECS/EKS deployment artifacts, and CI/CD pipeline configurations.

→ Discovers running Java/.NET processes → captures runtime dependencies → packages into a Docker image → pushes to ECR → generates ECS Task Definition or EKS deployment YAML.
→ No code changes required to the application — A2C handles the containerization automatically.
→ Supports: Java apps on Tomcat, JBoss, WebLogic. .NET apps on Windows IIS. Runs on Linux and Windows EC2 instances.
→ Optional pipeline generation: creates CloudFormation templates and CodePipeline configuration for CI/CD.
EXAM TIP"Migrate legacy Java or .NET applications from EC2 to containers without rewriting code" → AWS App2Container. It is a migration acceleration tool, not a production service. The output is Docker images + ECS/EKS deployment files that you then manage yourself.
