Module 1 — CloudFront
CloudFront Overview
AWS's global Content Delivery Network — cache content at 450+ edge locations worldwide for low-latency delivery.
CDN
What is CloudFront? core
Global CDN — serve cached content from the edge, not the origin
▶

CloudFront is AWS's Content Delivery Network (CDN). It caches copies of your content at Edge Locations distributed around the world, so users receive content from a nearby edge location instead of your origin server — dramatically reducing latency and origin load.

→ CloudFront has 450+ edge locations globally (also called Points of Presence / PoPs). These are distinct from AWS Regions and AZs — they are smaller caching nodes.
→ Cache hit: request is served directly from the edge location — fast, no origin request. Cache miss: edge location fetches from the origin, caches the response, then serves the user.
→ CloudFront reduces origin load — instead of every user request hitting your S3 bucket or ALB, most requests are served from cache. Reduces costs and protects origin from traffic spikes.
→ Content is cached for a configurable duration — the TTL (Time To Live). After TTL expires, the next request triggers a fresh fetch from the origin.
→ CloudFront supports both static content (images, JS, CSS, videos) and dynamic content (API responses, personalized pages). Dynamic content can be forwarded to the origin with full header/cookie/query-string passthrough.
→ CloudFront integrates natively with AWS Shield (DDoS protection), AWS WAF (web application firewall), ACM (free SSL certificates), Route 53, and Lambda@Edge / CloudFront Functions.
EXAM TIP"Improve global read performance for static content" → CloudFront. "Protect against DDoS attacks at the edge" → CloudFront + Shield. "Add HTTPS to an S3 static website" → CloudFront (S3 website hosting is HTTP only). CloudFront is always the answer when the question involves global caching, low-latency delivery, or edge-level SSL termination.
OAC
CloudFront with S3 — Origin Access Control core
Lock down S3 so only CloudFront can read it — OAC replaces OAI
▶

When CloudFront serves content from an S3 bucket, you should block direct public access to the S3 bucket so users must go through CloudFront. This is done using Origin Access Control (OAC) — the modern replacement for the older Origin Access Identity (OAI).

→ OAC (Origin Access Control): the current recommended approach. Creates an IAM-based identity that CloudFront uses to sign requests to S3. The S3 bucket policy grants s3:GetObject only to the CloudFront distribution's OAC principal.
→ OAI (Origin Access Identity): the legacy approach (still valid, still tested). A CloudFront special user identity granted access via the bucket policy. OAC is preferred because it supports all S3 regions including SSE-KMS encrypted buckets.
→ With OAC/OAI configured: S3 Block Public Access remains enabled. Users who access the S3 URL directly get a 403 Forbidden. Only requests through CloudFront are authorized.
→ OAC supports: S3 buckets in all regions, SSE-KMS encrypted buckets, and S3 Object Lambda.
→ The S3 bucket used as a CloudFront origin should be configured as a REST API endpoint (bucket name format), not the S3 website endpoint — unless you need website-specific features like custom 404 pages.
EXAM TIP"Restrict S3 bucket access so only CloudFront can serve content" → OAC (or OAI for legacy). S3 Block Public Access stays ON. Users hitting S3 directly = 403. Users hitting CloudFront = content served. OAC is the modern approach; OAI still appears in exam questions — know both.
SECURITY WARNINGNever disable S3 Block Public Access when using CloudFront + S3. The whole point of OAC/OAI is that the bucket stays private. If you make the bucket public, you bypass CloudFront security entirely (WAF, signed URLs, geo restrictions).
Module 2 — Origins & Caching
Origins & Cache Behavior
All supported origin types, how CloudFront caches responses, and how cache behavior is configured.
ORI
CloudFront Origins core
S3, ALB, EC2, API Gateway, and custom HTTP origins
▶

A CloudFront origin is the source of truth for your content. CloudFront fetches content from the origin on a cache miss and stores a copy at the edge. Multiple origins can be configured per distribution, with different cache behaviors routing different URL patterns to different origins.

Origin TypeAccess methodKey considerations

→ For ALB or EC2 origins: the backend Security Group must allow inbound traffic from CloudFront's managed prefix list (or published IP ranges). CloudFront edge nodes have known IP ranges that AWS publishes and maintains.
→ Origin Groups: configure a primary and failover origin. If the primary returns a 5xx error, CloudFront automatically retries the request on the failover origin — provides origin-level HA.
EXAM TIP"CloudFront with EC2/ALB" → the EC2/ALB must be publicly accessible (CloudFront is outside your VPC). "Keep EC2 private, only CloudFront can reach it" → put an ALB in front (ALB public, EC2 private), use CloudFront → ALB. EC2 cannot be private if CloudFront needs to reach it directly.
CBH
Cache Behaviors & TTL core
Control what gets cached, for how long, and what reaches the origin
▶

Cache behaviors are rules that determine how CloudFront handles requests for different URL patterns. Each behavior maps a path pattern to an origin and specifies caching settings, allowed HTTP methods, and what headers/cookies/query strings to forward.

→ Cache behavior path patterns are matched in order of specificity. A default behavior (*) catches everything not matched by more specific patterns.
→ TTL settings: Default TTL (used when the origin doesn't send cache headers), Minimum TTL, and Maximum TTL. If the origin sends Cache-Control headers, CloudFront respects them within the min/max TTL range.
→ Cache key: by default, the cache key is just the URL. You can include headers, cookies, and query strings in the cache key — but this reduces cache hit rate (more unique keys = fewer hits).
→ Origin request policy: controls what CloudFront sends to the origin (headers, cookies, query strings) separately from what's in the cache key. Allows forwarding context to the origin without fragmenting the cache.
→ HTTP methods: GET/HEAD always cached. PUT/POST/PATCH/DELETE are never cached — always forwarded to origin. OPTIONS can be cached or forwarded.
EXAM TIPIncluding cookies or query strings in the cache key reduces cache hit ratio — more specific keys = fewer cache hits. For dynamic APIs, forward query strings to origin but don't include them in the cache key unless you specifically need per-query caching.
Module 3 — Security
Security & Geo Restriction
HTTPS enforcement, signed URLs/cookies, geo blocking, and WAF integration at the edge.
GEO
CloudFront Geo Restriction core
Block or allow content delivery based on the user's country
▶

CloudFront Geo Restriction (also called Geoblocking) allows you to restrict which countries can access your content — either with an allowlist (only listed countries can access) or a blocklist (listed countries are denied).

→ Allowlist: only users in the specified countries can access content. All other countries receive a 403 Forbidden.
→ Blocklist: users in the listed countries are blocked (403). All other countries can access.
→ Country determination is based on a third-party geo-IP database that maps IP addresses to countries. Not 100% accurate — users with VPNs can bypass it.
→ Use cases: copyright and licensing restrictions (streaming rights by country), export compliance (OFAC sanctions), regulatory requirements.
→ Restriction is applied at the CloudFront distribution level — applies to all content in the distribution unless overridden by a more granular solution like Lambda@Edge.
EXAM TIPCloudFront Geo Restriction blocks at the country level using IP geolocation. Compare to Route 53 Geolocation routing, which controls DNS resolution by country. These are complementary, not identical — CloudFront blocks content delivery; Route 53 controls which DNS record is returned. "Block users in specific countries from accessing content" → CloudFront Geo Restriction.
SGN
Signed URLs & Signed Cookies advanced
Time-limited, user-specific access to private CloudFront content
▶

CloudFront Signed URLs and Signed Cookies let you restrict access to content so only authenticated and authorized users can access it — with an expiration time and optional IP restriction.

Signed URLSigned Cookie
Grants access toOne specific file / objectMultiple files / an entire path pattern
How it worksURL includes signature, expiry, and optional IP. Share the URL with the user.Cookie includes signature and expiry. User's browser sends cookie on each request.
Use caseShare a single video, single download link, individual premium resourceMembership site with access to a whole content library, subscriber area
URL change required?Yes — URL changes per signed resourceNo — existing URLs work, cookie authorizes them
→ Signed URLs from CloudFront vs S3 Pre-signed URLs: CloudFront signed URLs go through CloudFront (caching, WAF, geo restriction apply). S3 pre-signed URLs go directly to S3 (bypass CloudFront entirely).
→ Signed URLs are created using CloudFront Key Pairs (trusted key groups) — managed separately from IAM. The private key signs the URL; CloudFront verifies with the public key.
→ Can restrict access by: expiration date/time, IP address range (optional), and path pattern (which files are accessible).
EXAM TIP"Paid video streaming — only subscribers can watch" → CloudFront Signed URL (per video) or Signed Cookie (subscriber library). "S3 Pre-signed URL vs CloudFront Signed URL" → CloudFront signed = through CDN (caching, security policies apply). S3 pre-signed = bypasses CloudFront, goes direct to S3.
HTTPS
HTTPS & SSL Enforcement core
Viewer protocol policy, SSL certificates, and end-to-end encryption
▶
→ Viewer Protocol Policy: controls how CloudFront handles HTTP vs HTTPS from viewers. Options: HTTP and HTTPS (allow both), Redirect HTTP to HTTPS (preferred), HTTPS Only (reject HTTP).
→ Origin Protocol Policy: controls how CloudFront communicates with the origin. Options: HTTP Only, HTTPS Only, or Match Viewer (use whatever the viewer used).
→ CloudFront uses ACM certificates for HTTPS. ACM certificates for CloudFront must be created in us-east-1 (N. Virginia) — regardless of the distribution's origin region.
→ Custom SSL certificates are supported via SNI (no extra cost) or dedicated IP (legacy, $600/month per distribution — avoid).
EXAM TIPACM certificate for CloudFront must be in us-east-1 — this is a very common trap. A certificate in eu-west-1 will NOT work with CloudFront even if the origin is there. Always create CloudFront SSL certificates in us-east-1.
Module 4 — Advanced CloudFront
Cache Invalidation & Advanced Features
Force content refresh at the edge, Lambda@Edge, CloudFront Functions, and pricing model.
INV
Cache Invalidation core
Force CloudFront edge locations to discard cached content before TTL expires
▶

Normally, cached content is served until the TTL expires — even if you've updated the origin. Cache Invalidation forces all edge locations to remove specific cached files immediately, so the next request fetches a fresh copy from the origin.

→ Invalidation paths support wildcards: /images/* invalidates all files in the images prefix. /* invalidates the entire distribution (use sparingly — increases origin load).
→ Invalidation is not instant — it propagates to all edge locations, which takes a few seconds to a couple of minutes. Not suitable as a real-time solution.
→ Cost: first 1,000 invalidation paths per month are free. Additional paths are $0.005 per path. Wildcard counts as one path regardless of how many objects it matches.
→ Best practice: instead of frequent invalidations, use versioned file names (e.g. app.v2.js instead of app.js). New filename → guaranteed fresh fetch, no invalidation needed, old version still cached for users who haven't refreshed.
EXAM TIP"Updated S3 objects are not showing up in CloudFront" → run a cache invalidation or reduce TTL. "Most cost-effective way to ensure users always get the latest version" → versioned file names (no invalidation cost, no TTL wait). Cache invalidation is the immediate but costlier solution.

Run code at the edge — customize request/response without a round-trip to origin
▶

PRC
CloudFront Pricing advanced
Price classes and data transfer costs
▶
→ Pricing varies by region — data transfer out from North America edge locations is cheapest; Asia Pacific and South America are most expensive.
→ Price Classes: reduce cost by limiting which edge locations are used. Three options: All (all edge locations, best performance), Price Class 200 (excludes most expensive regions), Price Class 100 (only North America and Europe — cheapest).
→ Data transfer from CloudFront to viewers is billed per GB. Data transfer from origin to CloudFront edge (origin fetch) is also billed — but much less than direct origin-to-viewer transfer since the edge caches reduce origin fetches.
→ Free tier: 1 TB of data transfer out and 10 million HTTP requests per month.
EXAM TIP"Reduce CloudFront costs for a global application" → use a lower Price Class (e.g. Price Class 100 if your users are mainly in North America/Europe). Trade-off: higher latency for users in excluded regions. If global performance matters → Price Class All.
Module 5 — Global Accelerator
AWS Global Accelerator
Route TCP/UDP traffic over AWS's private global network — static IPs, anycast routing, and instant failover.
GA
Global Accelerator Overview core
Improve global application performance using AWS's private backbone network
▶

AWS Global Accelerator routes your users' traffic through AWS's global private network instead of the public internet. Users connect to the nearest AWS edge location (via anycast), and their traffic travels over AWS's high-throughput, low-latency private backbone to your application endpoints.

→ Global Accelerator provides 2 static anycast IPv4 addresses (and optionally 2 IPv6) that are globally fixed. These IPs are your application's single entry point worldwide — no DNS changes needed for failover.
→ Anycast routing: both IP addresses are announced from all AWS edge locations simultaneously. The user's network automatically routes to the nearest edge location using BGP routing protocols — no DNS lookup required.
→ Works at Layer 3 / Layer 4 — supports TCP and UDP. Does not understand HTTP (no content-based routing, no caching). This distinguishes it from CloudFront.
→ Endpoints can be: ALBs, NLBs, EC2 instances, or Elastic IPs in any AWS region. You can have multiple endpoints across regions — Global Accelerator routes to the healthiest, nearest one.
→ Traffic dials: assign a percentage (0–100%) of traffic to each endpoint group (region). Use to gradually shift traffic during deployments or maintenance — similar to weighted routing but for live TCP connections.
→ Health checks: monitors endpoints continuously. On failure, traffic is rerouted to healthy endpoints within less than 1 minute — without DNS TTL delays (unlike Route 53 failover).
→ Integrated with AWS Shield Standard for DDoS protection at no additional cost. The anycast static IPs are protected by Shield.
EXAM TIP"Static IP for a global application with instant failover" → Global Accelerator. "Non-HTTP protocol (TCP/UDP gaming, IoT, VoIP) needing global performance" → Global Accelerator. "Clients need to whitelist a fixed IP for a globally distributed app" → Global Accelerator (2 static IPs). The key differentiator from CloudFront: Global Accelerator has NO caching — it's about network routing, not content delivery.
HOW
How Global Accelerator Works advanced
Anycast, edge locations, and the private AWS backbone
▶

Understanding the traffic flow through Global Accelerator clarifies why it improves performance and provides unique benefits over DNS-based solutions.

1. User's request hits the nearest AWS edge location (same 450+ PoPs as CloudFront) — typically within milliseconds of the user.
2. At the edge, traffic is handed off to the AWS global private network — bypassing the unpredictable public internet for the remainder of the journey.
3. Traffic travels over AWS's dedicated fiber backbone to the optimal application endpoint (determined by health, proximity, and traffic dial settings).
4. The response follows the same path back to the user via the edge location.
→ The public internet is only used for the first mile (user → nearest edge). Everything else is AWS's private network — more reliable, lower latency, consistent throughput.
→ Contrast with DNS-based solutions (Route 53 latency routing): DNS routes to a region, but the traffic still travels the public internet between the user and the regional endpoint.
EXAM TIPThe performance gain from Global Accelerator is greatest for users geographically distant from the application region — e.g. a user in Sydney accessing an app in us-east-1. The Sydney user hits an Australian edge immediately, then rides AWS's backbone the rest of the way. Without GA, the entire path is over the public internet.
