Module 1 — Foundations
High Availability & Scalability
The core vocabulary and mental model behind every HA and scaling decision on the exam.
SCL
Scalability core
Vertical vs horizontal scaling — scale up vs scale out
▶

Scalability means an application can handle greater load by adapting. There are two fundamentally different strategies, and the exam tests whether you know which one to apply.

Vertical Scaling (Scale Up)Horizontal Scaling (Scale Out)
What it meansIncrease the size/power of a single instance — more CPU, RAM, storage.Add more instances of the same size to distribute load.
ExampleUpgrade from t3.medium to t3.2xlargeAdd 4 more t3.medium instances behind a load balancer
LimitHardware limit — instances have a maximum size (e.g. u-24tb1.metal)No practical limit — can add thousands of instances
Downtime?Usually requires stop/start — brief downtimeNone — new instances added to existing pool
Best forNon-distributed systems (databases, single-server apps)Distributed systems, stateless web/app tiers, microservices
AWS servicesRDS read replicas (scale vertically for writes), ElastiCacheEC2 Auto Scaling Groups, ECS, Lambda
EXAM TIP"Increase the instance size" = vertical scaling. "Add more instances" = horizontal scaling. The exam almost always prefers horizontal scaling as the architecturally correct answer because it avoids single points of failure and hardware limits.
HA
High Availability core
Running across multiple AZs so the system survives a single data-centre failure
▶

High Availability (HA) means your system can survive the loss of a component — most commonly an entire Availability Zone — without user-visible downtime. HA is achieved by running redundant copies in at least two AZs with automatic failover.

→ HA is distinct from scalability. You can be highly available without being scalable, and vice versa. HA is about surviving failures; scalability is about handling more load.
→ Active-Active HA: all nodes serve traffic simultaneously. An ELB distributing across instances in multiple AZs is active-active.
→ Active-Passive HA: one node is primary (active) and one is standby (passive). RDS Multi-AZ is active-passive — the standby only takes over on failover.
→ The minimum for HA in AWS: deploy resources in at least 2 Availability Zones.
→ AWS ELBs are inherently multi-AZ — they provision load balancer nodes in each enabled AZ automatically.
EXAM TIPIf the exam asks "how to make this architecture highly available" → deploy across at least 2 AZs + add a Load Balancer. Active-Active = all nodes live (ELB + ASG). Active-Passive = primary + standby (RDS Multi-AZ). Know which services use which model.
Module 2 — Load Balancers
Elastic Load Balancing Overview
What ELB is, why you use it, the four load balancer types, and the core concepts all types share.
ELB
ELB Overview core
Managed load balancing service — distributes traffic across targets
▶

Elastic Load Balancing (ELB) is a managed service that automatically distributes incoming traffic across multiple targets (EC2 instances, containers, Lambda functions, IP addresses) across one or more Availability Zones. AWS provisions, maintains, and scales the load balancer infrastructure.

→ ELB provides a single DNS endpoint for your application — clients never know about individual backend instances. DNS resolves to the ELB, which routes to healthy targets.
→ Health checks: ELB continuously probes each registered target (HTTP, HTTPS, TCP, gRPC). Unhealthy targets are removed from rotation automatically. If all targets are unhealthy, requests are routed to all of them (to avoid total blackout).
→ ELBs are regional and span multiple AZs. An ELB node is deployed in each enabled AZ — if an AZ goes down, the ELB routes only to healthy AZs.
→ Security: the ELB Security Group accepts internet traffic (e.g. port 80/443 from 0.0.0.0/0). EC2 Security Groups should only allow traffic from the ELB's Security Group, not from the internet directly.
→ ELBs integrate with: EC2 Auto Scaling Groups, ECS, ACM (SSL certificates), CloudWatch (metrics), Route 53, WAF, Shield, and Cognito.
EXAM TIPELB Security Group = allow 0.0.0.0/0 on port 80/443. EC2 Security Group = allow only from ELB Security Group (not 0.0.0.0/0). This is the correct layered security pattern — the exam tests it frequently.

TGT
Target Groups core
Named groups of targets that receive routed traffic
▶

A Target Group is a logical grouping of targets (EC2 instances, IP addresses, Lambda functions, or another ALB) that a load balancer routes requests to. Health checks are configured per target group.

→ Target types: Instance (EC2 instance ID), IP (any IP in a VPC or on-premises via VPN/Direct Connect), Lambda (ALB only), ALB (NLB can forward to an ALB target group).
→ A single load balancer can route to multiple target groups using rules — e.g. /api/* → API target group, /static/* → static content target group.
→ Health check settings (path, interval, healthy/unhealthy threshold) are configured at the target group level, not the listener level.
→ Targets in a target group can span multiple Availability Zones — the ELB distributes traffic across all healthy targets regardless of AZ (subject to cross-zone load balancing settings).
Module 3 — Application Load Balancer
ALB — Application Load Balancer
Layer 7 HTTP/HTTPS load balancer — content-based routing, microservices, and modern web architectures.
ALB
ALB Core Capabilities core
Layer 7 routing, supported protocols, and target types
▶

ALB operates at Layer 7 (HTTP/HTTPS) and can make routing decisions based on the content of the request — not just the IP address or port. It is the go-to load balancer for web applications, microservices, and container-based architectures.

→ Supported protocols: HTTP (port 80), HTTPS (port 443), WebSocket, and HTTP/2. Also supports gRPC for microservice communication.
→ Can route to: EC2 instances, ECS containers (dynamic port mapping), Lambda functions (HTTP invocation), and IP addresses (including on-premises).
→ The ALB sees the actual HTTP request and can log request headers, path, query parameters, HTTP method, and more — visibility that Layer 4 balancers don't have.
→ ALB has a fixed hostname (e.g. xxx.region.elb.amazonaws.com). The ALB's IP changes — never hardcode an ALB's IP address, always use the DNS name.
→ The application servers see the ALB's private IP as the source IP (not the client IP). The actual client IP is passed in the X-Forwarded-For header. Client port → X-Forwarded-Port. Protocol → X-Forwarded-Proto.
EXAM TIP"Application needs to know the client's original IP address" → read the X-Forwarded-For header (not the source IP, which is the ALB's IP). ALB IPs are dynamic — always use DNS name. NLB preserves client IP by default; ALB does not.
RUL
ALB Routing Rules core
Path-based, host-based, header-based, query string, and IP routing
▶

ALB listener rules evaluate incoming requests and route them to different target groups based on conditions. Rules are evaluated in priority order; the first match wins. A default rule (lowest priority) catches everything else.

Rule TypeExample conditionUse case
Path-based/api/* → API servers
/images/* → static server		Route different URL paths to different microservices or backend pools.
Host-basedmobile.example.com → mobile TG
www.example.com → web TG		Multiple domains (subdomains) served from a single ALB.
HTTP headerUser-Agent: Mobile*Route mobile users to a different version of the app.
HTTP methodPOST → write TG
GET → read TG		Separate read and write traffic to different backend pools.
Query string?version=v2 → v2 TGRoute based on query parameters — A/B testing, feature flags.
Source IP (CIDR)10.0.0.0/24 → internal TGRoute internal corporate traffic differently from external users.
→ Rule actions: Forward (to target group), Redirect (HTTP → HTTPS, or URL rewrite), Fixed Response (return a static HTTP response, e.g. 403 Forbidden or maintenance page).
→ Conditions can be combined with AND logic within a single rule. Multiple rules use OR logic — the first matching rule wins.
EXAM TIPALB routing rules are the primary exam topic for ALB. "Route /api to one group and /web to another" → path-based rules. "Two subdomains on one ALB" → host-based rules. "Redirect HTTP to HTTPS at the load balancer" → ALB redirect rule (no app code needed).
ECS
ALB with ECS & Microservices advanced
Dynamic port mapping and one ALB for many services
▶

ALB is the preferred load balancer for ECS (container) workloads because it supports dynamic port mapping — ECS assigns random host ports to containers, and ALB automatically discovers them via target group registration.

→ ECS registers each container task as a target with its dynamic port — no need to pre-configure ports. ALB discovers them via the ECS service integration.
→ A single ALB can serve many microservices using path-based or host-based routing — each service gets its own target group. Eliminates the cost of one NLB/ALB per service.
→ ALB supports Lambda function targets — ALB invokes the Lambda directly on HTTP request. No API Gateway needed for simple HTTP invocations.
EXAM TIP"Route different URL paths to different ECS microservices" → ALB with path-based rules + separate target groups per service. "Invoke Lambda via HTTP" → ALB with Lambda target group (cheaper than API Gateway for simple use cases).
Module 4 — Network & Gateway Load Balancers
NLB & Gateway Load Balancer
NLB for ultra-low latency TCP/UDP traffic; GWLB for transparent inline security appliances.
NLB
Network Load Balancer (NLB) core
Layer 4 TCP/UDP — static IPs, extreme performance, TLS passthrough
▶

NLB operates at Layer 4 (TCP/UDP) and routes connections based purely on IP protocol data — it cannot inspect HTTP content. It is designed for extreme throughput and latency-sensitive workloads: gaming, real-time streaming, financial trading.

→ Can handle millions of requests per second with latency around ~100 milliseconds (vs ~400ms for ALB). No request parsing overhead.
→ NLB provides one static IP address per Availability Zone. You can also assign an Elastic IP per AZ — critical when clients need to whitelist a fixed IP in their firewall.
→ NLB preserves the client's source IP address natively — the backend sees the actual client IP, not the NLB's IP (unlike ALB, which uses X-Forwarded-For).
→ Supports: TCP, UDP, TLS (TCP with TLS termination), and TCP_UDP listeners.
→ NLB target groups can contain: EC2 instances, IP addresses, and ALB (NLB → ALB chaining — gives you a static IP with Layer 7 routing).
→ Health checks support TCP, HTTP, and HTTPS — allowing health check even on non-HTTP protocols.
→ Cross-zone load balancing is disabled by default for NLB (unlike ALB where it's enabled by default). When disabled, each AZ's NLB node only routes to targets in its own AZ.
EXAM TIP"Client needs to whitelist a static IP for the load balancer" → NLB with Elastic IP. "TCP/UDP protocol, not HTTP" → NLB. "Client's source IP must reach the backend" → NLB (preserves source IP). "NLB + content-based routing" → NLB → ALB target group combo.
GWLB
Gateway Load Balancer (GWLB) advanced
Layer 3 — transparent traffic inspection through virtual appliances
▶

GWLB allows you to deploy, scale, and manage a fleet of third-party virtual network appliances (firewalls, intrusion detection/prevention systems, deep packet inspection) transparently in the traffic path. Traffic goes through the appliance before reaching your application.

→ Operates at Layer 3 (Network layer) — forwards entire IP packets, not just TCP connections.
→ Uses the GENEVE protocol on port 6081 to encapsulate and forward traffic to appliances.
→ Traffic flow: Internet → GWLB → 3rd-party appliance (e.g. Palo Alto firewall) → GWLB → application. The appliance inspects and either allows or drops the traffic.
→ Combines two functions: transparent network gateway (single entry/exit point for all traffic) + load balancer (distributes traffic across appliances for scale).
→ Appliances are registered as targets in a GWLB target group. GWLB scales the appliance fleet automatically.
EXAM TIP"All traffic must pass through a firewall / IDS / IPS before reaching the application" → GWLB. "Inspect all inbound internet traffic with a third-party security appliance" → GWLB. GWLB is the only ELB type that operates at Layer 3 and uses GENEVE protocol.
Module 5 — ELB Advanced Features
ELB Advanced Features
Sticky sessions, cross-zone load balancing, SSL/TLS termination, SNI, and connection draining.
STK
Sticky Sessions (Session Affinity) core
Route a client's requests to the same target for the duration of a session
▶

Sticky sessions (also called session affinity) ensure that a specific client is always routed to the same backend instance for the duration of their session. Implemented using a cookie that the ELB sets on the client.

Cookie typeNameDurationWho controls it
Application-based cookieCustom name set by your app (not AWSALB or AWSALBAPP — those are reserved)Set by the applicationYour application generates the cookie. ALB uses it for routing.
Duration-based cookieAWSALB (ALB) / AWSELB (CLB)1 second to 7 daysELB generates and manages the cookie. Simplest to set up.
→ Supported on ALB and CLB. NLB uses flow hash routing (same client IP to same target) instead of cookies.
→ Stickiness can cause imbalanced load if one target gets many long-lived sessions — some instances may be overloaded while others are idle.
→ Use case: stateful applications that store session data locally (shopping cart, user session). Best practice is to use a shared session store (ElastiCache/DynamoDB) and keep the app stateless instead.
EXAM TIP"Users lose their shopping cart when redirected to a different server" → enable sticky sessions. But the architecturally correct answer is "move session data to ElastiCache or DynamoDB so all servers can access it." Both may appear as answer options — the exam context determines which is being asked.
CZ
Cross-Zone Load Balancing core
Distribute traffic evenly across all instances in all AZs, not just the local AZ
▶

Without cross-zone load balancing, each ELB node routes traffic only to targets in its own Availability Zone. With it enabled, each ELB node can send traffic to any target in any AZ — balancing load even when AZs have different numbers of instances.

→ Scenario: AZ-A has 2 instances, AZ-B has 8 instances. Without cross-zone: each AZ gets 50% of traffic → AZ-A instances each handle 25%, AZ-B instances each handle 6.25% — very uneven. With cross-zone: all 10 instances each handle 10%.
→ ALB: cross-zone load balancing is always enabled (cannot be disabled). No inter-AZ data transfer charge.
→ NLB and GWLB: cross-zone load balancing is disabled by default. Can be enabled — but incurs inter-AZ data transfer charges.
→ CLB: disabled by default, no charge when enabled.
EXAM TIPThis is a high-frequency exam topic. Remember the defaults: ALB = always on, free. NLB/GWLB = off by default, costs money when enabled. CLB = off by default, free when enabled. The uneven distribution scenario with different instance counts per AZ is a classic exam question.
SSL
SSL/TLS & SNI core
Terminate HTTPS at the load balancer — ACM integration and SNI for multiple certs
▶

ELBs can terminate SSL/TLS connections — decrypting HTTPS traffic and forwarding plain HTTP to backend instances. This offloads the computational overhead of encryption from your application servers.

→ SSL certificates are managed via AWS Certificate Manager (ACM). ACM handles provisioning, renewal, and deployment to the ELB automatically — no manual cert rotation.
→ HTTPS listener: configure on port 443. You specify a default SSL certificate. You can optionally add multiple certs for multiple domains.
→ SNI (Server Name Indication): allows a single ALB or NLB to serve multiple SSL certificates (one per domain) on the same port 443. The client indicates the desired hostname in the TLS handshake, and the ELB serves the matching cert.
→ SNI is supported on ALB and NLB, but not CLB — CLB only supports one certificate per listener.
→ Security policies control the SSL protocols and ciphers accepted by the ELB. AWS provides predefined policies (e.g. ELBSecurityPolicy-TLS13-1-2-2021-06) — choose the latest TLS-1.3 compatible policy for best security.
→ Traffic between ELB and backend instances travels on the private AWS network — can be plain HTTP (common) or HTTPS (end-to-end encryption). For HTTPS to backends, they need valid certs too.
EXAM TIP"Multiple SSL certificates for multiple domains on one load balancer" → SNI on ALB or NLB (not CLB). "Automate SSL certificate renewal" → ACM. CLB requires one certificate per listener — to host multiple domains with CLB, you need multiple CLBs. SNI was designed specifically to solve this problem.
DRN
Connection Draining (Deregistration Delay) core
Allow in-flight requests to complete before removing an instance from the ELB
▶

When an instance is deregistered from a target group (due to scale-in, failure, or maintenance), Connection Draining allows existing in-flight requests to complete before the instance is fully removed. No new requests are sent to the draining instance.

→ Called Connection Draining on CLB, and Deregistration Delay on ALB and NLB — same concept, different names.
→ Default draining period: 300 seconds (5 minutes). Range: 1 to 3,600 seconds. Can be set to 0 to disable (immediate deregistration).
→ During the draining period: existing connections are kept open and served. No new requests are routed to the draining instance.
→ Set a low value for short-lived requests (API calls, static assets). Set a high value for long-lived operations (file uploads, database migrations, report generation).
EXAM TIP"Users getting errors when instances are removed during a deployment or scale-in event" → connection draining not configured or set too low. "Long-running operations are being interrupted" → increase deregistration delay. Default is 300 seconds — this number is tested.
Module 6 — Auto Scaling
Auto Scaling Groups — Core
Automatically add and remove EC2 instances to match demand — the backbone of elastic AWS architectures.
ASG
ASG Overview core
Scale EC2 fleets automatically — min, max, and desired capacity
▶

An Auto Scaling Group manages a collection of EC2 instances as a logical unit. It automatically launches instances when load increases (scale out) and terminates instances when load decreases (scale in), always maintaining capacity within your defined boundaries.

→ Three capacity settings: Minimum (floor — ASG never goes below this), Maximum (ceiling — ASG never exceeds this), Desired (target — ASG tries to maintain this exact count).
→ ASG is free — you pay only for the EC2 instances it launches, not for the ASG itself.
→ ASG replaces unhealthy instances automatically. If an EC2 instance fails its health check (from ASG or an attached ELB), ASG terminates it and launches a replacement.
→ ASG distributes instances across multiple AZs evenly. If one AZ loses capacity, ASG launches replacements in other AZs — this is how high availability is maintained.
→ When an ASG is attached to an ELB, new instances are automatically registered with the target group on launch and deregistered on termination.
EXAM TIPASG = elastic + highly available. Min/Max/Desired is a must-know. If an ASG is set to min=2 max=10 desired=4 and an instance fails, ASG launches a new one to get back to 4. ASG integrates with ELB health checks — ELB-unhealthy instances get terminated and replaced (more powerful than EC2 status checks alone).
LT
Launch Templates (vs Launch Configurations) core
Blueprint for instances launched by an ASG
▶

A Launch Template (or the older Launch Configuration) defines the EC2 instance parameters that the ASG uses when launching new instances: AMI, instance type, key pair, security groups, storage, user data, IAM role, network settings, and more.

Launch Template (recommended)Launch Configuration (legacy)
VersioningSupports versions — you can update a template and specify which version the ASG usesImmutable — must create a new one to make changes
Spot InstancesCan mix On-Demand and Spot in the same ASGCannot mix instance types
Multiple instance typesYes — specify multiple instance types for better Spot availabilityNo — single instance type only
AWS recommendationUse this for all new ASGsLegacy — do not use for new deployments
→ Launch Templates can also be used independently (without ASG) to launch individual EC2 instances consistently.
→ Key parameters in a Launch Template: AMI ID, instance type(s), key pair, security groups, IAM instance profile, user data script, EBS volume configurations, placement group, tenancy, and network interfaces.
EXAM TIPAlways use Launch Templates over Launch Configurations for new setups. Launch Template = versioned, supports mixed instances (On-Demand + Spot + multiple types). Launch Configuration = immutable, one instance type. The exam may specifically mention which one an ASG uses when testing capabilities.
HLT
ASG Health Checks core
EC2 status checks vs ELB health checks — and the termination/replacement process
▶
→ EC2 health check (default): checks the EC2 instance system status and instance status checks. Marks unhealthy if the instance itself is failing at the hardware/OS level.
→ ELB health check: if the ASG is attached to an ELB, you can optionally use ELB health checks. The ELB health check tests the application layer (HTTP 200 response) — catches app crashes even when the EC2 instance itself is running.
→ Recommendation: use ELB health checks with ASG when the ASG is behind an ELB. ELB checks are more accurate — they verify the application is responding, not just that the instance is running.
→ Termination process: ASG marks instance unhealthy → waits for ELB to drain connections → terminates the instance → launches replacement → registers with ELB.
→ Health check grace period: a period of time after launch during which ASG ignores health check failures. Default: 300 seconds. Set this long enough for your application to start up before health checks begin.
EXAM TIP"ASG keeps terminating newly launched instances immediately" → health check grace period is too short (app hasn't started up yet before health check fires). Increase the grace period. "EC2 instance is running but app is crashed — ASG should replace it" → enable ELB health checks on the ASG.
TRM
ASG Termination Policies advanced
Which instance gets terminated first during scale-in
▶

When ASG needs to scale in (remove instances), it follows a termination policy to decide which instance to terminate. The default policy is designed to keep instances balanced across AZs.

1. Select the AZ with the most instances (to rebalance across AZs).
2. Within that AZ, terminate the instance using the oldest launch configuration or launch template.
3. If tied, terminate the instance closest to the next billing hour (to minimize wasted spend).
→ Lifecycle hooks: pause instance termination (or launch) and perform custom actions (drain connections, back up data, run tests) before the ASG proceeds. Integrates with EventBridge, SQS, SNS, or Lambda.
→ Instance Refresh: rolling replacement of all instances in the ASG when you update the launch template (new AMI, new config). Controlled by minimum healthy percentage — ensures a percentage of instances remain healthy throughout the refresh.
EXAM TIPDefault termination = oldest launch config/template first, in the AZ with the most instances. Instance Refresh = rolling AMI update without manual termination. Lifecycle hooks = pause before terminate or after launch to run custom logic (e.g. drain in-flight work from a queue).
Module 7 — Auto Scaling Policies
ASG Scaling Policies
Dynamic scaling, predictive scaling, scheduled scaling — and how to choose the right one.
DYN
Dynamic Scaling Policies core
Target tracking, simple/step — react to live CloudWatch metrics
▶

Dynamic scaling policies automatically adjust ASG capacity based on real-time CloudWatch metrics (CPU, request count, network, custom metrics). There are three types, each with different behavior.

Policy TypeHow it worksBest for
Target Tracking ScalingDefine a target metric value (e.g. "keep average CPU at 40%"). ASG automatically adds/removes instances to maintain that target. Like a thermostat.Simplest to set up. Best for metrics with a clear target (CPU, ALB request count per target, SQS queue depth).
Simple ScalingCloudWatch alarm triggers a single scaling action (e.g. "if CPU > 70%, add 1 instance"). Then waits for a cooldown period before evaluating again.Predictable, simple workloads where one action at a time is sufficient. Legacy — step scaling is usually preferred.
Step ScalingMultiple scaling steps based on alarm breach size (e.g. "CPU 70–80%: add 1 instance; CPU 80–90%: add 2; CPU > 90%: add 4"). Responds proportionally.Workloads with variable spikes needing faster, proportional responses. No cooldown between steps.
→ Cooldown period: after a scaling activity, ASG waits (default 300 seconds) before responding to additional alarms. Prevents rapid repeated scaling. Target tracking manages its own cooldown — no manual configuration needed.
→ Common target tracking metrics: ASGAverageCPUUtilization, ALBRequestCountPerTarget, ASGAverageNetworkIn/Out. Custom metrics are also supported.
EXAM TIPTarget Tracking is the recommended and most common answer. "Keep CPU at 50%" → Target Tracking. "Add instances proportionally to how bad the overload is" → Step Scaling. Simple Scaling is the oldest — one action per alarm breach with a cooldown wait. Step Scaling doesn't wait for cooldown between steps.
SCH
Scheduled Scaling core
Pre-scale at known times — events, peak hours, business hours
▶

Scheduled scaling adjusts ASG capacity at specific times — useful when you know in advance when demand will be high (business hours, weekly promotions, sporting events).

→ Defined using cron expressions or one-time dates. Example: "every Monday at 8 AM, set min/max/desired to 10/50/20".
→ Proactively scales before the demand hits — unlike reactive scaling which responds after metrics change. Eliminates the warm-up lag of dynamic policies.
→ Can be combined with dynamic scaling — schedule a larger minimum during business hours, then let dynamic scaling handle intra-hour fluctuations.
EXAM TIP"Traffic spikes every Friday at 5 PM for a flash sale" → Scheduled scaling to pre-provision before the event. "Traffic is unpredictable and fluctuates throughout the day" → Target Tracking. Use Scheduled for known patterns, Dynamic for unknown fluctuations.
PRD
Predictive Scaling advanced
ML-powered forecasting — pre-scale based on historical patterns
▶

Predictive scaling uses machine learning to analyze historical CloudWatch metrics and forecasts future traffic patterns. It proactively adjusts ASG capacity ahead of predicted demand spikes — before they happen.

→ Analyzes at least 14 days of historical data to build a forecast model. Generates a capacity schedule for the next 48 hours.
→ Best for workloads with cyclical or recurring patterns (daily business hours, weekly peaks) that repeat consistently.
→ Can be combined with dynamic scaling — predictive handles anticipated load; dynamic handles unexpected spikes on top.
→ Eliminates the scale-out lag — instances are ready before the traffic arrives, not scrambling to catch up after it hits.
EXAM TIP"Traffic follows a predictable daily/weekly pattern and we want instances ready before the spike" → Predictive Scaling. Requires 14+ days of data. Combines well with Target Tracking for the best of both: predictive for baseline + reactive for unexpected load. Distinct from Scheduled: scheduled is manual cron; predictive is ML-automated.
