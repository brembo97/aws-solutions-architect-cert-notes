EC2 Fundamentals — Module 1
EC2 Basics
What EC2 is, how it works, and the key configuration choices you make when launching an instance.
EC2
What is EC2? core
Elastic Compute Cloud — virtual servers in the cloud
▶

EC2 (Elastic Compute Cloud) is AWS's core IaaS offering — it lets you rent virtual machines (instances) running on AWS infrastructure. You choose the OS, CPU, RAM, storage, and networking. EC2 is the foundation of most AWS architectures.

→ EC2 = Infrastructure as a Service (IaaS) — you manage the OS and everything above it; AWS manages the physical hardware.
→ Key choices at launch: AMI (OS image), Instance Type (CPU/RAM), Storage (EBS/instance store), Security Groups, Key Pair, and VPC/Subnet.
→ User Data: a bootstrap script (shell script or cloud-init) that runs once on first boot as root. Used to install software, configure settings, etc. Max size is 16 KB.
→ Instance Metadata: accessible from within the instance at http://169.254.169.254/latest/meta-data/ — returns instance ID, public IP, IAM role credentials, and more. No auth required.
→ EC2 instances can be stopped (persisted), started, rebooted, terminated (deleted), or hibernated.
EXAM TIPUser Data runs only on the first boot, not on every restart. The metadata URL 169.254.169.254 is a link-local address — memorize it. If the exam asks where an EC2 instance retrieves its IAM role credentials, the answer is the instance metadata endpoint.
CON
EC2 Instance Connect tool
Browser-based SSH — no key pair needed on your machine
▶

EC2 Instance Connect is a browser-based terminal for connecting to EC2 Linux instances directly from the AWS Console. AWS temporarily pushes a one-time SSH key to the instance, so you don't need your own key pair file on your local machine.

→ Works only with Amazon Linux 2, Amazon Linux 2023, Ubuntu 20.04+, and some other supported AMIs — not Windows.
→ Requires the instance's Security Group to allow inbound SSH (port 22) from AWS IP ranges used by Instance Connect.
→ IAM permissions required: ec2-instance-connect:SendSSHPublicKey.
→ The temporary key is valid for 60 seconds — just long enough to establish the connection.
EXAM TIPEC2 Instance Connect is the preferred answer when a question asks how to connect to an instance without distributing a private key file, or when a user lost their PEM key. It still requires port 22 open in the Security Group.
SSH
SSH Access core
Connecting to Linux instances via key pair
▶

SSH (Secure Shell) is the standard way to connect to Linux EC2 instances. You authenticate using a key pair: AWS holds the public key (installed on the instance), and you hold the private key (.pem file).

→ Linux/Mac: ssh -i "key.pem" ec2-user@<public-ip>. Default usernames: ec2-user (Amazon Linux), ubuntu (Ubuntu), admin (Debian).
→ Windows (older): use PuTTY + convert .pem to .ppk via PuTTYgen. Windows 10+: native OpenSSH supports .pem directly.
→ The .pem file must have restrictive permissions: chmod 0400 key.pem on Linux/Mac — SSH will refuse to connect if permissions are too open (error: "Unprotected private key file").
→ Port 22 must be allowed in the instance's Security Group from your IP (or 0.0.0.0/0 for testing — not recommended for production).
SECURITY WARNINGNever open port 22 to 0.0.0.0/0 in production. Restrict SSH access to known IP ranges or use Systems Manager Session Manager to eliminate the need for port 22 entirely.

EC2 Fundamentals — Module 3
Security Groups & Classic Ports
The EC2 firewall layer — controlling what traffic is allowed in and out of your instances.
SG
Security Groups core
Virtual firewall controlling EC2 traffic — stateful, allow-only
▶

Security Groups act as a virtual firewall for EC2 instances. They operate at the instance level (not the subnet level) and control inbound and outbound traffic. Security Groups are stateful — if inbound traffic is allowed, the response is automatically allowed regardless of outbound rules.

→ Security Groups contain only Allow rules — there are no Deny rules. Traffic not explicitly allowed is implicitly denied.
→ Rules specify: Type, Protocol, Port Range, and Source/Destination (IP CIDR or another Security Group ID).
→ One instance can have up to 5 Security Groups attached simultaneously. Rules from all attached SGs are combined.
→ Security Groups are regional — they exist within a VPC and cannot be used across VPCs or regions without re-creation.
→ You can reference another Security Group as a source/destination — this is more flexible than IP ranges and is the recommended approach for inter-service communication.
→ By default: all inbound traffic is blocked, all outbound traffic is allowed.
→ Changes to Security Group rules take effect immediately — no reboot needed.
EXAM TIPSecurity Groups are stateful; NACLs are stateless — this is the #1 SG vs NACL exam question. SGs = allow only, instance level, stateful. NACLs = allow + deny, subnet level, stateless. "Connection timeout" usually means a SG rule is blocking. "Connection refused" means the instance is rejecting it (app or OS level, not SG).

Private, public, and Elastic IPs — what they are, how they behave, and when to use each.

ENI
Elastic Network Interfaces (ENI) advanced
Virtual network cards that can be attached/detached from instances
▶

An ENI is a virtual network interface that can be attached to an EC2 instance within a VPC. Every instance has a primary ENI (eth0) created at launch. Additional ENIs can be created and attached independently.

→ Each ENI has: a private IPv4, optionally a public IP, one or more Elastic IPs, one or more Security Groups, and a MAC address.
→ ENIs are AZ-specific — you can only attach an ENI to an instance in the same Availability Zone.
→ You can move ENIs between instances in the same AZ — useful for network failover (move the ENI with its private IP from a failed instance to a healthy one).
→ ENIs are independent of instances — you can create them, configure SGs on them, and attach/detach them without terminating instances.
EXAM TIPENI failover scenario: if an EC2 instance fails and you need to preserve the private IP address for an internal DNS entry, detach the ENI from the failed instance and attach it to a replacement. The private IP moves with the ENI.

RI
Reserved Instances — Deep Dive advanced
Types, payment options, and the Convertible vs Standard distinction
▶
→ Standard RIs: up to 72% discount. Locked to specific instance family, OS, tenancy, and region. Can be sold on the RI Marketplace if unused.
→ Convertible RIs: up to 66% discount. Can change instance family, OS, tenancy during the term. Cannot be sold on the Marketplace.
→ Payment options: All Upfront (best discount), Partial Upfront, No Upfront (smallest discount). 3-year terms give larger discounts than 1-year.
→ RIs apply automatically to matching running instances — they are a billing concept, not a configuration on the instance itself.
→ Zonal vs Regional RIs: Regional RIs apply flexibility across instance sizes in the family; Zonal RIs reserve capacity in a specific AZ.
EXAM TIPStandard RI = cheaper, inflexible, can sell. Convertible RI = slightly less discount, flexible, cannot sell. If the question mentions changing instance type during the term → Convertible RI.
SPT
Spot Instances & Spot Fleet advanced
Up to 90% cheaper — but interruptible with 2-minute warning
▶

Spot Instances use spare EC2 capacity. You set a maximum price; if the Spot price exceeds it (or AWS needs the capacity back), your instance is interrupted with a 2-minute warning. Despite interruption risk, they are the cheapest EC2 option.

→ Two interruption behaviors: Terminate (default), Stop, or Hibernate — configured at request time.
→ Spot Block (deprecated): reserved spot capacity for 1–6 hours with no interruption. No longer available for new customers.
→ Spot Fleet: a collection of Spot Instances (and optionally On-Demand instances) that together meet a target capacity. Fleet automatically replaces interrupted instances and can span multiple instance types/AZs.
→ Spot Fleet allocation strategies: lowestPrice (cheapest pool), diversified (spread across pools for availability), capacityOptimized (pools with most available capacity — best for avoiding interruptions), priceCapacityOptimized (best balance, recommended).
→ Good for: big data, image rendering, batch jobs, CI/CD, containerized workloads. Not for: databases, critical stateful apps, anything that can't tolerate sudden termination.
EXAM TIP"Which strategy reduces Spot interruptions?" → capacityOptimized or priceCapacityOptimized. "Which strategy minimizes cost?" → lowestPrice. The exam loves Spot Fleet strategy questions — know all four.

EC2 Fundamentals — Module 7
Advanced EC2 Features
Hibernate, and key operational features that appear in exam scenarios.
HIB
EC2 Hibernate advanced
Pause an instance and resume exactly where you left off
▶

Hibernate saves the in-memory (RAM) state of an EC2 instance to the root EBS volume, then stops the instance. When started again, the RAM is restored, and the instance resumes exactly where it left off — running processes continue, no cold boot needed.

→ The root EBS volume must be encrypted to use Hibernate — the RAM contents are sensitive.
→ RAM size must be less than 150 GB.
→ The root EBS volume must be large enough to hold the RAM dump — ensure sufficient space.
→ Supported on: On-Demand, Reserved, and Spot instances. Not supported on bare metal instances.
→ Maximum hibernate duration: 60 days.
→ The instance keeps its private IPv4 address and Elastic IP during hibernate, but loses its public IP (if not EIP).
→ Difference from Stop: Stop = RAM is lost, OS boots fresh on start. Hibernate = RAM is preserved, OS resumes instantly.
EXAM TIP"An application takes a long time to warm up and should resume instantly" → Hibernate. "Hibernate requires encrypted root EBS" is a common exam fact. Max RAM = 150 GB, max duration = 60 days — both have been tested.
Instance Storage — Module 8
EBS Volumes
Elastic Block Store — the primary persistent storage for EC2 instances.
EBS
EBS Overview core
Network-attached block storage that persists independently of instances
▶

EBS (Elastic Block Store) volumes are network-attached block storage devices for EC2 instances. They persist independently of instance lifecycle — data survives instance termination (unless "Delete on Termination" is enabled). Think of EBS as a USB drive that attaches over the network.

→ EBS volumes are AZ-locked — a volume in us-east-1a cannot be attached to an instance in us-east-1b. Use snapshots to move data across AZs.
→ By default, root EBS volumes have "Delete on Termination" enabled; additional volumes do not. This can be changed at launch or via CLI.
→ EBS volumes can be attached to one instance at a time (except Multi-Attach — io1/io2 only).
→ You can increase size and change volume type on the fly (without detaching) — this is called Elastic Volumes.
→ Maximum volume size: 64 TiB (for io2 Block Express); gp2/gp3/io1 up to 16 TiB.
EXAM TIPEBS is AZ-specific — this is the most-tested EBS constraint. To move a volume to another AZ or region: take a snapshot → copy snapshot to target region → create volume from snapshot.

IST
EC2 Instance Store core
Physically attached NVMe storage — fast but ephemeral
▶

Instance Store is physically attached storage on the host server — unlike EBS which is network-attached. This gives much higher I/O performance but the data is lost when the instance stops, terminates, or the underlying hardware fails.

→ Ephemeral: data is lost on stop, termination, or hardware failure. Data survives reboots only.
→ Provides the highest possible disk throughput and IOPS — millions of IOPS on some instance types (NVMe SSD).
→ Cannot be detached and reattached. Cannot be resized. Fixed to the instance type.
→ Good for: buffers, caches, scratch data, temporary content, distributed processing where data is replicated elsewhere (e.g., Hadoop nodes).
→ Instance types with instance store: i3, i4i, d2, d3, h1, and some others. Not all instance types have it.
EXAM TIP"Highest I/O performance" → Instance Store. "Ephemeral storage" = Instance Store. If the question asks for persistent storage with high performance → io2 EBS (not instance store, because instance store is lost on stop). The key differentiator: Instance Store = maximum speed, zero persistence.
DATA LOSS RISKInstance store data is lost on stop or termination. Never store data you can't afford to lose without replicating it elsewhere (S3, EBS, database). It is not a substitute for persistent storage.
Instance Storage — Module 9
EBS Features
Snapshots, encryption, Multi-Attach — the deeper EBS capabilities tested on the exam.
SNP
EBS Snapshots core
Point-in-time backups stored in S3 — used to copy volumes across AZs and regions
▶

Snapshots are incremental point-in-time backups of EBS volumes stored in S3 (managed by AWS). They are the primary mechanism for moving EBS data across Availability Zones and regions.

→ Snapshots are incremental — only changed blocks since the last snapshot are stored. First snapshot is a full copy.
→ You don't need to detach a volume to snapshot it, but it is recommended for data consistency.
→ To move a volume to another AZ: snapshot → create volume from snapshot in the target AZ.
→ To move a volume to another region: snapshot → copy snapshot to target region → create volume.
→ EBS Snapshot Archive: move snapshots to a cheaper archive tier (75% cheaper). Restoring from archive takes 24–72 hours.
→ Recycle Bin: optionally retain deleted snapshots for 1 day to 1 year, allowing recovery from accidental deletion.
→ Fast Snapshot Restore (FSR): eliminates I/O latency on first use of a restored volume. Costs extra — billed per AZ per snapshot enabled.
EXAM TIP"How do you migrate an EBS volume from one AZ to another?" → Snapshot + create from snapshot in new AZ. "How do you move to a different region?" → Snapshot + copy snapshot to region + create volume. The snapshot is in S3 and is region-scoped; copying to another region makes a separate copy there.
ENC
EBS Encryption core
AES-256 encryption at rest and in transit — powered by KMS
▶

EBS encryption encrypts data at rest on the volume, data in flight between the instance and the volume, snapshots, and volumes created from those snapshots — all transparently with no performance impact (hardware encryption on Nitro-based instances).

→ Encryption uses AWS KMS keys (either the default aws/ebs key or a customer-managed CMK).
→ Encryption is not enabled by default — you must opt in per volume or enable "Encryption by default" at the account/region level.
→ An unencrypted volume cannot be directly encrypted in place. Workaround: create snapshot → copy snapshot with encryption enabled → create new volume from encrypted snapshot.
→ Snapshots of encrypted volumes are automatically encrypted. Volumes created from encrypted snapshots are automatically encrypted.
→ You can share encrypted snapshots, but the recipient must have access to the KMS key used for encryption.
EXAM TIP"How do you encrypt an existing unencrypted EBS volume?" → Snapshot → copy with encryption → create encrypted volume. You cannot encrypt a volume in-place. Encrypted snapshots → encrypted volumes; unencrypted snapshots → can create encrypted volumes by specifying encryption at volume creation.
MA
EBS Multi-Attach advanced
Attach one io1/io2 volume to multiple instances simultaneously
▶

Multi-Attach allows a single EBS io1 or io2 volume to be attached to multiple EC2 instances in the same Availability Zone simultaneously. Each instance has full read/write access.

→ Supported only on io1 and io2 volume types — not available on gp2, gp3, st1, or sc1.
→ Instances must be in the same AZ as the volume.
→ Maximum of 16 instances attached simultaneously.
→ The application must handle concurrent writes — use a cluster-aware file system (e.g., GFS2, OCFS2) or application-level locking. Standard Linux ext4/xfs are not cluster-aware.
→ Use cases: clustered databases (Oracle RAC), high-availability applications that manage concurrent write operations.
EXAM TIPMulti-Attach = io1/io2 only, same AZ, up to 16 instances, application must handle concurrent writes. Don't confuse with EFS (which is a managed shared file system for many instances across AZs — much simpler to use for shared storage).
Instance Storage — Module 10
AMI — Amazon Machine Images
Pre-configured OS images used to launch EC2 instances — the blueprint for your servers.
AMI
AMI Overview core
What AMIs are, how they work, and how to create custom ones
▶

An AMI (Amazon Machine Image) is a template that contains the software configuration (OS, application server, applications) required to launch an EC2 instance. AMIs are regional — an AMI in us-east-1 must be copied to use in eu-west-1.

→ AMIs contain: a root volume snapshot, launch permissions (who can use it), and block device mapping (which volumes to attach at launch).
→ Three sources of AMIs: AWS-provided (Amazon Linux, Windows, Ubuntu, etc.), AWS Marketplace (third-party commercial), and Custom/Community (your own or public community AMIs).
→ Custom AMI creation: start an instance → customize (install software, configure) → stop the instance → create AMI. AWS takes a snapshot of the root volume automatically.
→ Custom AMIs enable faster EC2 Auto Scaling — pre-baking software means instances launch ready without running user data scripts on every boot.
→ AMIs are region-specific. To use in another region, copy the AMI (this copies the underlying snapshots).
→ AMI permissions: private by default. Can be made public, or shared with specific AWS account IDs.
EXAM TIP"How to launch pre-configured instances quickly at scale?" → Custom AMI. "AMIs are region-specific" is frequently tested — must be copied to use in another region. Do not confuse AMI (launch template) with a Snapshot (volume backup).
Instance Storage — Module 11
EFS & Storage Comparisons
Elastic File System — shared NFS storage — and the definitive EBS vs EFS vs Instance Store comparison.
EFS
Amazon EFS core
Managed NFS file system — shared across many instances and AZs
▶

EFS (Elastic File System) is a managed NFS (Network File System) that can be mounted simultaneously on hundreds of EC2 instances across multiple Availability Zones. Unlike EBS, it automatically scales — you pay only for what you use.

→ Uses NFSv4.1 protocol. Mounts via a mount target in each AZ. Security Groups control access to the mount target.
→ Scales automatically — no provisioning needed. Capacity grows and shrinks as you add/remove files. Can reach petabyte scale.
→ Multi-AZ by default (Standard storage class) — data is replicated across multiple AZs. One Availability Zone storage class is cheaper (single AZ, dev/test).
→ Compatible with Linux-based AMIs only — not Windows.
→ Approximately 3x more expensive than EBS gp2 per GB but you pay only for what you use (no pre-provisioning).
→ Performance modes: General Purpose (default, low latency) and Max I/O (higher throughput, higher latency — for highly parallel workloads).
→ Throughput modes: Bursting (throughput scales with storage size), Provisioned (set throughput independently), Elastic (auto-scales — recommended for spiky workloads).
→ EFS Lifecycle Management: automatically moves files not accessed for N days to EFS-IA (Infrequent Access) storage class — up to 92% cheaper.
EXAM TIPEFS = shared file system, multi-AZ, Linux only, auto-scaling, more expensive per GB. The exam often asks "how to share a file system between multiple EC2 instances across AZs?" → EFS. Do not use EFS for Windows instances — use FSx for Windows File Server instead.
