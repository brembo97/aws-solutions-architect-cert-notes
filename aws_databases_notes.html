Module 1 — Relational Database Service
Amazon RDS
Managed relational databases in the cloud — engines, features, and when RDS is the right choice.
RDS
RDS Overview core
Managed relational DB service — AWS handles patching, backups, and scaling
▶

RDS (Relational Database Service) is a managed service that makes it easy to set up, operate, and scale relational databases in the cloud. AWS handles OS patching, backups, failover, and hardware provisioning — you manage the schema and queries.

→ Supported engines: MySQL, PostgreSQL, MariaDB, Oracle, Microsoft SQL Server, and Amazon Aurora (Aurora is RDS-compatible but architecturally different).
→ RDS is not serverless (except Aurora Serverless) — you provision instance types and storage upfront. You pay for the instance whether it's idle or busy.
→ Storage is backed by EBS (gp2, gp3, or io1). Storage auto-scaling can be enabled — RDS increases storage automatically when free space is low, up to a configured max.
→ You cannot SSH into an RDS instance — it is a managed service. AWS controls the underlying OS.
→ RDS runs inside your VPC. Access is controlled by Security Groups, and it should not be publicly accessible in production.
→ RDS supports Multi-AZ for high availability and Read Replicas for read scaling — these are different features with different purposes.
EXAM TIPRDS is "managed" but not "serverless" — you still choose instance sizes. The key advantage over self-managed EC2 databases is that AWS handles patching and backups. If a question asks why to use RDS over running a database on EC2, the answer is reduced operational overhead, not performance.
REP
Read Replicas vs Multi-AZ core
The most-tested RDS comparison on the exam
▶

Read Replicas and Multi-AZ serve completely different purposes and are frequently confused on the exam. Understanding the difference is critical.

Read ReplicasMulti-AZ
PurposeScale read traffic — offload SELECT queries from the primaryHigh availability — automatic failover if primary fails
ReplicationAsynchronous — slight replication lag possibleSynchronous — standby is always in sync
Standby accepts traffic?Yes — each replica has its own DNS endpoint and can be read fromNo — standby is passive, only used on failover
Failover?Manual — you must promote a replica to primary if neededAutomatic — DNS CNAME flips to standby within 1–2 min
Cross-region?Yes — replicas can be in different regionsSame region only (different AZ)
CostBilled as a separate DB instanceBilled as standby instance (same price as primary)
Use caseAnalytics, reporting, read-heavy appsProduction databases requiring HA and no data loss
→ You can have up to 5 Read Replicas for MySQL, MariaDB, PostgreSQL. Each has its own DNS endpoint.
→ Read Replicas can be promoted to standalone databases — they then break replication and become independent.
→ Network cost: Read Replica replication within the same region is free. Cross-region replication incurs data transfer costs.
→ Multi-AZ failover is triggered by: primary instance failure, AZ outage, DB instance class change, or OS patching.
EXAM TIP"Improve read performance" → Read Replica. "Improve availability / disaster recovery" → Multi-AZ. These are the two most common scenario questions for RDS. Multi-AZ standby cannot be read from — it is purely for failover. If the exam says "zero downtime during failover" → Multi-AZ.
CST
RDS Custom advanced
Full OS + database access for Oracle and SQL Server
▶

RDS Custom gives you access to the underlying OS and database settings — something not possible with standard RDS. It is only available for Oracle and Microsoft SQL Server.

→ Allows SSH/SSM access to the underlying EC2 instance running the database.
→ Use cases: legacy enterprise apps that require specific OS configurations, custom database patches, or direct file system access.
→ You must deactivate Automation Mode before performing customizations — otherwise RDS may revert your changes.
→ Supported only for Oracle and SQL Server — not available for MySQL, PostgreSQL, or MariaDB.
EXAM TIPIf the question mentions needing OS-level access to a managed database, or custom Oracle/SQL Server configurations → RDS Custom. Standard RDS blocks all OS access.

BAK

Automated backups vs manual snapshots — key differences
▶

TriggerAutomatic — daily during backup windowManual — user-initiated anytime
Retention1–35 days (configurable). Deleted when DB is deleted.Kept indefinitely until you delete them. Survive DB deletion.

→ Restoring an RDS backup always creates a new DB instance — you cannot restore in-place.
→ For long-term retention (compliance), use manual snapshots — automated backups are purged after the retention window.
→ Snapshots can be copied to other regions and shared with other AWS accounts — useful for cross-region DR.

EXAM TIP"Recover to any point in the last 7 days" → Automated backups with PITR. "Preserve a backup for 3 years for compliance" → Manual snapshot (automated backups are deleted after max 35 days). Restoring always creates a NEW DB instance — you can't overwrite the existing one.
MON
RDS Monitoring tool
CloudWatch metrics, Enhanced Monitoring, and Performance Insights
▶

RDS integrates with multiple monitoring layers. Understanding what each one monitors and at what granularity is important for both operations and exam questions.

→ CloudWatch metrics: hypervisor-level metrics — CPU, storage, network. Collected every 1 minute. Does not show per-process data.
→ Enhanced Monitoring: OS-level metrics from an agent running on the DB instance. Granularity down to 1 second. Shows individual process CPU and memory usage.
→ Performance Insights: database-level query performance analysis. Shows which SQL queries are consuming the most load. Retention up to 2 years.
→ Key CloudWatch alarms to set: FreeStorageSpace (low disk), CPUUtilization (high CPU), DatabaseConnections (connection pool exhaustion).
EXAM TIPCloudWatch = hypervisor-level (no per-process). Enhanced Monitoring = OS-level (per-process). Performance Insights = SQL query-level. If the question asks "how to find which queries are slowing down the database" → Performance Insights.
Module 1 — RDS
RDS Security & RDS Proxy
Encryption, IAM authentication, network isolation, and the connection pooling proxy layer.
SEC
RDS Security core
Encryption at rest, in transit, IAM auth, and network controls
▶
→ Encryption at rest: uses AWS KMS (AES-256). Must be enabled at launch — cannot enable encryption on an existing unencrypted DB. Workaround: snapshot → copy with encryption → restore from encrypted snapshot.
→ Encryption in transit: SSL/TLS between application and RDS. Enforced by setting rds.force_ssl=1 parameter (PostgreSQL) or requiring SSL in MySQL grants.
→ IAM Database Authentication: supported for MySQL and PostgreSQL. Authenticates using an IAM-generated token instead of a password. Token is valid for 15 minutes. No password stored in the DB.
→ Network isolation: deploy RDS in a private subnet with no public access. Use Security Groups to restrict access to specific application subnets or EC2 instances.
→ Secrets Manager: best practice for storing and rotating DB credentials automatically. RDS integrates natively — credentials auto-rotate without app changes.
EXAM TIP"How to encrypt an existing unencrypted RDS database?" → Snapshot → copy snapshot with encryption enabled → restore. Cannot enable encryption in-place — same constraint as EBS. IAM Database Auth works only for MySQL and PostgreSQL, not Oracle or SQL Server.
SECURITY WARNINGNever make an RDS instance publicly accessible in production. Place it in a private subnet, and only allow inbound traffic from the application layer's Security Group (not 0.0.0.0/0).
PXY
RDS Proxy advanced
Managed connection pooler — critical for Lambda + RDS architectures
▶

RDS Proxy sits between your application and RDS, pooling and sharing database connections. It prevents connection exhaustion when many short-lived clients (especially Lambda functions) connect to the database simultaneously.

→ Connection pooling: instead of each Lambda invocation opening a new DB connection, Proxy maintains a warm pool. Reduces DB connection overhead by up to 66%.
→ RDS Proxy is fully managed, serverless, and auto-scaling — no instance to provision.
→ Supports failover time reduction: during Multi-AZ failover, Proxy queues connections and reconnects automatically. Failover time drops from ~60s to ~66% less.
→ Enforces IAM authentication at the proxy layer — stores credentials in Secrets Manager. Apps authenticate to the Proxy via IAM, not with raw DB passwords.
→ RDS Proxy is VPC-only — it is never publicly accessible.
→ Supports: MySQL, PostgreSQL, MariaDB, Microsoft SQL Server, and Aurora (MySQL + PostgreSQL compatible).
EXAM TIPThe canonical RDS Proxy exam scenario is Lambda + RDS. Lambda can spawn thousands of concurrent invocations, each opening a DB connection — this exhausts connection limits quickly. RDS Proxy solves this. If you see "Lambda connecting to RDS" and a problem with "too many connections" → RDS Proxy.
Module 2 — Aurora
Amazon Aurora
AWS's cloud-native relational database — faster, more available, and more scalable than standard RDS engines.
AUR
Aurora Overview core
AWS-proprietary engine compatible with MySQL and PostgreSQL
▶

Aurora is AWS's proprietary cloud-native relational database. It's drop-in compatible with MySQL and PostgreSQL drivers, but internally uses a distributed, fault-tolerant architecture that delivers significantly better performance and availability than standard RDS.

→ Compatible with MySQL 5x faster and PostgreSQL 3x faster than standard RDS engines on equivalent hardware.
→ Storage auto-scales from 10 GB up to 128 TB in 10 GB increments automatically — you never provision storage upfront.
→ Storage is replicated 6 times across 3 AZs. Tolerates: up to 2 AZ failures for writes, up to 3 AZ failures for reads.
→ Up to 15 Read Replicas (vs 5 for standard RDS). Replicas use the shared storage layer — replication lag is typically under 10ms.
→ Failover is automatic and takes less than 30 seconds. Aurora promotes a replica to primary without any data loss.
→ Costs approximately 20% more than standard RDS, but the performance and availability improvements typically justify the cost.
EXAM TIPAurora storage is NOT the same as EBS — it's a custom distributed shared storage layer. Key numbers: 6 copies across 3 AZs, up to 128 TB, up to 15 replicas, failover < 30 seconds. When the exam says "highest availability relational database" → Aurora.
END
Aurora Endpoints core
Writer, reader, custom, and instance endpoints
▶

Aurora provides multiple endpoint types so you can direct different workloads to the appropriate instances without hardcoding individual instance addresses.

EndpointPoints toUse case
Writer EndpointAlways points to the current primary (auto-updates on failover)All writes (INSERT, UPDATE, DELETE)
Reader EndpointLoad-balances across all available Read ReplicasRead-heavy workloads — single DNS, distributes queries
Custom EndpointA subset of instances you define (e.g., larger instances only)Analytical queries needing specific instance sizes
Instance EndpointOne specific DB instanceDiagnostics, direct connection to a specific node
EXAM TIPApplication code should use Writer Endpoint (not a specific instance endpoint) so that failovers are transparent. Reader Endpoint provides built-in load balancing across replicas — your app doesn't need to know how many replicas exist.
SRV
Aurora Serverless advanced
Auto-scaling capacity — ideal for intermittent or unpredictable workloads
▶

Aurora Serverless automatically starts up, shuts down, and scales compute capacity based on your application's needs. You only pay for the database storage and the compute seconds actually consumed.

→ Scales by adding or removing Aurora Capacity Units (ACUs). Each ACU is ~2 GB of memory plus proportional CPU and networking.
→ Can scale to zero when not in use (v2 keeps a minimum of 0.5 ACU). Ideal for dev/test, infrequent workloads, and new apps with unpredictable load.
→ Aurora Serverless v2 scales in fine-grained increments (0.5 ACU) and much faster than v1 — suitable for production workloads.
→ Accessed via the Data API — an HTTP endpoint that eliminates the need for persistent DB connections (great for Lambda).
EXAM TIP"Unpredictable or intermittent database workload" → Aurora Serverless. "Variable traffic with cost optimization" → Aurora Serverless. If the question mentions the Data API or accessing Aurora from Lambda without connection management → Aurora Serverless.
Module 2 — Aurora
Aurora Advanced Concepts
Global Database, Multi-Master, cloning, and Aurora-specific backups and security.
GLB
Aurora Global Database advanced
One primary region + up to 5 secondary regions with ~1 second lag
▶

Aurora Global Database spans multiple AWS regions, with a primary region for reads/writes and up to 5 secondary regions that receive replicated data. Designed for globally distributed applications and cross-region disaster recovery.

→ Replication lag to secondary regions is typically under 1 second (storage-level replication, not application-level).
→ Each secondary region can have up to 16 Read Replicas.
→ Cross-region failover (promoting a secondary to primary) takes less than 1 minute with an RTO of under 1 minute.
→ Useful for: globally distributed apps needing low-latency local reads, GDPR-compliant data residency, and cross-region DR.
EXAM TIP"Cross-region disaster recovery with RPO < 1 second and RTO < 1 minute for a relational database" → Aurora Global Database. Standard RDS Multi-AZ does not span regions — it is same-region only.
CLN
Aurora Cloning advanced
Fast, space-efficient database clone — copy-on-write protocol
▶

Aurora cloning creates a new DB cluster that shares the same underlying storage as the source, using a copy-on-write protocol. It is much faster than restoring a snapshot because no data is copied initially.

→ The clone starts immediately and shares the source's data pages. Only modified pages are duplicated going forward.
→ Creates a separate, isolated cluster — changes to the clone don't affect the source, and vice versa.
→ Use case: create a production-identical staging or test environment without waiting for a full snapshot restore.
EXAM TIP"Create a copy of production database for testing without impacting production and without waiting a long time" → Aurora Cloning. Much faster than snapshot + restore for large databases.
BAK
Aurora Backups & Security core
Continuous backups, backtrack, and security model
▶
→ Aurora backups are continuous and automatic — stored in S3. Cannot be disabled. Retention window: 1–35 days.
→ Backtrack: rewind an Aurora database to a specific point in the past without restoring from a snapshot. Available for Aurora MySQL only. Max backtrack window: 72 hours.
→ Encryption: same as RDS — KMS at rest, TLS in transit. Must be enabled at creation. Encrypt existing unencrypted cluster via snapshot copy.
→ Supports IAM Database Authentication (MySQL and PostgreSQL modes).
EXAM TIPAurora Backtrack ≠ restoring from backup. Backtrack rewinds the live cluster in-place — no new cluster is created. "Undo an accidental DELETE without creating a new cluster" → Backtrack (Aurora MySQL only, within 72 hours).
Module 3 — In-Memory Caching
Amazon ElastiCache
Managed Redis and Memcached — sub-millisecond latency caching for databases and applications.
ELC
ElastiCache Overview core
Managed in-memory data store — reduces database load dramatically
▶

ElastiCache provides managed Redis or Memcached clusters in AWS. It sits in front of a database (typically RDS) and serves frequently-read data from memory, reducing database load and response latency from milliseconds to microseconds.

→ AWS manages OS patching, node replacements, monitoring, backups (Redis), and cluster scaling.
→ Using ElastiCache requires application code changes — the app must be written to check the cache before querying the DB, and to invalidate cache entries when data changes.
→ Common use cases: database query caching, session stores, leaderboards, real-time analytics, rate limiting.

Module 4 — NoSQL
Amazon DynamoDB
Fully managed, serverless NoSQL key-value and document database with single-digit millisecond performance at any scale.
DDB
DynamoDB Overview core
Serverless NoSQL — scales to trillions of rows with consistent performance
▶

DynamoDB is a fully managed, serverless, key-value and document NoSQL database. AWS handles all infrastructure, replication, and scaling — there are no servers to manage, no storage to provision, and no connection limits to worry about.

→ Data is automatically replicated across 3 AZs within a region — highly available by default.
→ Scales horizontally to handle millions of requests per second and trillions of items with consistent single-digit millisecond latency.
→ Tables store items (rows) with attributes (columns). Schema-less — each item can have different attributes.
→ Primary key options: Partition Key (hash key) alone, or Partition Key + Sort Key (range key) composite. Must be defined at table creation and cannot be changed.
→ Maximum item size: 400 KB. Not suitable for storing large binary objects (use S3 instead).
→ Integrates natively with Lambda, API Gateway, AppSync, and IAM — a natural fit for serverless architectures.
EXAM TIPDynamoDB is the go-to answer for: serverless applications, mobile backends, gaming leaderboards, IoT event data, session storage at scale, and any workload needing single-digit ms latency without operational overhead. It is NOT a relational DB — no joins, no SQL.

Module 4 — DynamoDB
DynamoDB Advanced Features
DAX, Streams, TTL, Global Tables, backups — the features that appear in advanced exam scenarios.
DAX
DynamoDB Accelerator (DAX) advanced
In-memory cache built specifically for DynamoDB — microsecond latency
▶

DAX is a fully managed, DynamoDB-compatible in-memory cache. It sits transparently between your application and DynamoDB — no application code changes needed beyond pointing to the DAX cluster endpoint instead of DynamoDB directly.

→ Reduces DynamoDB read latency from milliseconds to microseconds (up to 10x improvement).
→ DAX is DynamoDB-specific — unlike ElastiCache, it only works with DynamoDB. ElastiCache can cache results from any database.
→ Default cache TTL: 5 minutes. Can be configured per request.
→ DAX handles cache invalidation automatically — but only for item-level reads. If you have complex query caching needs, use ElastiCache instead.
→ Multi-AZ with replication — handles node failures automatically.
EXAM TIPDAX vs ElastiCache for DynamoDB: DAX is a simpler, native drop-in cache for DynamoDB individual item reads (GetItem, Query, Scan). ElastiCache is more flexible and can cache aggregated query results or complex computations. "Reduce DynamoDB latency with minimal code change" → DAX.
STR
DynamoDB Streams advanced
Ordered change log — triggers Lambda or replication on table changes
▶

DynamoDB Streams captures a time-ordered sequence of item-level modifications (create, update, delete) in a DynamoDB table. Records are retained for 24 hours.

→ Each stream record contains: the type of modification, a timestamp, and optionally the before/after images of the item.
→ View types: KEYS_ONLY, NEW_IMAGE, OLD_IMAGE, NEW_AND_OLD_IMAGES.
→ Common use cases: trigger Lambda on data changes, cross-region replication, audit trails, real-time analytics pipelines, search index updates (Elasticsearch sync).
→ Streams are consumed by Lambda using an event source mapping — Lambda polls the stream and processes records in batches.
EXAM TIP"Trigger an action every time a DynamoDB item is modified" → DynamoDB Streams + Lambda. "Replicate a DynamoDB table to another region" → DynamoDB Streams or Global Tables. Stream records are available for 24 hours — not long-term storage.
GTT
Global Tables & TTL advanced
Multi-region active-active replication and automatic item expiry
▶
→ Global Tables: fully managed multi-region, multi-active replication. Reads and writes can be performed in any region — all replicas sync within typically under 1 second.
→ Global Tables require DynamoDB Streams to be enabled — it uses streams internally to propagate changes.
→ Conflict resolution: last-writer-wins based on timestamps. Applications should avoid writing to the same item from multiple regions simultaneously.
→ TTL (Time to Live): define a Unix timestamp attribute on items. DynamoDB automatically deletes expired items within 48 hours of expiration — at no cost.
→ TTL is not instant — items expire within 48 hours, not immediately. Filter expired items in your queries if exactness matters.
EXAM TIP"DynamoDB available in multiple regions with low-latency reads and writes globally" → Global Tables. "Automatically delete session data after 30 minutes" → TTL attribute. TTL deletions do not consume WCUs — they are free.
BAK
DynamoDB Backups core
PITR and on-demand backups — no performance impact
▶
→ Point-in-Time Recovery (PITR): enables recovery to any second in the last 35 days. Must be explicitly enabled. Restores to a new table.
→ On-demand backups: full backup at any time, retained indefinitely until deleted. No impact on performance or availability.
→ Backups do not consume RCU/WCU — zero performance impact on the table.
→ Restores always create a new DynamoDB table — not an in-place restore.
EXAM TIPPITR must be explicitly enabled on each table — it is not on by default. "Restore DynamoDB table to a specific point in time from the last month" → PITR. "Long-term backup for compliance" → On-demand backup (indefinite retention).
Module 5 — Specialty Databases
Other AWS Databases
Purpose-built databases for specific data models — DocumentDB, Neptune, Keyspaces, and Timestream.
DOC
Amazon DocumentDB core
MongoDB-compatible document database — Aurora architecture, JSON documents
▶

DocumentDB is AWS's fully managed document database compatible with MongoDB. Like Aurora is to MySQL/PostgreSQL, DocumentDB is to MongoDB — it uses the Aurora-style distributed storage architecture under the hood.

→ Compatible with MongoDB drivers, tools, and queries — existing MongoDB apps can migrate with minimal changes.
→ Stores JSON documents. Supports flexible schemas, rich queries, and nested data structures.
→ Storage auto-scales up to 64 TB. Data replicated 3 times across 3 AZs.
→ Supports up to 15 read replicas. Automatic failover in under 30 seconds.
→ Use cases: content management systems, catalogs, user profiles, applications migrating from MongoDB.
EXAM TIP"MongoDB workload on AWS" or "migrate MongoDB to managed AWS service" → DocumentDB. It is NOT DynamoDB — DocumentDB is MongoDB-compatible, DynamoDB is a key-value/document store with a completely different API.
NEP
Amazon Neptune core
Fully managed graph database — optimized for connected data
▶

Neptune is a fully managed graph database service. It stores data as nodes and edges (relationships), making it exceptionally fast at traversing complex relationships that would be expensive in a relational database.

→ Supports two graph query languages: Gremlin (Property Graph / TinkerPop) and SPARQL (RDF / W3C standard).
→ Storage: up to 64 TB, replicated 6 times across 3 AZs. Up to 15 read replicas.
→ Optimized for queries with many hops through relationships — e.g. "find all friends of friends who also follow X".
→ Use cases: social networks, recommendation engines, fraud detection, knowledge graphs, network/IT topology, life sciences (drug interactions).
→ Supports Neptune Streams for change data capture (similar to DynamoDB Streams).
EXAM TIPIf the exam mentions graph data, relationships between entities, social networks, or recommendation engines → Neptune. Trigger phrases: "highly connected datasets", "graph traversal", "fraud detection using relationships", "social graph".
KSP
Amazon Keyspaces (for Apache Cassandra) advanced
Managed Cassandra-compatible wide-column database
▶

Amazon Keyspaces is a fully managed, serverless, wide-column database compatible with Apache Cassandra. Existing Cassandra applications can migrate using the Cassandra Query Language (CQL) with minimal changes.

→ Uses CQL (Cassandra Query Language) — compatible with existing Cassandra drivers and tools.
→ Serverless — tables scale up and down automatically based on traffic. No cluster management.
→ Data replicated 3 times across multiple AZs. Single-digit millisecond latency at any scale.
→ Supports On-Demand and Provisioned capacity modes (like DynamoDB).
→ Use cases: IoT device data, time-series data, high-velocity industrial telemetry, any Cassandra migration to AWS.
EXAM TIP"Migrate Apache Cassandra workload to AWS" or "managed Cassandra" → Amazon Keyspaces. Key differentiator from DynamoDB: Keyspaces uses CQL and is designed for Cassandra migrations. DynamoDB uses its own API.
TMS
Amazon Timestream advanced
Serverless time-series database — IoT and operational metrics
▶

Timestream is a fully managed, serverless time-series database optimized for storing and analyzing data that changes over time — sensor readings, metrics, events. It automatically stores recent data in memory and moves older data to cheaper magnetic storage.

→ Purpose-built for time-series data — timestamps are first-class citizens. Faster and cheaper than storing time-series in a relational DB or DynamoDB.
→ Storage tiering: in-memory tier for recent data (hot), magnetic tier for historical data (cold). Automatic lifecycle policies move data between tiers.
→ Built-in time-series analytics functions: smoothing, approximation, interpolation — no need to write these yourself.
→ Stores and analyzes trillions of events per day. Scales automatically — no cluster management.
→ Use cases: IoT telemetry, application metrics (DevOps), industrial equipment monitoring, financial tick data, clickstream analytics.
→ Integrates with: Amazon QuickSight (visualization), Apache Kafka, Kinesis, IoT Core, Grafana.
EXAM TIP"Store IoT sensor data over time" or "analyze metrics that change over time" → Timestream. The key trigger is time-series data — measurements timestamped over a period. Much more efficient than DynamoDB for this pattern.
