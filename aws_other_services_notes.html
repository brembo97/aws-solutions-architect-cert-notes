Infrastructure as Code & Management
CloudFormation & Systems Manager
Define infrastructure as code and manage EC2 fleets at scale — two foundational AWS operations services.
CFN
AWS CloudFormation core
Declarative infrastructure as code — provision entire AWS environments from a template
▶

CloudFormation lets you define your entire AWS infrastructure in a JSON or YAML template file — VPCs, EC2, RDS, IAM roles, Lambda functions, everything. You submit the template and CloudFormation creates, updates, or deletes all the resources as a single unit called a stack.

→ Declarative: you describe the desired end state ("I want 2 EC2 instances behind an ALB"), not the steps to get there. CloudFormation figures out the order and dependencies.
→ Stacks: a collection of AWS resources managed as one unit. Create, update, or delete the entire stack atomically. If creation fails, CloudFormation rolls back all resources (all-or-nothing).
→ Change Sets: preview what CloudFormation will change before applying an update — see which resources will be added, modified, or replaced. Prevents surprises.
→ StackSets: deploy the same CloudFormation template across multiple AWS accounts and regions in one operation. Used with AWS Organizations for governance and standardization.
→ Service Role: a CloudFormation service role lets CloudFormation create resources on your behalf using the role's permissions instead of your own. Enables least-privilege — give CloudFormation exactly the permissions it needs without granting users broad access.
→ Drift detection: compare the actual state of deployed resources to the template. Identifies manual changes made outside of CloudFormation.
EXAM TIP"Deploy identical infrastructure across 10 accounts and 3 regions" → CloudFormation StackSets. "Preview changes before deploying" → Change Sets. "Provision and delete a dev environment daily" → CloudFormation stack create/delete. CloudFormation is free — you pay only for the resources it provisions.
SSM
AWS Systems Manager — Session Manager & Other Services core
Manage EC2 at scale — shell access, patching, inventory, run commands, all without SSH
▶

AWS Systems Manager (SSM) is a suite of tools for managing and operating EC2 instances and on-premises servers at scale. The SSM Agent runs on each managed instance and communicates with the SSM service — no SSH or bastion needed.

→ Session Manager: secure shell access to EC2 instances through the browser or CLI — no SSH keys, no port 22 open, no bastion host, no public IP required. All sessions are logged to S3/CloudWatch for audit. IAM controls who can start sessions.
→ Run Command: execute commands or scripts across hundreds of instances simultaneously — without SSH. Use for installing software, applying patches, collecting diagnostics. Results are tracked per instance.
→ Patch Manager: automate OS and application patching across your fleet. Define patch baselines (which patches to approve) and maintenance windows (when to apply). Compliance reports show which instances are missing patches.
→ Automation: define runbooks (automation documents) for common operational tasks — restart instances, create AMIs, remediate Config findings, rotate secrets. Triggered manually, by EventBridge, or by Config remediation.
→ Inventory: collect metadata from managed instances — installed software, network config, Windows updates, running services. Query centrally to understand your fleet.
→ Parameter Store: covered in the Security module — hierarchical config/secrets storage. Part of the SSM family.
EXAM TIP"Shell access to private EC2 without bastion/SSH" → Session Manager. "Patch 500 instances on a schedule" → Patch Manager + Maintenance Window. "Run a script on all prod instances at once" → Run Command. SSM Agent + IAM instance role is the prerequisite for all SSM features. Works on on-premises servers too.
Compute, Edge & Data Integration
Batch, Outposts & AppFlow
Managed batch processing, on-premises AWS infrastructure, and no-code SaaS data integration.
BAT
AWS Batch core
Fully managed batch processing — run hundreds of thousands of jobs without managing infrastructure
▶

AWS Batch plans, schedules, and executes batch computing workloads across a dynamically provisioned fleet of EC2 instances or Fargate tasks. You submit jobs (Docker containers) and Batch handles queuing, scheduling, and compute provisioning.

→ Jobs are Docker containers. Batch provisions the right number and type of EC2 instances (or Fargate tasks) based on the job queue depth and resource requirements you define.
→ Batch vs Lambda: Lambda has a 15-minute timeout and limited resources (10 GB RAM). Batch jobs have no time limit and can use any EC2 instance type (including GPU). Use Batch for long-running, resource-heavy workloads. Use Lambda for short event-driven functions.
→ Supports Spot Instances for massive cost savings on fault-tolerant batch workloads.
EXAM TIP"Run a containerized job that takes 2 hours and needs 64 GB RAM" → AWS Batch (Lambda max 15 min / 10 GB). "Process 10,000 files in parallel" → Batch (auto-provisions the fleet). Batch = no time limit, any instance type, managed queuing. Lambda = 15-min limit, serverless, event-driven.
OUT
AWS Outposts advanced
AWS infrastructure rack installed in your own data centre — same APIs, local processing
▶

AWS Outposts is a fully managed service that extends AWS infrastructure, APIs, and tools to your on-premises data centre or co-location facility. AWS delivers, installs, and maintains physical racks of AWS compute and storage in your building.

→ Run EC2, EBS, S3 (S3 on Outposts), ECS, EKS, RDS, and EMR locally — using the exact same AWS APIs and tools you use in the cloud.
→ Use cases: ultra-low latency (data must stay local), data residency requirements (data cannot leave a specific country/facility), local data processing before sending results to the cloud.
→ Outposts connects back to the parent AWS region for management, monitoring, and access to other AWS services.
EXAM TIP"Run AWS services on-premises with the same APIs" → AWS Outposts. "Data residency — data cannot leave the building" → Outposts (process locally). Don't confuse with Snow Family (offline data transfer) or Local Zones (AWS-managed micro data centres near cities).
APF
Amazon AppFlow tool
No-code data integration between SaaS apps and AWS services
▶

Amazon AppFlow is a fully managed integration service that transfers data securely between SaaS applications (Salesforce, Slack, ServiceNow, Google Analytics, SAP, Zendesk, etc.) and AWS services (S3, Redshift, EventBridge) — without writing code or managing servers.

→ Supports scheduled transfers (hourly/daily), event-driven triggers (new Salesforce record → S3), or on-demand runs. Includes built-in data transformation (filtering, field mapping, validation, masking PII).
→ Traffic goes over the AWS private network (not the internet) when transferring between SaaS connectors and AWS — via PrivateLink.
EXAM TIP"Transfer Salesforce data to S3 or Redshift without code" → AppFlow. "Integrate SaaS applications with AWS" → AppFlow. It's the no-code ETL bridge between SaaS and AWS.
Customer Communication
SES & Pinpoint
Send transactional and marketing emails at scale — two services with different focuses.
SES
Amazon SES — Simple Email Service core
High-volume transactional email — send and receive emails using API or SMTP
▶

Amazon SES is a cloud-based email sending and receiving service designed for transactional, notification, and marketing emails at scale. It provides raw email infrastructure — you handle the content, audience, and logic.

→ Send via SMTP interface (drop-in replacement for any SMTP server) or the SES API (programmatic sending from Lambda, EC2, etc.).
→ Can also receive emails — process incoming email with rules (save to S3, trigger Lambda, forward to SNS).
→ Use cases: order confirmations, password resets, system alerts, marketing newsletters, invoice delivery.
EXAM TIP"Send transactional emails from your application" → SES. "Receive and process incoming emails" → SES (receive rules). SES is raw email infrastructure. For marketing campaigns with audience management → Pinpoint.
PIN
Amazon Pinpoint advanced
Multi-channel marketing — campaigns, segments, analytics across email, SMS, push, and voice
▶

Amazon Pinpoint is a multi-channel outbound communication and marketing service. Unlike SES (raw email infrastructure), Pinpoint provides campaign management, audience segmentation, message templates, A/B testing, and engagement analytics — it's a marketing platform, not just an email sender.

→ Channels: Email, SMS, Push Notifications, Voice, In-App. All managed from one service. SES only handles email.
→ Built-in audience segmentation (create segments based on user attributes/events), campaign scheduling, and delivery analytics (open rates, click rates, conversions).
EXAM TIPSES = developer email infrastructure (API, SMTP, raw sending). Pinpoint = marketer tool (campaigns, segments, multi-channel, analytics). "Send a targeted push notification to Android users who haven't opened the app in 30 days" → Pinpoint. "Send a password reset email from Lambda" → SES.
Cost Management & Scheduling
Cost Explorer, Anomaly Detection & Instance Scheduler
Visualize spending, detect unusual charges, and automatically stop/start instances to save money.
CEX
AWS Cost Explorer core
Visualize, analyze, and forecast your AWS spending over time
▶

Cost Explorer is a free tool in the AWS console that lets you visualize your AWS costs and usage with interactive graphs. Filter and group spending by service, account, region, tag, instance type, or any combination — to understand where money is going.

→ View costs at daily or monthly granularity. Historical data available for up to 12 months. Forecast future spending up to 12 months ahead based on usage trends.
→ RI/Savings Plan recommendations: Cost Explorer recommends optimal Reserved Instance or Savings Plan purchases based on your usage patterns to maximize savings.
→ Right-sizing recommendations: identifies EC2 instances that are over-provisioned and recommends smaller instance types.
EXAM TIP"Visualize AWS costs by service and forecast next quarter's spending" → Cost Explorer. "Get RI purchase recommendations" → Cost Explorer. "Detect unusual spending" → Cost Anomaly Detection (separate service — below).
ANM
AWS Cost Anomaly Detection advanced
ML-powered — automatically detect unexpected spending spikes
▶

Cost Anomaly Detection uses machine learning to continuously monitor your spending patterns and automatically detect unusual cost increases — accidental resource over-provisioning, misconfigured services, or unexpected usage spikes. No manual threshold setting required.

→ Create monitors by service, account, cost category, or cost allocation tag. Set alert thresholds (e.g. notify me if anomaly impact exceeds $100).
→ Notifications via SNS or email. Root cause analysis identifies which service/account/usage type caused the anomaly.
EXAM TIP"Automatically detect and alert on unexpected AWS spending increases" → Cost Anomaly Detection. It's ML-based — learns your normal patterns, no manual thresholds to maintain. Separate from Cost Explorer (which is for visualization and forecasting).
SCH
Instance Scheduler on AWS tool
Automatically stop and start EC2 and RDS instances on a schedule to reduce costs
▶

Instance Scheduler is an AWS Solution (CloudFormation-deployed) that automatically stops and starts EC2 instances and RDS databases based on a defined schedule — e.g. run dev instances only during business hours (8am–6pm weekdays), stop them overnight and weekends.

→ Deployed via a CloudFormation template. Runs as a Lambda function on a schedule. Instances are tagged (e.g. Schedule=office-hours) and the scheduler matches tags to schedule definitions stored in DynamoDB.
→ Potential savings: ~65% on dev/test instances that only need to run during business hours.
EXAM TIP"Stop dev/test EC2 instances outside business hours to save money" → Instance Scheduler on AWS. It uses tags to identify which instances to schedule. For single instances, EventBridge Scheduler → Lambda (stop/start) also works.
Developer Tools & Migration
Amplify & AWS Migration Service
Full-stack app development for frontend/mobile developers and database migration.
AMP
AWS Amplify advanced
Full-stack development platform for web and mobile — frontend hosting + backend services
▶

AWS Amplify is a set of tools and services for building full-stack web and mobile applications. It lets frontend developers build AWS-powered backends without deep AWS knowledge — authentication, APIs, storage, and hosting with minimal configuration.

→ Amplify Hosting: CI/CD for static sites and SSR (server-side rendered) web apps. Connect a Git repo (GitHub, GitLab, Bitbucket), push code, and Amplify builds and deploys automatically. Like Vercel/Netlify but on AWS.
→ Amplify Backend: provision AWS backend resources (Cognito auth, AppSync GraphQL API, S3 storage, DynamoDB, Lambda functions) from the frontend code using a CLI or visual builder.
→ Supports React, Next.js, Vue, Angular, Flutter, React Native, iOS, Android.
EXAM TIP"Full-stack web app with CI/CD hosting + Cognito + AppSync backend — simplest path for frontend developers" → Amplify. Think of Amplify as the "App Runner of full-stack" — highest abstraction, least ops. Don't confuse with Elastic Beanstalk (server-based app deployment) or App Runner (container/API deployment).
DMS
AWS Database Migration Service (DMS) core
Migrate databases to AWS with minimal downtime — supports homogeneous and heterogeneous migrations
▶

AWS DMS migrates databases to AWS quickly and securely. The source database remains fully operational during migration — minimal downtime. DMS handles initial data load and then continuous replication of ongoing changes (CDC — Change Data Capture) until you cut over.

→ Homogeneous migration (same engine → same engine): Oracle → RDS Oracle, MySQL → Aurora MySQL. Straightforward — DMS handles replication directly.
→ Heterogeneous migration (different engine → different engine): Oracle → Aurora PostgreSQL, SQL Server → Aurora MySQL. Requires AWS Schema Conversion Tool (SCT) first to convert the schema and stored procedures, then DMS to migrate the data.
→ Sources: on-premises databases, EC2-hosted databases, RDS, Aurora, S3, MongoDB, DocumentDB. Targets: RDS, Aurora, DynamoDB, Redshift, S3, OpenSearch, Kinesis Data Streams, DocumentDB, Neptune.
→ Continuous replication (CDC): after the initial full load, DMS keeps replicating changes from source to target in near-real-time. Used for ongoing data synchronization or live migration with minimal cutover window.
→ DMS runs on a replication instance (EC2 under the hood) — you choose the instance size based on data volume and change rate.
EXAM TIP"Migrate Oracle on-premises to Aurora PostgreSQL" → SCT (schema conversion) + DMS (data migration). "Migrate MySQL on-premises to RDS MySQL with minimal downtime" → DMS only (homogeneous, no SCT needed). "Continuous data replication from on-premises DB to S3" → DMS with CDC. DMS is the standard answer for any "migrate a database to AWS" question.

CMP
Easy-to-Confuse Pairs core
The pairs the exam uses as distractors in this module
▶
→ SES vs Pinpoint: SES = raw email infrastructure (API/SMTP, developer tool). Pinpoint = marketing platform (campaigns, segments, multi-channel, analytics). "Send a password reset" → SES. "Send a promo campaign to inactive users" → Pinpoint.
→ Batch vs Lambda: Batch = no time limit, any instance type, managed queuing, Docker containers. Lambda = 15-min max, 10 GB RAM max, event-driven, serverless. "Long-running job" → Batch. "Quick event handler" → Lambda.
→ Cost Explorer vs Cost Anomaly Detection: Cost Explorer = visualization and forecasting (you look at it). Anomaly Detection = ML alerts (it tells you when something's wrong). They complement each other.
→ Amplify vs Elastic Beanstalk vs App Runner: Amplify = frontend/mobile full-stack (React, Next.js + backend). Beanstalk = traditional server apps (Java, .NET, Node on EC2). App Runner = container/API deployment. Different audiences, different abstraction levels.
→ DMS vs MGN (Application Migration Service): DMS = database migration (data + schema). MGN = server migration (entire EC2/VM lift-and-shift). "Migrate a database" → DMS. "Migrate an entire server/OS to EC2" → MGN.
→ Outposts vs Snow Family vs Local Zones: Outposts = permanent AWS rack in your data centre (long-term on-prem processing). Snow = portable offline data transfer devices (large data migration). Local Zones = AWS mini-regions near cities (low-latency edge for specific metros).
