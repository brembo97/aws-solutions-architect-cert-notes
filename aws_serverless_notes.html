Module 1 — Lambda
AWS Lambda — Core
Function-as-a-Service: run code without servers, pay per invocation, scale automatically from zero to thousands.
λ
Lambda Overview core
Event-driven FaaS — zero server management, pay per 1ms of execution
▶

AWS Lambda runs your code in response to events — HTTP requests, file uploads, queue messages, database changes, scheduled triggers, and more. You provide the code; AWS handles every aspect of the runtime environment, OS, scaling, and availability.

→ No servers to manage — no EC2, no patching, no capacity planning. Lambda scales automatically from 0 to thousands of concurrent executions in seconds.
→ Pricing: pay per number of invocations + duration (billed in 1ms increments). Free tier: 1 million invocations and 400,000 GB-seconds of compute per month — never expires.
→ Supported runtimes: Node.js, Python, Java, .NET (C#/PowerShell), Ruby, Go. Also supports custom runtimes (any language via the Lambda runtime API) and container images (up to 10 GB).
→ Execution environment: Lambda runs in an isolated, single-use container. The /tmp directory provides up to 10 GB of ephemeral storage per execution. Memory: 128 MB to 10,240 MB (10 GB), adjustable in 1 MB increments.
→ CPU is proportional to memory — you cannot configure CPU directly. At 1,769 MB, Lambda gets 1 full vCPU. At 10,240 MB, Lambda gets ~6 vCPUs.
→ IAM Execution Role: every Lambda function must have an IAM execution role granting it permissions to call other AWS services (write to S3, read from DynamoDB, publish to SNS, etc.).
EXAM TIP"Need more CPU in Lambda" → increase memory (CPU scales with memory, not independently). Free tier is 1M invocations + 400K GB-seconds/month, permanent. Lambda container images up to 10 GB are useful for large ML models or dependencies. Always attach an execution role for cross-service calls.
TRG
Lambda Triggers & Event Sources core
What can invoke Lambda — synchronous vs asynchronous vs poll-based
▶

Lambda can be invoked by dozens of AWS services. Knowing the invocation model (sync, async, or poll-based) is important because it determines error handling and retry behavior.

ModelSourcesError handling
SynchronousAPI Gateway, ALB, Lambda URLs, Cognito, CloudFront, Lex, Alexa, Kinesis Data AnalyticsCaller receives the error response directly. No automatic retry by Lambda — the caller is responsible.
AsynchronousS3, SNS, EventBridge, SES, CloudFormation, CodeCommit, CodePipeline, CloudWatch Logs, IoTLambda retries automatically: up to 2 retries (3 total attempts). Failed events go to a DLQ (SQS or SNS) if configured. Use idempotent functions.
Poll-based (Event Source Mapping)SQS, Kinesis Data Streams, DynamoDB Streams, MSK, self-managed KafkaLambda polls the source. Failed batches go back to the stream/queue. DLQ on SQS, or bisect-on-error for streams.
→ Event Source Mapping (ESM): Lambda internally polls SQS/Kinesis/DynamoDB Streams on your behalf. You configure batch size (number of records per invocation) and starting position. Lambda scales the number of pollers based on queue/stream depth.
→ For SQS: Lambda deletes processed messages automatically. For Kinesis/DynamoDB Streams: Lambda checkpoints position — streams are ordered and replayable.
EXAM TIPAsynchronous invocations retry 2 times automatically (3 total). Always use idempotent functions for async sources. S3 → Lambda is always async. API Gateway → Lambda is sync. SQS → Lambda uses ESM (poll-based). Kinesis → Lambda uses ESM (poll-based, ordered).

CON
Lambda Concurrency core
Reserved concurrency, provisioned concurrency, throttling, and the 1,000 limit
▶

Concurrency is the number of Lambda function instances running simultaneously. Each invocation runs in its own isolated execution environment — one invocation per environment at a time.

→ Account concurrency limit: default 1,000 concurrent executions per region across ALL functions. This is a shared pool.
→ Throttling: when concurrency limit is reached, new invocations are throttled. Synchronous invocations return a 429 Too Many Requests error. Async invocations are retried automatically for up to 6 hours.
→ Reserved Concurrency: guarantee a maximum concurrency for a specific function (e.g. reserve 200 for function A). This caps function A at 200 AND reserves those 200 from the account pool — other functions can use the remaining 800. Ensures one function can't consume all shared concurrency. Also ensures the function will never scale beyond the reserved limit.
→ Provisioned Concurrency: pre-warm a number of execution environments before invocations arrive. Eliminates cold starts for those environments. Costs money — you pay for provisioned environments even when not invoked. Use for latency-sensitive production functions.
→ Cold start: when a new execution environment is initialized (no warm environment available) — Lambda must download code, start the runtime, and run initialization code. Takes 100ms–1s+ depending on runtime and package size. Java and .NET have longer cold starts than Node.js/Python.
→ Warm start: when an existing execution environment is reused. Initialization code doesn't run again. Code outside the handler function (global scope, DB connections) persists between warm invocations.
EXAM TIPReserved concurrency = cap + reserve (protects other functions). Provisioned concurrency = pre-warmed environments (eliminates cold starts, costs money). "One Lambda is consuming all concurrency, starving other functions" → set Reserved Concurrency on the greedy function. "Eliminate cold starts for a latency-sensitive API" → Provisioned Concurrency.
SNP
Lambda SnapStart advanced
10x faster cold starts for Java — snapshot the initialized environment
▶

Lambda SnapStart is a feature that dramatically reduces cold start latency for Java functions. Instead of initializing the runtime from scratch on every cold start, Lambda takes a snapshot of the fully initialized execution environment and restores it instantly on new invocations.

→ Currently supports Java 11 and Java 17 runtimes only. Not available for Node.js, Python, or other runtimes.
→ Lambda runs the initialization phase (Init) once, takes a snapshot of memory and disk state, then caches and restores that snapshot for subsequent cold starts. Startup time can improve by up to 10x.
→ The snapshot is created when you publish a new function version. Each version gets its own snapshot.
→ No additional cost for SnapStart — you pay the same Lambda execution pricing.
→ Caveat: if your initialization code generates unique state (random seeds, unique IDs, network connections), that state will be the same in every restored environment. Use RuntimeHook callbacks to re-generate unique state after restore.
EXAM TIP"Reduce Java Lambda cold starts" → SnapStart (Java 11/17 only). "Eliminate cold starts entirely for all invocations" → Provisioned Concurrency (works for all runtimes). SnapStart reduces cold start duration; Provisioned Concurrency eliminates cold starts by pre-warming. Know the difference.
Module 3 — Lambda
Advanced Lambda
VPC deployment, edge execution, RDS integration, layers, and Lambda URLs.
VPC
Lambda in a VPC core
Access private resources — RDS, ElastiCache, private APIs — from Lambda
▶

By default, Lambda runs in an AWS-managed VPC with internet access but no access to your private VPC resources (RDS in a private subnet, ElastiCache, etc.). Deploying Lambda inside your own VPC gives it private network access — at the cost of losing direct internet access.

→ Lambda in a VPC attaches an ENI (Elastic Network Interface) in each specified subnet. The function gets a private IP in your VPC and can reach private resources using Security Group rules.
→ Internet access in a VPC Lambda: once inside a VPC, Lambda loses direct internet access. To restore it: deploy Lambda in a private subnet and route outbound traffic through a NAT Gateway in a public subnet.
→ Lambda in a public subnet does NOT get internet access — Lambda doesn't use public IPs even in a public subnet. Always use private subnet + NAT Gateway.
→ For AWS service calls (S3, DynamoDB, SQS) from a VPC Lambda: either use a NAT Gateway (internet path, costs money) or a VPC Endpoint (private path, stays within AWS network, cheaper and faster).
→ Lambda requires ec2:CreateNetworkInterface, ec2:DescribeNetworkInterfaces, and ec2:DeleteNetworkInterface permissions in its execution role to create/manage ENIs.
EXAM TIP"Lambda needs to access RDS in a private subnet" → deploy Lambda in same VPC, configure Security Groups to allow Lambda ENI to reach RDS port. "Lambda in VPC needs internet access" → private subnet + NAT Gateway (NOT public subnet — Lambda doesn't use public IPs). "Lambda in VPC calling S3 without internet" → S3 VPC Gateway Endpoint.

Run code at CloudFront edge locations — no origin round-trip for request/response manipulation

RDS
Lambda & RDS — Invoking Lambda + Event Notifications advanced
RDS-initiated Lambda calls and database event notifications
▶

Lambda and RDS can interact in two directions: Lambda can call RDS (standard pattern), and RDS can invoke Lambda or send event notifications outbound.

→ Invoking Lambda from RDS/Aurora: RDS for PostgreSQL and Aurora MySQL/PostgreSQL can invoke a Lambda function directly from within a stored procedure or trigger — e.g. send an email when a record is inserted. RDS must have network access to Lambda (VPC or internet) and must be granted IAM permission to invoke Lambda.
→ RDS Event Notifications: subscribe to RDS operational events (DB instance started/stopped, failover, snapshot created, parameter group change) via SNS. These are infrastructure-level events, NOT data-level changes (row inserts, updates).
→ For data-level change events (row-level CDC): use RDS → Lambda via stored procedure/trigger (directly), or Aurora with DynamoDB Streams-style change capture patterns (Aurora → Lambda via native integration).
→ RDS Proxy + Lambda: for Lambda functions connecting to RDS, always use RDS Proxy to pool connections. Lambda at scale can open thousands of connections simultaneously — RDS Proxy prevents connection exhaustion.
EXAM TIPRDS Event Notifications → operational events (failover, maintenance) via SNS. NOT for data changes. "Trigger Lambda when a row is inserted in RDS" → Lambda invoked from a stored procedure/trigger (or DMS CDC to Kinesis). "Lambda connecting to RDS at scale" → always use RDS Proxy to prevent connection pool exhaustion.
LYR
Lambda Layers & Lambda URLs advanced
Shared dependencies + direct HTTPS endpoints without API Gateway
▶
→ Lambda Layers: ZIP archives containing shared libraries, custom runtimes, or data. Attached to functions at deploy time — the layer is mounted at /opt in the execution environment. A function can use up to 5 layers. Total unzipped size (function + all layers) must not exceed 250 MB.
→ Layers can be shared across functions and accounts — publish a layer publicly for community use or share privately with specific AWS accounts.
→ AWS provides public layers (e.g. AWS Lambda Powertools, AWS Data Wrangler).
→ Lambda Function URLs: a dedicated HTTPS endpoint for a Lambda function — no API Gateway required. URL format: https://<url-id>.lambda-url.<region>.on.aws. Supports auth (IAM SigV4) or no auth (public).
→ Function URLs are simpler and cheaper than API Gateway for basic HTTP invocations — no request routing, no API management features. Use when you just need a public endpoint for one function.
EXAM TIPLambda Layers = share code/libraries between functions, max 5 layers, 250 MB total unzipped. Lambda Function URLs = direct HTTPS endpoint without API Gateway — simpler but no routing, throttling policies, or usage plans. "Invoke Lambda over HTTPS without API Gateway" → Lambda Function URL.
Module 4 — API Gateway
Amazon API Gateway
Managed API front door — create, publish, secure, and scale REST, HTTP, and WebSocket APIs.
APIG
API Gateway Overview core
Serverless API management — routing, auth, throttling, and caching in one service
▶

API Gateway is a fully managed service for creating and managing APIs at any scale. It handles the undifferentiated work of API management — request routing, authentication, authorization, SSL termination, throttling, caching, monitoring, and versioning — so you can focus on business logic.

→ Three API types: REST API (feature-rich, supports all capabilities), HTTP API (lower latency, lower cost, subset of features — recommended for Lambda and HTTP backends), WebSocket API (bidirectional, stateful connections — chat, gaming, real-time dashboards).
→ Integrations: Lambda Function (most common), HTTP endpoints (any publicly accessible HTTP backend), AWS Services (direct integration with SQS, Step Functions, Kinesis, DynamoDB, etc. without Lambda), Mock (return static responses for testing).
→ Endpoint types: Edge-Optimized (default — CloudFront edge, best for global clients), Regional (single region, use with your own CloudFront for more control), Private (VPC-only access via VPC Interface Endpoint).
→ Throttling: default 10,000 req/s with a burst of 5,000. Configure usage plans and API keys to throttle per client. Returns 429 Too Many Requests when throttled.
→ Caching: REST API supports response caching (TTL 0–3,600 seconds). Reduces backend calls for repeated identical requests. Caching is per stage and costs extra (cache size 0.5 GB to 237 GB).
→ CORS: must be enabled explicitly on API Gateway when frontend JS (e.g. S3 static site) calls the API from a different domain. Configure Access-Control-Allow-Origin headers in the API response.
EXAM TIPHTTP API = cheaper and lower latency but fewer features (no usage plans, no API key-based throttling, no request/response transformation). REST API = full-featured. WebSocket API = real-time bidirectional. "Client getting 429 from API Gateway" → throttling limit hit — increase limit or add a SQS buffer. "Serverless REST API" → API Gateway + Lambda + DynamoDB (the canonical serverless CRUD stack).

IAM, Lambda Authorizer, Cognito — three ways to secure an API

STG
Stages, Deployments & Canary Releases advanced
Manage API versions and gradually roll out changes
▶
→ API changes must be deployed to a stage to take effect. Stages are named deployments (e.g. dev, staging, prod). Each stage has its own URL, settings, and can have its own variables.
→ Stage variables: key-value pairs available to your API — use them to point different stages to different Lambda functions or backends (e.g. dev stage → Lambda dev alias, prod stage → Lambda prod alias).
→ Canary deployments: route a percentage of traffic (e.g. 5%) to a new deployment before promoting it fully. Test new versions in production with minimal risk. If issues arise, roll back instantly.
→ Lambda aliases work naturally with API Gateway canary: point a stage to a Lambda alias (e.g. $LATEST) and use Lambda alias weights to split traffic between function versions.
EXAM TIP"Test a new API version in production with 5% of traffic" → API Gateway Canary deployment. "Use the same API across multiple environments without duplicating it" → stages + stage variables pointing to different backends. Stage URL format: https://api-id.execute-api.region.amazonaws.com/stage-name.
Module 5 — Orchestration
AWS Step Functions
Visual serverless orchestration — coordinate Lambda, ECS, SQS, and other services into complex multi-step workflows.
SFN
Step Functions Overview core
Serverless state machine — orchestrate workflows beyond Lambda's 15-minute limit
▶

AWS Step Functions lets you model complex workflows as visual state machines using Amazon States Language (ASL — JSON/YAML). Each step in the workflow is a state. Steps can invoke Lambda, run ECS tasks, call other AWS services, wait, retry, catch errors, and branch on conditions.

→ Maximum execution duration: 1 year — far beyond Lambda's 15-minute limit. Workflows can include human approval steps that wait indefinitely.
→ Two workflow types: Standard (exactly-once execution, full history, up to 1 year, ~$0.025/1,000 state transitions) and Express (at-least-once, up to 5 minutes, much cheaper — for high-volume IoT/event processing).
→ State types: Task (invoke a service), Choice (conditional branching), Wait (pause for time or callback), Parallel (concurrent branches), Map (iterate over a list), Pass (transform data), Succeed, Fail.
→ Built-in error handling: Retry (retry a failing task with configurable backoff) and Catch (catch errors and route to a fallback state).
→ Use cases: order processing pipelines, data processing workflows, ML training pipelines, human approval workflows, multi-step ETL jobs, microservice orchestration.
EXAM TIP"Workflow that takes more than 15 minutes" → Step Functions (Standard, up to 1 year). "Complex multi-step process with retries and branching" → Step Functions. "High-volume low-latency event processing workflow" → Step Functions Express. Step Functions = orchestration (manages state externally). Lambda = compute inside each step.
Module 6 — Identity
Amazon Cognito
Managed user identity — sign-up, sign-in, social login, and AWS credential vending for mobile and web apps.
COG
Cognito User Pools vs Identity Pools core
Authentication (who you are) vs Authorization (what AWS resources you can access)
▶

Cognito has two distinct components that solve different problems — they are frequently confused on the exam. Understanding which component does what is critical.

User PoolsIdentity Pools (Federated Identities)
PurposeAuthentication — who is this user? User sign-up, sign-in, MFA, password reset, social login (Google, Facebook, Apple, SAML/OIDC IdP).Authorization — what can this user access? Vends temporary AWS credentials (via STS AssumeRoleWithWebIdentity) so users can access AWS services directly.
OutputJWT tokens: ID Token, Access Token, Refresh TokenTemporary AWS credentials: Access Key, Secret Key, Session Token
IntegrationAPI Gateway (validate JWT directly). ALB (authenticate before routing).Direct AWS SDK calls from mobile/web apps (S3, DynamoDB, etc.) using temporary credentials.
Guest accessNo — users must sign inYes — can grant limited AWS credentials to unauthenticated (guest) users
→ Combined flow: User signs in via User Pool → gets JWT → exchanges JWT with Identity Pool → gets temporary AWS credentials → calls S3/DynamoDB directly from the app. This is the canonical mobile backend pattern.
→ User Pool features: customizable hosted UI for sign-in, MFA (TOTP, SMS), email/phone verification, custom Lambda triggers (pre-sign-up validation, post-authentication enrichment), user groups, app clients.
→ ALB + Cognito User Pool: ALB can authenticate users via Cognito before routing requests to targets — offloads auth from the application entirely.
EXAM TIPUser Pool = login/auth → JWTs. Identity Pool = exchange token for AWS credentials → direct AWS service calls. "Mobile app users uploading to S3" → User Pool (sign in) → Identity Pool (get S3 credentials) → S3 PutObject. "API Gateway auth for web/mobile users" → User Pool JWT authorizer. These two are among the most confused concepts on the exam.
